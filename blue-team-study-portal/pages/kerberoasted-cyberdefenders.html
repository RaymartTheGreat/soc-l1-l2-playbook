Kerberoasted (CyberDefenders)

Hello these are my notes taken while solving the lab, not a polished writeup. I was practicing for the ECIR exam at the time and plan to revisit the lab to add screenshots and turn this into a full write-up in the future.

Scenario

As a diligent cyber threat hunter, your investigation begins with a hypothesis: 'Recent trends suggest an upsurge in Kerberoasting attacks within the industry. Could your organization be a potential target for this attack technique?' This hypothesis lays the foundation for your comprehensive investigation, starting with an in-depth analysis of the domain controller logs to detect and mitigate any potential threats to the security landscape.

Note: Your Domain Controller is configured to audit Kerberos Service Ticket Operations, which is necessary to investigate kerberoasting attacks. Additionally, Sysmon is installed for enhanced monitoring.

https://cyberdefenders.org/blueteam-ctf-challenges/kerberoasted/

before solving we need to know

what is kerberoasting attack:

the attacker requests a service tickets (TGS) for service accounts in AD then extract and crack them offline to retrieve the passwords in plaintext

after that he has access to the AD enviroment with the privileges of the compromised account

How to detect it

1-

🔍 Key Event IDs for Detecting Kerberoasting

Event IDDescriptionRelevance to Kerberoasting
4769TGS (Service Ticket) RequestAttackers request service tickets for cracking. Look for high volume requests or RC4-HMAC (0x17) encryption.
4662Directory Services AccessIndicates LDAP SPN enumeration before ticket requests.
5145File Access (SMB)Attackers might use SMB enumeration to find service accounts.
4624 (Type 9)Logon via TicketDetects Pass-the-Ticket (PtT) attacks using stolen Kerberos tickets.
10 (Sysmon)LSASS AccessIndicates possible ticket dumping using Mimikatz or Rubeus.
7045New Service InstalledAttackers may install services for persistence after Kerberoasting.

2-

look for Ticket_Encryption_Type="0x17" → RC4-HMAC Encryption

which is an outdated encryption method for Kerberos

Attackers prefer RC4-HMAC because they can forge tickets without needing AES encryption keys.

If an account suddenly requests TGTs using RC4-HMAC, but other users use AES, it might be an attacker using a forged ticket.

3-

Multiple 4769 (TGS Request) events from a single user.

4-

Look for unauthorized processes interacting with lsass.exe (e.g., rundll32.exe, powershell.exe).

5-

also look for famous tools like mimikatz.exe, rubeus.exe

🛠 Detection Query in Splunk

jsx
index=security EventCode=4769 Ticket_Encryption_Type="0x17"
| stats count by Account_Name, Service_Name, Client_Address
| sort - count

Important note :

if a user mistypes a password, there should be a 4768 (TGT request) followed by 4771 (pre-authentication failure).

If there’s no 4771, but authentication still succeeds, it may indicate a forged ticket.

#Forged Ticket refers to a maliciously created Kerberos ticket used to gain unauthorized access.

  • Golden Ticket → Forged TGT (Ticket-Granting Ticket) for full domain access.
  • Silver Ticket → Forged TGS (Service Ticket) to access a specific service.

Encryption Analysis and Kerberos Service Ticket Requests

First, I found that the encryption type is 0x17. With Event.code = 4769 (Kerberos service ticket, i.e., TGS, was requested), I observed 163 Kerberos ticket requests. Notably, the user john.doe requested two TGS tickets for two important services—FileShareService and SQLServicewithin the same second. This behavior suggests that it might be a script or an automated tool.

Successful Logon Analysis

Next, I looked at Event.code = 4624 (successful logon). The logs showed successful logon attempts from accounts like DC01$, SYSTEM, and Administrator. These are common system accounts used in regular operations, so they are not particularly concerning. However, I also found that SQLService had five successful logins, which is worrisome given its high value as a target. By examining who accessed SQLService, I identified the attacker’s IP as “10.0.0.154.” With this IP, further investigation can be initiated.

New Service Installation

Event.code = 7045 (a new service was installed on the system) revealed that two services were installed on the domain controller DC01:

  • iOOEDsXjWeGRAyGl
  • YeDIRrUiXDmvRLyq

These services were installed on the local system with obfuscated service names, indicating they are malicious. Attackers often use such naming conventions to evade detection and blend in with legitimate system processes.

Registry Modification for RDP

Event.code = 1 (process creation) revealed a registry modification at the following path:

"hklm\system\currentcontrolset\control\terminal server"

This registry path controls RDP settings. By modifying it, the attacker enabled RDP on the Domain Controller, granting themselves the ability to remotely control the system at will.

This change is significant because it not only provides persistent access but also allows attackers to bypass other remote management restrictions that might have been in place. Enabling RDP post compromise is a common tactic since it allows remote interaction with the desktop of a Windows machine as if the user were physically present, often without triggering administrative alerts.

Persistence Using WMI

For persistence, the attacker employed WMI event subscription (referencing Persistence with WMI Event Subscription – T1546.003). Sysmon Event IDs 19, 20, and 21 capture the creation of WMI filters, consumers, and bindings.

I identified a WMI event consumer named Updater that was created using a PowerShell script. The script was executed with flags such as -nop (no profile), -w hidden (hidden window), and -noninteractive. These are typical indicators of malicious intent, designed to avoid user detection and minimize forensic artifacts.

In the WMI event filter for Updater, the following query was found:

jsx
SELECT * FROM __InstanceCreationEvent WITHIN 60
WHERE TargetInstance ISA 'Win32_NTLogEvent'
  AND TargetInstance.EventCode = '4625'
  AND TargetInstance.Message LIKE '%johndoe%'

This query indicates that the WMI event subscription filter targets the Win32_NTLogEvent class, which represents entries in the Windows event log. The filter is designed to trigger when a failed login attempt (EventCode 4625) containing the username johndoe is detected. By targeting this class, the attacker ensures that their malicious WMI event consumer executes whenever there is a failed login attempt for the johndoe account. This setup could serve multiple purposes, such as triggering further exploitation attempts, logging failed brute-force attempts for future use, or even masking their presence by manipulating logs in response to failed access attempts.
