Scenario

As a cybersecurity analyst at SecureTech Industries, you've been alerted to unusual activities within the company's Active Directory environment. Your mission is to analyze log data, trace the attack's progression, determine the breach's scope, and identify the attacker's tactics, techniques, and procedures (TTPs).

https://cyberdefenders.org/blueteam-ctf-challenges/shadowroast/

Initial Access

Malicious File Execution

  • File Name: AdobeUpdater.exe
  • Execution Path: C:\Users\Public\Downloads\AdobeUpdater.exe
  • Execution Method: Launched via Windows GUI (explorer.exe) with high integrity, suggesting elevated privileges.
  • Behavior: Exhibited characteristics of ApacheBench (ab.exe), indicating renaming as a deceptive technique.

Detection Method

  • Event ID 1 (Process Creation) → Monitored process executions to detect unauthorized executables.
  • Splunk Query:
jsx
index="shadowroast" AND event.code=1 AND winlog.computer_name=Office-PC.CORPNET.local AND (winlog.event_data.Image=*cmd* OR winlog.event_data.Image=*powershell*)
jsx
 index="shadowroast" AND event.code=1 AND winlog.computer_name=Office-PC.CORPNET.local AND (winlog.event_data.ProcessId=4780 OR winlog.event_data.ParentProcessId=4780)
  • Conclusion:
    • The attacker likely used social engineering to deliver and execute AdobeUpdater.exe, gaining initial access with elevated privileges.

Credential Access

AS-REP Roasting Attack

(Rubeus was used to perform an AS-REP Roasting attack, a technique targeting accounts with Kerberos pre-authentication disabled to extract hashes that can be cracked offline.)

  • Tool Used: BackupUtility.exe (Identified as Rubeus.exe)
  • SHA256 Hash: 1bfbefa4ff4d0df3ee0090b5079cf84ed2e8d5377ba5b7a30afd88367d57b9ff
  • Attack Technique: Extracted Kerberos AS-REP hashes for accounts with preauthentication disabled to crack them offline.

Detection Method

  • Event ID 1 (Process Creation) → Detected the execution of BackupUtility.exe.
  • Splunk Query:

``jsx index="shadowroast" event.code=1 winlog.event_data.Image="BackupUtility.exe" | table _time winlog.event_data.CommandLine winlog.event_data.ParentCommandLine ``

  • Conclusion:
    • The attacker extracted Kerberos hashes, targeting vulnerable accounts to obtain plaintext credentials.

Persistence

Registry Run Key Modification

  • Registry Path: HKCU\Software\Microsoft\Windows\CurrentVersion\Run\AdobeUpdater
  • Associated File: AdobeUpdater.exe

Detection Method

  • Event ID 13 (Registry Value Set) → Identified modifications in the Windows Registry.
  • Splunk Query:

``jsx index="shadowroast" event.code=13 winlog.event_data.TargetObject=* |table _time winlog.event_data.TargetObject winlog.event_data.Image ``

  • Conclusion:
    • The attacker added a Run key to ensure the execution of AdobeUpdater.exe upon system startup, achieving persistence.

Privilege Escalation

Credential Dumping with Mimikatz

(After executing Rubeus, the attacker was seen executing Mimikatz (Under the name of “DefragTool.exe”) revealing a different parent user, confirming a second compromised user, CORPNET\tcooper)

  • Tool Used: DefragTool.exe (Identified as Mimikatz)
  • SHA256 Hash: 92804faaab2175dc501d73e814663058c78c0a042675a8937266357bcfb96c50
  • Purpose: Extracted NTLM hashes to gain SYSTEM privileges.

Detection Method

  • Event ID 1 (Process Creation) → Monitored DefragTool.exe execution.
  • Splunk Query:

``jsx index="shadowroast" event.code=1 winlog.event_data.Image="DefragTool.exe" | table _time winlog.event_data.CommandLine winlog.event_data.ParentCommandLine ``

  • Conclusion:
    • The attacker successfully dumped credentials, enabling further privilege escalation and lateral movement.

Enabling Remote Desktop Protocol (RDP)

  • Registry Key Modified: HKLM\SYSTEM\CurrentControlSet\Control\Terminal Server\fDenyTSConnections
  • Modification: Set to 0 (Enabled RDP for lateral movement).

  • Conclusion:
    • The attacker modified RDP settings to enable lateral movement across the network.

Techniques Used

1- AS-REP Roasting

Overview

AS-REP Roasting is an attack technique targeting Active Directory (AD) user accounts that have Kerberos preauthentication disabled. It allows attackers to obtain a user’s encrypted Ticket Granting Ticket (TGT) preauthentication data, which can be brute-forced offline to recover plaintext passwords.

How It Works

  1. User Authentication in Kerberos:
    • When a user wants to log in, they request an Authentication Service (AS) Ticket from the Key Distribution Center (KDC) (which is usually the Domain Controller).
    • By default, the user must first prove their identity by encrypting a timestamp with their password hash (preauthentication).
    • If preauthentication is enabled, an attacker cannot request an AS ticket without knowing the user’s password.
  2. What Happens When Preauthentication Is Disabled?
    • If a user’s account has preauthentication disabled, the KDC immediately sends back an AS-REP (Authentication Service Response), which contains an encrypted TGT.
    • Since this TGT is encrypted using the user’s NTLM hash, an attacker can capture it and use offline brute-force attacks to crack the password.

Detection Methods

  • Event ID 4768 (Kerberos Authentication Ticket Request) → Check for accounts requesting AS tickets without preauthentication.
  • Event ID 4771 (Kerberos Pre-authentication Failed) → Look for failures due to preauthentication attempts.
  • Unusual AS-REP requests for multiple users in a short timeframe may indicate an attack.

How the Attacker Used AS-REP Roasting in ShadowRoast Lab

  • The attacker executed BackupUtility.exe (Rubeus.exe) to enumerate accounts with preauthentication disabled.
  • They requested AS-REP responses for these accounts, capturing TGTs encrypted with user password hashes.
  • The attacker then brute-forced these hashes offline to retrieve plaintext credentials, allowing them to move further into the network.

2- DCShadow Attack

Overview

DCShadow is an advanced Active Directory attack that allows an attacker to replicate unauthorized changes to the AD database. It is part of the Mimikatz toolset and enables an attacker to register a rogue Domain Controller (DC) and push malicious modifications to the NTDS.dit database (which stores AD user data and passwords).

How It Works

  1. Understanding Active Directory Replication:
    • In a normal AD environment, Domain Controllers synchronize changes to ensure consistency across the network.
    • When an admin modifies an object (e.g., changes a user’s group membership), the change is replicated to all DCs using Directory Replication Service (DRS).
  2. DCShadow Attack Steps:
    • The attacker compromises a high-privilege account (such as Domain Admin or Enterprise Admin).
    • They use Mimikatz (DCShadow feature) to register a fake DC within the AD infrastructure.
    • Once registered, the rogue DC injects malicious changes into the directory and forces legitimate DCs to replicate these changes.
    • The attacker can then modify user permissions, create hidden admin accounts, or even disable security logs to maintain persistence.

Detection Methods

  • Event ID 5137 (Directory Service Object Creation) → Look for unauthorized additions of new DCs.
  • Event ID 4662 (Directory Service Access) → Check for suspicious modifications to high-value AD objects.
  • Event ID 4742 (Changes to privileged accounts) → Monitor for changes in sensitive admin accounts.
  • Event ID 4929 logs when an Active Directory (AD) replica source naming context is removed

a key indicator of DCShadow activity

  • Look for unusual Kerberos Ticket requests (Event ID 4769) from unknown sources.

How the Attacker Used DCShadow

  • The attacker used Mimikatz (DCShadow feature) with the compromised tcooper account.
  • They registered a rogue Domain Controller and pushed unauthorized Active Directory modifications.
  • This allowed them to manipulate security policies, escalate privileges, and create backdoors for future access.
