Important Statistics to Check

Wireshark's statistics tools can be helpful for general analysis .

  • Conversations: Displays active conversations between endpoints (IP or MAC). This is useful for identifying connections and traffic volumes.
    • Path: Statistics > Conversations
  • Protocol Hierarchy: Shows the breakdown of protocols used in the capture, making it easy to spot unusual or unexpected protocols.
    • Path: Statistics > Protocol Hierarchy
  • I/O Graphs: Provides a visualization of packet rates over time, which can reveal spikes or drops often associated with attacks or network events.
    • Path: Statistics > I/O Graphs

Filters

IP Address Filters

  • Specific IP address: ip.addr == <IP>
  • Destination IP: ip.dst == <IP>
  • Source IP: ip.src == <IP>
  • Subnet filter: ip.addr == 10.10.50.1/24
  • Filter out IP address: !(ip.addr == 10.10.50.1)

Port Filters

  • Specific port: tcp.port == <port> udp.port== <port>

Example: tcp.port == 80 : HTTP Traffic || tcp.port == 443 : HTTPS Traffic || udp.port == 53 : DNS Traffic

  • Destination port: tcp.dstport == 23
  • Source port: tcp.srcport == <port>

HTTP

  • Filter by URL: http.host == "<hostname>"

Example: http.host contains "malicious-domain"

  • HTTP response code: http.response.code == <code>

http.response.code == 200 successful request

http.response.code == 404 requested resource was not found.

  • Show only HTTP content-type data: http.content_type
  • All HTTP requests: http.request

http.request.method == "POST” || http.request.method == "PUT” ||

http.request.method == "GET” || http.request.method == "DELETE”

  • All HTTP responses: http.response
  • Finds HTTP requests with “login” in the URL http.request.uri contains "login”
  • look for sus user-agent http.user_agent contains "MaliciousAgent”
  • http.body contains ".bat"
  • http.authbasic : used to identify HTTP Basic Authentication traffic.
  • http.request.uri contains ".exe”
  • http.body contains "powershell"
  • http.request.uri matches ".*\.php”
  • http.request.uri contains "login”
  • http contains DOS Filter for Executable files

Tip : filter using http.request and http.response. These can reveal data such as credentials or tokens in headers (e.g., Authorization, Cookie).

MAC Address Filter

  • MAC address: eth.addr == <MAC address>

TCP Flag Filters

  • SYN flag: tcp.flags.syn == 1 and tcp.flags.ack == 0
  • RST flag: tcp.flags.reset == 1

TCP 3-Way Handshake :

  1. Client ➝ Server: SYN (0x002)

Filter: tcp.flags == 0x002

  1. Server ➝ Client: SYN-ACK (0x012)

Filter: tcp.flags == 0x012 ← this is what your filter catches

  1. Client ➝ Server: ACK (0x010)

Filter: tcp.flags == 0x010

DNS Filters

  • DNS: dns

Isolate DNS traffic .

  • DNS queries : dns.qry.name

Example: dns.qry.name contains "sus-domain"

(Use contains to match substrings in queried names.)

  • Show only DNS queries (requests): dns.flags.response == 0

Shows outgoing queries from clients.

  • Show only DNS responses (answers): dns.flags.response == 1

Shows replies from resolvers or other responders.

  • DNS transport (common):
    • UDP DNS: udp.port == 53
    • TCP DNS (large responses / zone transfers): tcp.port == 53
  • DNS query type: dns.qry.type == <type-number>

Common types:

    • dns.qry.type == 1 for A (IPv4) records
    • dns.qry.type == 28 for AAAA (IPv6) records
    • dns.qry.type == 15 for MX (mail exchange) records
    • dns.qry.type == 16 for TXT records
    • dns.qry.type == 5 for CNAME records

Example: dns.qry.type == 1 && dns.flags.response == 0 (outgoing A-type queries)

  • Filter DNS queries sent from a specific source IP:

ip.src == 192.168.1.1 && dns.flags.response == 0

(Shows DNS queries originating from the host 192.168.1.1.)

  • DNS responses that include a specific answer IP:

dns.flags.response == 1 && dns.a == 192.168.1.1

(Find replies that answer with the IPv4 192.168.1.1.)

  • Find responses by transaction ID:

dns.id == 0x1a2b

(Use the DNS transaction ID to correlate queries and responses.)

  • Look for low TTL values:

dns.flags.response == 1 && dns.resp.ttl <= 30

(Short TTLs can be suspicious in poisoning attempts.)

  • Search Info column for domain answers:

dns.flags.response == 1 && _ws.col.info contains "sus-domain.potato"

(Quick way to surface replies mentioning the domain in the Info column.)

FTP Filters

ftp

Displays all FTP protocol traffic.

ftp.request.command == "STOR" / ftp contains "STOR”

Initiates the upload of a file to the FTP server

ftp.request.command == "RETR”

Requests to retrieve (download) a file from the FTP server

ftp-data

Shows the actual data transfer packets in FTP sessions.

ftp.request.command == "USER" || ftp.request.command == "PASS”

Displays login attempts, including the username and password commands.

SMB Filters

  • SMB protocols: smb || smb2
  • ntlmssp.auth.username

Displays NTLM authentication username attempts.

  • ntlmssp.challenge

Shows NTLM challenge packets.

  • smb2.session_id

Displays SMB2 session IDs used for authentication.

  • smb2.filename

Filters packets that include filenames accessed via SMB.

  • smb.file

Filters SMB traffic with file names being accessed (SMB1).

  • smb.command == 0x06

Filters for the "Create" command, which is used to open files.

  • smb2.create

Filters SMB2 create requests for file/directory creation.

  • smb2.query_directory Displays queries for directory contents in SMB2.

TCP Filters

  • tcp.flags == 0x12 → Captures SYN-ACK packets (Can help detect open ports during TCP scans).
  • tcp.flags.syn == 1 and tcp.flags.ack == 0 → Captures SYN packets (Common in TCP SYN scans where attackers check for open ports).
  • tcp.flags.reset == 1 → Captures RST (Reset) packets (May indicate port scanning, failed connections, or active service enumeration).
  • tcp.dstport > 1024 → Captures TCP traffic to ports above 1024 (Useful for detecting suspicious outbound connections or C2 communication).
  • tcp.dstport > 60000 → Captures TCP traffic to very high-numbered ports (May indicate malware communication, covert channels, or exfiltration attempts).
  • ip.dst == <IP> && (tcp.flags.syn == 1 && tcp.flags.ack == 0) → Captures TCP connection attempts to a specific destination (Useful for detecting reconnaissance activity or unauthorized access attempts).
  • UDP Filters
    • udp && ip.src == <IP> → Captures UDP packets from a specific source (Can help identify UDP-based attacks like UDP scanning or reflection attacks).
  • ICMP Filters
    • icmp.type == 3 and icmp.code == 3 → Captures ICMP Destination Unreachable (Port Unreachable) messages (Can reveal UDP port scanning attempts or misconfigured services).

ARP Filters

  • ARP Request: arp.opcode == 1

Displays ARP Request packets.

  • ARP Reply: arp.opcode == 2

Displays ARP Reply packets.

  • Gratuitous ARP Responses: arp.isgratuitous

Shows unsolicited is-at packets used for self-advertising or to poison caches.

  • Gratuitous ARP: arp.src.proto_ipv4 == arp.dst.proto_ipv4

Displays Gratuitous ARP packets, typically used for self-advertising or detecting IP conflicts.

  • Filter Specific MAC and IP Mappings:

arp.src.proto_ipv4 == 192.168.1.1 && arp.hw.src == 00:11:22:33:44:55

Filters ARP traffic for a specific MAC address and IP address mapping.

  • Duplicate Address Frame: arp.duplicate-address-frame

Displays ARP packets related to duplicate address detection.

  • ARP Broadcasts: eth.dst == ff:ff:ff:ff:ff:ff && arp

Detects ARP packets broadcasted to all devices.

Investigating ARP / Man-in-the-Middle

  • All ARP traffic: arp

Quick start filter to isolate ARP only.

  • ARP replies claiming the gateway IP:

arp.opcode == 2 && arp.src.proto_ipv4 == 192.168.10.1

See who is advertising the gateway IP.

  • Replies for gateway but from a different MAC than the known router:

arp.opcode == 2 && arp.src.proto_ipv4 == 192.168.10.1 && eth.src != 02:aa:bb:cc:00:01

Replace 02:aa:bb:cc:00:01 with the legit router MAC to catch spoofing.

  • ARP frames from a suspected attacker MAC:

arp && eth.src == 02:fe:bb:cd:55:55

Track all ARP activity from a specific MAC.

  • Gratuitous ARP: arp.isgratuitous

Shows unsolicited is-at packets attackers use to poison caches.

  • Duplicate IP-to-MAC detection:

arp.duplicate-address-detected || arp.duplicate-address-frame

Flags duplicate-address events Wireshark detects.

  • Find ARP replies that contain a specific Info text:

arp.opcode == 2 && _ws.col.info contains "192.168.10.1 is at"

Quick-search using the Info column content.

  • Rapid ARP bursts (suspicious high frequency):

arp && frame.time_delta < 0.01

Flags ARP packets arriving less than 10 ms after the previous frame.

  • Check who is being targeted (victim MAC):

arp && eth.dst == 00:aa:bb:cc:dd:ee

See which ARP frames are delivered to a victim MAC.

Quick investigation checklist

  1. Start with arp to view timeline and volume.
  2. Run arp.opcode == 2 && arp.src.proto_ipv4 == <gateway-ip>. If multiple eth.src MACs appear, that is suspicious.
  3. Run arp.opcode == 2 && arp.src.proto_ipv4 == <gateway-ip> && eth.src != <legit-gateway-mac> to highlight spoofing replies.
  4. Check arp.isgratuitous for unsolicited replies the attacker may send repeatedly.
  5. Use arp.duplicate-address-detected || arp.duplicate-address-frame to confirm duplicate-address events.
  6. Correlate the attacker MAC with other traffic: eth.src == <attacker-mac> and inspect IP traffic to prove MITM.

SSH Filters

  • SSH protocol: ssh
  • SSH default port: tcp.port == 22
  • Auth requests: ssh.msg_code == 50
  • Key exchange init: ssh.msg_code == 20
  • Service request: ssh.msg_code == 5
  • Handshake/auth only: ssh.msg_code < 50

ICMP Filters

icmp

Shows all ICMP (IPv4) traffic.

icmp.type == 8

Shows Echo Request (outgoing ping requests). To catch the replies use icmp.type == 0 (Echo Reply).

icmp.type == 3

Destination Unreachable messages (useful for mapping blocked/invalid hosts). Note: type 3 has multiple codes (port unreachable, host unreachable, etc.).

icmp.type == 5

Redirect messages.

icmp.type == 11

Time Exceeded messages (useful for traceroute-like behaviour).

icmp.type == 8 && frame.len > 100

Flags packets with unusually large payloads. Normal pings are ~74 bytes total. Anything over 100 is suspicious.

Kerberos Filters

kerberos

Shows Kerberos protocol traffic (TCP/UDP port 88).

kerberos.msg_type == 10

AS-REQ (Authentication Service request) — client requests an initial TGT from the KDC. Useful to spot initial authentications / logon attempts.

kerberos.msg_type == 11

AS-REP (Authentication Service reply) — KDC responds to an AS-REQ (contains the TGT if authentication succeeded).

kerberos.msg_type == 12

TGS-REQ (Ticket-Granting Service request) — client requests a service ticket (commonly seen in Kerberoasting workflows when requesting SPN service tickets).

kerberos.msg_type == 13

TGS-REP (Ticket-Granting Service reply) — KDC replies with the service ticket.

kerberos.msg_type == 30

KRB-ERROR — Kerberos error messages (useful to detect failed or anomalous requests).

Tip: combine with ports or IPs, e.g.

kerberos && (tcp.port == 88 || udp.port == 88)

or

kerberos.msg_type == 12 && ip.addr == 10.0.0.5

LDAP Filters

ldap

Shows all LDAP protocol traffic (queries, binds, search results).

ldap.bind.request

Shows LDAP Bind requests (authentication to the DC). Useful to spot the start of an enumeration session.

ldap.search.request

Shows LDAP Search requests. High volume of these from a single client often indicates directory enumeration.

ldap.search.resultEntry

Shows individual search results returned by the server (useful to see which objects were enumerated).

ldap.unbindRequest_element or ldap.protocolOp == 2

Shows LDAP Unbind requests (session termination). This can help identify BloodHound-style collection where the client: Bind -> many Search requests -> Unbind (end).

ldap && (tcp.port == 389 || udp.port == 389)

Restrict LDAP to the default LDAP port to reduce noise.

tcp.port == 636

Shows LDAPS (LDAP over SSL) traffic. If LDAPS is used, LDAP fields may be encrypted and harder to inspect.

ldap.search.filter

Show the actual LDAP search filter used (e.g., (objectClass=*) or (&(objectClass=user)(memberOf=...))) to quickly see what the client is enumerating.

Tip: to detect a single host doing large enumeration, combine filters, for example:

ldap.search.request && ip.src == 10.0.0.5

or count search requests in a time window from the same source IP to the DC.

DHCP / BOOTP Filters

bootp

Shows BOOTP/DHCP traffic. Wireshark uses the bootp dissector for DHCP.

udp.port == 67 || udp.port == 68

Show DHCP traffic by UDP ports (server 67, client 68).

bootp.option.dhcp

Filters on the DHCP Message Type option

(the DHCP option that indicates Discover/Offer/Request/ACK, etc.). Use numeric values to narrow by message type (Wireshark exposes the DHCP message type under this field). Common message type numbers: 1 = Discover, 2 = Offer, 3 = Request, 5 = ACK.

bootp && bootp.option.dhcp == 1

Shows DHCP Discover messages (clients starting address acquisition).

bootp && bootp.option.dhcp == 2

Shows DHCP Offer messages (server offers an IP).

bootp && bootp.option.dhcp == 3

Shows DHCP Request messages (client requests offered IP).

bootp && bootp.option.dhcp == 5

Shows DHCP ACK messages (lease granted).

Tip: combine with MAC or IP filters when troubleshooting a single host, for example:

bootp && eth.src == 00:11:22:33:44:55

or

bootp && bootp.client_ip == 0.0.0.0 && ip.dst == 255.255.255.255

to focus on initial DHCP handshakes.

NTLMSSP Messages (NTLM authentication)

Look for NTLMSSP negotiation/auth messages in protocols that carry NTLM (SMB, HTTP Negotiate, etc.). In Wireshark you can filter for the NTLMSSP dissector:

ntlmssp

Shows NTLMSSP messages (Negotiate / Challenge / Authenticate).

Common message names you’ll see in the packet details: NTLMSSP_NEGOTIATE, NTLMSSP_CHALLENGE, NTLMSSP_AUTH — these reveal username, domain, and authentication details (challenge/response structures). To find NTLM over HTTP specifically you can also use:

http.auth && frame contains "NTLM"

Advanced Filters

These filters allow for more refined searches .

  • String Contains: field contains "string" # case sensitive #

    • frame contains "malware.exe”

    • Example: http.request.uri contains "login" – Finds HTTP requests where the URI contains "login".
  • Regex Match: field matches "regex" #case insensitive #
    • Example: http.request.uri matches ".*\.php" – Finds HTTP requests ending in .php.
  • Set Membership: field in {val1, val2, ...}
    • Example: ip.addr in {10.10.10.10, 192.168.1.1} – Filters packets matching any of the IPs listed.

Additional Tip

  • Follow Streams: Right-click on a packet and select “Follow TCP Stream” to view the entire conversation. This is especially useful for piecing together requests and responses.
