Persistence in the Windows Registry(T1547.001)

MITRE ATT&amp;CK

Registry Structure & Terminology

The Windows Registry functions similarly to a file system. Understanding its structure helps in identifying potential persistence mechanisms.

  • Keys & Sub-Keys → These act like folders and subfolders.
  • Values → These are like files inside a folder, containing configuration data.
  • Setting a Value → This is equivalent to adding content to a file.

Lists in the registry appear in three forms:

  1. Lists of Sub-Keys – Similar to a directory containing multiple subdirectories (e.g., a list of port monitors).
  2. A Key with Multiple Values – Functions like a folder containing multiple files (e.g., Run Keys).
  3. A Multi-Line Value – Each line represents an item (e.g., authentication packages for LSA).

Registry terminology example

List of values in the Run Key

Registry Run Keys & Malware Persistence

Windows Run Keys allow programs to start automatically when a user logs in. Attackers leverage these keys to execute malware upon system startup.

Important Registry Hives

The registry consists of several "hives" major sections storing system and user data. The two most important for persistence are:

  • HKEY_CURRENT_USER (HKCU) → Stores user-specific settings.
  • HKEY_LOCAL_MACHINE (HKLM) → Contains system-wide settings affecting all users.

Common Run Keys Used for Persistence

Attackers modify or create values in these registry keys to maintain persistence:

  • HKEY_LOCAL_MACHINE\Software\Microsoft\Windows\CurrentVersion\Run
  • HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Run

By adding a new value under these keys, malware ensures that it executes whenever a user logs in. This can be done using:

  • The Windows Registry Editor (GUI)
  • Command-line tools like reg.exe

Example of Malware Persistence in Registry

An attacker might add a registry entry like this:

text
HKCU\Software\Microsoft\Windows\CurrentVersion\Run
Malware -> C:\Windows\Temp\Malware.exe

This ensures that Malware.exe executes every time the user logs in.

Detecting Registry-Based Persistence

Windows provides auditing features to track registry modifications. The Security Event Log records certain event IDs that indicate registry access or changes.

Event IDEvent NameDescription
4656Handle to an object requestedIndicates access to a registry key
4657A registry value was modifiedTracks modifications to registry values
4658Handle to an object was closedLogs when a registry key is closed
4660An object was deletedDetects registry deletions
4663An attempt was made to access an objectMonitors access attempts

Among these, Event ID 4657 is particularly useful in tracking persistence mechanisms because it directly logs registry modifications.

Detecting Abnormal Access

look for unusual access patterns, such as:

  • PowerShell or CMD modifying registry run keys
  • Registry modifications outside of normal working hours
  • Unauthorized accounts making registry changes
  • Correlation with process creation (Event ID 4688) or registry success/failure (Event ID 4663)
Since registry modification auditing is not enabled by default, administrators must configure System Access Control Lists (SACLs) to monitor specific registry keys.

Methods of Registry-Based Persistence

Attackers often abuse the Windows Registry to maintain persistence. They can override existing keys, create new keys, or append entries to existing lists. Monitoring these changes is critical for threat hunting.

1- Overriding an Existing Key

Attackers can hijack a registry key that Windows relies on to launch system processes.

🔹 Example: UserInit Key Manipulation

  • Path: HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Winlogon\UserInit
  • Default Value: C:\Windows\system32\userinit.exe
  • Malware can modify this key to launch itself instead of or in addition to userinit.exe at login.

🔹 Detection:

  • Use Procmon (Process Monitor) to track registry modifications targeting keys ending in .exe.
  • Correlate registry changes with unexpected process launches at login.

🔹 Attacker Trade-Offs:

  • Simple to implement, but abnormal behavior may trigger alerts or be noticed by users.

2- Creating a New Key for Persistence

Instead of modifying existing keys, attackers can create new keys that legitimate applications load.

🔹 Example: Hidden Microsoft Office Key

  • Path: HKLM\Software\Microsoft\Office\Test\Special\Perf
  • Setting the default value to a malicious DLL path ensures Office loads the malware whenever it starts.

🔹 Detection:

  • Use Procmon to identify registry keys checked by applications but not present by default.
  • Monitor for newly created keys under application paths with unusual executable or DLL references.

🔹 Attacker Trade-Offs:

  • Malware executes within a legitimate process, increasing stealth.
  • Detection is harder but possible if baseline key inventory or application whitelists are monitored.

3- Adding an Item to a List

Some registry keys store lists rather than single values. Attackers can append malicious entries instead of replacing existing values.

🔹 Example: LSA Authentication Packages

  • Path: HKLM\SYSTEM\CurrentControlSet\Control\Lsa\Authentication Packages
  • Contains a list of DLLs loaded during authentication.
  • Adding a malicious DLL ensures it loads with system authentication processes.

🔹 Detection:

  • Monitor for unusual additions to registry lists.
  • Use Procmon to track registry enumeration events and compare against baseline list entries.
  • Correlate list modifications with process creation events of loaded DLLs.

🔹 Attacker Trade-Offs:

  • More stealthy than overwriting keys.
  • Malicious DLLs may still be detected by integrity monitoring or security baselines.

Additional Important Registry-Based Persistence Techniques

🔹 Image File Execution Options (IFEO) Hijacking

  • The IFEO registry keys allow debugging of applications.
  • Malware can modify these to launch a malicious debugger instead of the intended application.

🔹 AppInit DLLs

  • Windows can load DLLs from the registry into all user-mode processes.
  • Malware can exploit this by inserting a DLL path into the AppInit_DLLs key.

🔹 Winlogon Registry Keys

  • The Winlogon registry keys control login behavior.
  • Modifying the Shell or Userinit values can make malware persist through reboots.

🔹 COM Hijacking

  • Malware can modify Component Object Model (COM) registry keys to hijack legitimate processes.
  • Example: Changing the CLSID/ProgID of a system component to point to malicious code.

Tracking in Windows Event Logs

Event ID 4656 logs when a process requests access to an object, such as a registry key, file, or SAM. It contains four key sections:

  1. Subject: Identifies the user performing the action.
  2. Object: Specifies the accessed resource.
    • Object Server: Always "Security."
    • Object Type: Indicates if the object is a file, registry key, or SAM (registry keys are important for persistence).
    • Object Name: Displays the full path of the accessed registry key.
  3. Process Information: Shows the process that requested access (e.g., reg.exe, powershell.exe).
  4. Access Request Information: Lists the permissions requested (less relevant for investigations).
What is a Handle?

A handle is a reference (or pointer) to a system resource, such as a file, registry key, process, or thread. In Windows, when a program or user wants to interact with a resource, it must first request a handle from the operating system.

Conclusion

The Windows Registry is a critical vector for malware persistence, providing attackers with multiple avenues to maintain control over a system. By modifying existing keys, creating new keys, or appending entries to lists, adversaries can ensure their code executes automatically, often blending with legitimate system processes.
