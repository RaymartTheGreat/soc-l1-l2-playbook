powershell
 # List of all scheduled tasks, sorted by their creation date
PS C:\Users\Administrator> Get-ScheduledTask | Where-Object {$_.Date —ne $null —and $_.State —ne "Disabled"} | Sort-Object Date | select Date,TaskName,Author,State,TaskPath | ft
powershell
# List all enabled scheduled tasks with creation date and command to be executed, sorted by date and printing all additional information
$tasks = Get-ScheduledTask | Where-Object {$_.Date —ne $null —and $_.State —ne "Disabled" —and $_.Actions.Execute —ne $null} | Sort-Object Date

foreach ($task in $tasks) {
    $taskName = $task.TaskName
    $taskDate = $task.Date
    $taskPath = $task.TaskPath
    $taskAuthor = $task.Author
    $taskCommand = $task.Actions.Execute
    $taskArgs = $task.Actions.Arguments
    $taskRunAs = $task.Principal.UserId

    # Output service information
    Write-Host "Task Name: $taskName"
    Write-Host "Task Author: $taskAuthor"
    Write-Host "Creation Date: $taskDate"
    Write-Host "Task Path: $taskPath"
    Write-Host "Command: $taskCommand $taskArgs"
    Write-Host "Run As: $taskRunAs"
    Write-Host ""
}

Original Notes

Manual Inspection: Scheduled Tasks

As mentioned, analysing through event logs only applies when the advanced audit policies for scheduled task events are enabled. Given this prerequisite, we will proceed with the discussion of analysing scheduled tasks without relying on Windows Event Logs. First, let's leverage the GUI version of Task Scheduler by using CTRL+R (or access the Run dialogue box from the taskbar) and typing taskschd.msc.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/715bec86f7d8a6dcf26163f800d3b9a5.png

When investigating scheduled tasks, one good starting point is to look for the enabled ones. So, let's start this by clicking the Task Scheduler (Local) on the left sidebar.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/11f99a475379bdec7ac1b5f96cc06fc4.png

Based on the screenshot above, you will see the Active Tasks on the lower part of the window. This section shows all tasks that are currently enabled. Given the premise of hunting potentially malicious tasks, we can use this as a starting point to find unusual ones.

To pivot from this, we can build a simple methodology for determining an outlier:

  • Since the view is sorted based on the Next Run Time, inspect the soonest to be executed.
  • Based on the Task Name field, compare the naming convention used and select the ones that differ from a typical name.
  • Based on the Location, combine the observations above and see if there are potential anomalies. Typically, the location set will be \ ****if the location is not specified in the scheduled task's name during its creation.

Scenario: Investigating Persistence via Scheduled Tasks

Based on the alert generated by the machine up for investigation, one of the key detections is related to the scheduled task creation. According to some threat intelligence reports, threat actors are leveraging some typical applications installed as a name reference to hide persistence implants. Given that the workstations in your organization have Microsoft Office installed, you had the idea to review the scheduled tasks related to it. To continue your assumption, check the Office ClickToRun Service Monitor scheduled task for a quick example.

NOTE: You may click the back blue arrow button on the window's upper-left corner to return to the Active Tasks view.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/8dce03af7dae8ac1f218603b92541f6e.png

Based on the current view, we can infer the following information:

  • The task is executed at 4:00 AM daily and has a condition after being triggered.
  • The "Next" and "Last" run times and the creation date are shown.
  • The user context during execution is set to SYSTEM and the task runs whether the user is logged on.
  • Lastly, the task runs with the highest privileges.

In continuation, navigate to the Actions section to determine the command to be executed.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/467b535d4095bdeb6db8b869c606fb74.png

As shown above, the scheduled task starts a program and executes a binary named OfficeC2RClient.exe. Based on the binary's location, it is stored in a known directory. Sometimes, the scheduled task can already be deemed malicious based on the executable's location, as some threat actors store their malware in unusual locations, such as Temp or AppData folders. However, we should not always rely on this presumption based only on the location of the binary. If further investigation is needed to confirm the behaviour of the task, the next thing to do is to inspect the binary to conclude if it is malicious.

Lastly, the Settings section also provides additional insights that specify the exact behaviour of the task, such as run on demand, run as soon as possible, task failure settings, stop after a specific time, and auto-deletion.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/44ea757247bac0953372c34674bc58ba.png

Some threat actors leverage these settings to deploy the malicious scheduled tasks more persistently and covertly, and some use cases are:

  • Run task as soon as possible - This ensures that the malicious scheduled task is executed even if the specified schedule is not met, like when the machine is off during the scheduled time.
  • Task failure setting - This ensures that the malicious scheduled task will be executed even if it fails on its first attempt.
  • Stop after a specific time - This allows the task to stop its malicious activities and hide itself after its successful execution.
  • Auto deletion - This allows the malicious scheduled task to automatically clean itself up to remove any traces within the compromised machine.

You may have observed that inspecting the scheduled tasks one by one using this approach is tedious, but don't worry since we will introduce an optimal way to conduct this analysis in the succeeding sections.

Breaking Down the Scheduled Tasks

To understand scheduled tasks further, let's dive deep into their structure. To start with, scheduled tasks are stored in C:\Windows\System32\Tasks. Moreover, all details about the scheduled tasks you have seen are stored relatively from this location. Let's use the scheduled task above as an example.

Based on the details of the Office ClickToRun Service Monitor, its location is in \Microsoft\Office. Let's open a file explorer and navigate to the *C:\Windows\System32\Tasks\Microsoft\Office *directory.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/aacf90270a4b5e6bb4de06c8e7d9c04a.png

As you can see in the directory, there is also a file named Office ClickToRun Service Monitor. This file contains XML data that details the scheduled task. Try opening the file using a notepad to view its contents.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/f5460122fc860ef99aa1d250d0d0dc68.png

You may observe that the contents are similar to the information we gathered using the Task Scheduler application. Knowing this allows you to translate the knowledge in analysing scheduled tasks directly in cases where a full-disk image of the machine is given for an investigation.

Another thing to note on the directory view is the properties seen in the explorer view, like the file's last modified date.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/97d8a3383819b0a306635bb70db5de08.png

Moreover, the file's properties can also be checked to see the creation date of the task.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/8d7f04e52f16c2b0e0c25df56656363c.png

By checking the dates, it can be quickly confirmed when the task was created and last modified. Certain mismatches may allow you to determine if a scheduled task is modified with a potentially malicious command.

Utilising Command Line Utilities

In the previous sections, we have used the Task Scheduler application and Notepad to inspect and analyze the scheduled tasks. Now, let's divert into command-line utilities to optimize the investigation methodology.

For starters, fire up a PowerShell window and execute the command below to list all enabled scheduled tasks.

powershell
PS C:\Users\Administrator> Get-ScheduledTask | Where-Object {$_.State —ne "Disabled"}

Alternatively, you can also use schtasks.exe to get the same information. The command below lists all scheduled tasks, renders them in a CSV format and excludes all disabled tasks using findstr.exe.

powershell
PS C:\Users\Administrator> schtasks.exe /query /fo CSV | findstr /V Disabled

Next, let's list all enabled scheduled tasks sorted by their creation date. Moreover, we will output the creation date and author to add more context to each task.

powershell
 # List of all scheduled tasks, sorted by their creation date
PS C:\Users\Administrator> Get-ScheduledTask | Where-Object {$_.Date —ne $null —and $_.State —ne "Disabled"} | Sort-Object Date | select Date,TaskName,Author,State,TaskPath | ft

NOTE: To render the output in list view, replace the ft (Format-Table) command above with fl (Format-List).

Based on the output of the initial command, you may already observe some notable scheduled tasks. To view the additional details for a specific task, you may use the following commands to view its properties.

DetailsCommand
Command to be executed(Get-ScheduledTask -TaskName REPLACE_WITH_ACTUAL_TASKNAME).Actions
Execution Triggers(Get-ScheduledTask -TaskName REPLACE_WITH_ACTUAL_TASKNAME).Triggers
User Execution Context(Get-ScheduledTask -TaskName REPLACE_WITH_ACTUAL_TASKNAME).Principal

Note: We have enclosed the Get-ScheduledTask commands in parenthesis since we are accessing its Actions/Triggers/Principal properties.

Lastly, we can combine all these into a single script to list all scheduled tasks sorted by their creation date, together with all the significant information.

powershell
# List all enabled scheduled tasks with creation date and command to be executed, sorted by date and printing all additional information
$tasks = Get-ScheduledTask | Where-Object {$_.Date —ne $null —and $_.State —ne "Disabled" —and $_.Actions.Execute —ne $null} | Sort-Object Date

foreach ($task in $tasks) {
    $taskName = $task.TaskName
    $taskDate = $task.Date
    $taskPath = $task.TaskPath
    $taskAuthor = $task.Author
    $taskCommand = $task.Actions.Execute
    $taskArgs = $task.Actions.Arguments
    $taskRunAs = $task.Principal.UserId

    # Output service information
    Write-Host "Task Name: $taskName"
    Write-Host "Task Author: $taskAuthor"
    Write-Host "Creation Date: $taskDate"
    Write-Host "Task Path: $taskPath"
    Write-Host "Command: $taskCommand $taskArgs"
    Write-Host "Run As: $taskRunAs"
    Write-Host ""
}
