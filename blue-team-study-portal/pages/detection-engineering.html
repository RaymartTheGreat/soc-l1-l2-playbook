Original Notes

Detection Engineering

Detection Engineering is the backbone of modern security operations. As threats become more advanced and frequent, reactive security is no longer sufficient. Detection engineering shifts the approach: from generic alerting to precision-crafted, validated detection logic that targets adversarial behavior in context.

What is Detection Engineering?

Detection engineering is the continuous process of designing, building, testing, and maintaining detections that identify signs of adversary activity or security misconfigurations. This work bridges intelligence, operations, and engineering disciplines to ensure threat visibility is accurate, actionable, and sustainable.

It isn’t just about writing rules. It’s about understanding threats, mapping them to available data, and producing alert logic that’s resilient to evasion and scalable to change. It also requires a culture shift—security teams must work like developers: versioning rules, testing logic, automating pipelines, and continuously improving.

What Makes a Good Detection?

A good detection rule is not just one that "fires"; it’s one that contributes meaningfully to defense with clarity, reliability, and actionability. Below are the key factors that define an effective detection:

CriteriaDescription
Response FeedbackDoes the alert provide enough context for the response team to act? Is it missing key fields or logic?
False Positive Ratio / FidelityDoes the rule trigger too often on benign behavior? For example, 2 true positives out of 100 alerts indicates poor fidelity.
TimelinessWill the alert trigger in time for IR to respond and contain the activity before damage spreads?
SpecificityIs the rule too broad? Does it alert on unrelated or noisy activities? Specificity helps reduce fatigue and improve triage.
TestabilityCan the alert logic be tested using simulated or known inputs? This is essential for validation and tuning.
Compensating ControlsIf detection gaps exist, can this alert fill them by targeting visibility blind spots or tooling limitations?

Detection Types

Detection logic generally falls into one of two categories: environment-based or threat-based. Each serves a different function and complements the others.

Environment-Based Detections

These rely on understanding the structure, configuration, and behavior of the internal environment—what is normal, and what isn’t.

Configuration Detection

This form of detection evaluates current system, network, identity, or policy configurations against what is expected or compliant.

  • Best suited for static or well-defined environments.
  • Can quickly identify misalignments or insecure changes.
  • Challenges arise when configurations are undocumented or frequently changing, increasing false positives.

Configuration-based detections are especially useful when combined with other signals (e.g., combining misconfigured privileges with unusual process execution).

Modeling-Based Detection

Modeling-based detection relies on building behavioral baselines—tracking how systems, users, and services normally behave—and identifying outliers.

  • Good at detecting previously unknown threats without needing prior indicators.
  • Requires careful baseline curation to avoid learning malicious behaviors as “normal.”
  • Not context-rich on its own; often needs support from IR or threat intel for interpretation.

Ideal in stable environments where routine patterns are consistent and deviations are more suspicious by nature.

Threat-Based Detections

These focus directly on adversarial activity—matching specific indicators or behaviors associated with known tactics, tools, and procedures (TTPs).

Indicator-Based Detection (IOCs)

This involves using threat intelligence indicators—such as malicious domains, file hashes, or IP addresses—to trigger alerts.

  • Easy to create and deploy.
  • Useful in retrospective analysis and scoping post-breach.
  • Highly dependent on the freshness and context of the IOCs.
  • Often reactive—requires the threat to be observed first.

Example: Flagging outbound connections to a known C2 IP.

Baseline-Based Detection

Baseline-based detection focuses on spotting deviations from normal activity patterns within a given environment—such as typical user logon hours, process usage, or network behavior.

  • Works well in stable environments where behavior is predictable.
  • Useful for identifying insider threats or compromised accounts.
  • Must be tuned regularly to prevent alert fatigue from normal fluctuations.
  • Often supports behavior-based rules by helping define “normal.”

Example: Alerting when a user logs in from an unusual location or outside normal business hours.

Threat Behavior Detection

Rather than matching specific indicators, this detection type focuses on broader patterns of attacker behavior—based on tactics and techniques like credential dumping, lateral movement, or privilege escalation.

  • More resilient to changes in tools or infrastructure.
  • Lower false positive rates when designed well.
  • Requires more initial effort, baseline tuning, and context mapping.

Example: Detecting a parent-child relationship between winword.exe → powershell.exe regardless of the specific command.

Detection Engineering in Practice

Detection as Code (DaC)

Detection as Code is the practice of managing detections using software engineering principles—treating them like code rather than isolated configurations.

Instead of writing detections ad-hoc in the SIEM UI, engineers define them as versioned, testable artifacts in files (e.g., YAML, TOML). These files are stored in Git, tested via pipelines, and deployed through automation. This improves reproducibility, auditability, and team collaboration.

DaC practices typically include:

  • Version control of detection rules (e.g., via GitHub or GitLab)
  • CI/CD pipelines to test and validate rules before pushing to production
  • Vendor-agnostic formats like Sigma or YARA for portability
  • Reusable logic patterns to avoid duplication

DaC shifts detections from reactive operations to structured, predictable workflows—just like software development.

python
    A[Threat Modeling or Request] --> B[Write Detection Rule]
    B --> C[Store in Git / Version Control]
    C --> D[CI Pipeline: Lint + Test]
    D --> E[Validation in Lab or Simulated Attacks]
    E --> F[Deploy to Production]
    F --> G[Monitor + Tune + Review]
    G --> B

Detection Gap Analysis

Gap analysis is a structured method for identifying what detection coverage is missing—before attackers do.

There are two ways to approach it:

  • Reactive:

After an incident, examine what was missed and adjust detections accordingly.

  • Proactive:

Use frameworks like MITRE ATT&CK to map likely attacker techniques and assess if telemetry and rules exist for each.

This helps prioritize new detections based on actual threat relevance and closes the gap between assumed and real visibility.

Baseline Creation

A baseline is a documented profile of what "normal" looks like for systems, users, and services. Good baselining enables behavior-based detection and reduces alert fatigue.

Types of Baselines:

  • High-Level Baselines:

Policy-driven, platform-agnostic expectations.

Example: All endpoints must have EDR installed, MFA must be enabled.

  • Technical Baselines:

Detailed, system-specific configuration and activity patterns.

Example: Windows hosts normally generate 4624 logins from the helpdesk subnet, but never from random external IPs.

Baselines must evolve with the environment. Collaboration across departments is key—IT, security, and operations must contribute.

Rule Writing

At the core of detection engineering is writing rules that identify malicious or suspicious activity in logs.

  • Match behaviors or conditions in log data
  • Avoid unnecessary noise (false positives)
  • Include context for response

Example Rule Approaches:

  • Sigma: Write in a generic format, then convert to Splunk, Elastic, or other platforms
  • YARA: Detect patterns in binary or memory artifacts
  • Snort/Suricata: Focus on packet-level network detection

A good detection rule typically includes:

  • Clear description of what it detects
  • Log source and required fields
  • Condition logic
  • Known false positive patterns
  • Assigned severity or priority

Example:

Sigma Rule: Detecting certutil.exe File Downloads

yaml
title: Certutil Downloading File
id: 8a1b3c4d-5e6f-7a8b-9c0d-1e2f3a4b5c6d
status: stable
description: Detects the use of certutil.exe to download files from the internet, which is a common technique used by attackers to retrieve malicious payloads.
author: SigmaHQ
date: 2021/05/15
references:
  - https://attack.mitre.org/techniques/T1105/
tags:
  - attack.command_and_control
  - attack.t1105
logsource:
  product: windows
  category: process_creation
detection:
  selection:
    Image|endswith: '\certutil.exe'
    CommandLine|contains:
      - 'http'
      - '-urlcache'
      - '-split'
  condition: selection
falsepositives:
  - Legitimate use of certutil for certificate management
level: medium

Validation and Testing

Detection logic must be verified in a test or lab environment before production use. The goal is to confirm that:

  • The rule triggers when it should
  • It avoids false positives
  • It’s resilient to slight changes in attacker behavior

Example Methods:

  • Use Atomic Red Team or Caldera to simulate known TTPs
  • Manually replay logs or events
  • Review triggered logs and validate fields
  • Maintain unit-test cases for new and existing rules

Detection validation should be repeatable and integrated into the CI/CD process when Detection as Code is used.

Deployment, Automation, and Tuning

After validation, detections are deployed in production with continuous monitoring. This phase is not a one-time operation—it’s iterative.

  • Monitor alert volume and quality
  • Tune thresholds or add filters to reduce false positives
  • Adjust logic as threat behavior changes
  • Review detections periodically for gaps or decay

Good engineering practices like logging coverage metrics and tagging detections (e.g., ATT&CK.T1059) help teams manage complexity at scale.

Detection Strategies

Implementing Tripwires and Honeypots

Tripwires are deliberately placed artifacts or configurations designed to catch adversary activity early—especially when normal defenses fail or when behavior isn't otherwise suspicious.

Common Tripwire Types:

  • Honeypots: Decoy services or hosts that no legitimate user should touch. Any activity is likely malicious.
  • Hidden Files and Folders: Planted in known crawl paths. If touched, signals malware or automated scanning.
  • Object Access Auditing: Windows Event ID 4663 logs access to sensitive files; if access is unexpected, alerts can be generated.

These are not substitutes for regular detection—they are supplementary mechanisms for catching unknown techniques or attacker curiosity.

python
    A[Decoy File Accessed] --> B[Windows Audit Policy Logs Event ID 4663]
    B --> C[Log Forwarded to SIEM]
    C --> D[SIEM Rule Matches Trigger]
    D --> E[Alert Sent to Analysts]

IOC Transformation Using Sigma

When incident responders identify IOCs (e.g., malicious domains, file hashes), detection engineers can quickly convert them into actionable detection rules.

Instead of pasting IOCs into search queries, Sigma lets you write portable rules like this:

yaml
title: Executable Download from Suspicious Domains
logsource:
  category: proxy
detection:
  selection:
    c-uri-extension: ['exe']
    r-dns:
      - 'bad3xe69connection.io'
      - 'kind4bad.com'
  condition: selection

This rule can be automatically converted to Elastic or Splunk syntax using tools like Uncoder.

Only use IOC-based rules when you have contextual intel and expiry plans—they degrade fast.

Using Public Sigma Repositories and Uncoder

Instead of starting from scratch, detection engineers can leverage public Sigma rules submitted by the community—especially for 0-days.

Examples:

  • Follina (CVE-2022-30190) – Microsoft Office remote code execution
  • Log4Shell (CVE-2021-44228) – Java-based remote shell exploitation

These rules are available from:

  • SigmaHQ GitHub
  • Vendor-specific detection libraries
  • Shared threat intel platforms

Uncoder allows direct translation from Sigma to SIEM queries (Elastic, Splunk, etc), but results should always be tested and refined in-house.

Alerting & Detection Strategy (ADS Framework)

Palantir’s ADS Framework is a structured template for designing, documenting, and validating detection rules.

Key components:

  1. Goal – What behavior is the alert designed to detect?
  2. Categorization – Mapped to MITRE ATT&CK or other frameworks
  3. Strategy Abstract – Overview of logic, log source, enrichment
  4. Technical Context – Infra-specific details (platforms, versions)
  5. Blind Spots – Where the alert could fail
  6. False Positives – Expected and how to handle them
  7. Validation – Method to test and trigger a real alert
  8. Priority – Alert severity and urgency
  9. Response – Analyst guidance for triage

The framework improves alert consistency and avoids duplicated or shallow detections.

Detection Maturity Level (DML)

The Detection Maturity Level (DML) model by Ryan Stillions measures how advanced an organization’s detections are—ranging from simple IOC matches to identifying attacker strategy.

Simplified DML Levels:

LevelDetection Capability
DML-0No detection at all
DML-1Atomic indicators (e.g., IPs, hashes)
DML-2Host/network artifacts (e.g., file paths, registry keys)
DML-3Tool identification (e.g., PsExec, Cobalt Strike)
DML-4Adversary procedures (e.g., multi-step sequences)
DML-5Techniques (e.g., credential dumping, lateral movement)
DML-6Tactics (e.g., persistence, privilege escalation)
DML-7Adversary strategy and planning
DML-8Adversary goals and mission intent

Aim to raise the maturity of your detections progressively—moving from reactive IOC matches to context-aware behavior-based rules

python
    A[DML-0: No Detection] --> B[DML-1: Atomic Indicators]
    B --> C[DML-2: Host/Network Artifacts]
    C --> D[DML-3: Tool Detection]
    D --> E[DML-4: Procedures]
    E --> F[DML-5: Techniques]
    F --> G[DML-6: Tactics]
    G --> H[DML-7: Strategy]
    H --> I[DML-8: Goals]
