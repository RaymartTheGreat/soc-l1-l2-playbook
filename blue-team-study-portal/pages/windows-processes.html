Original Notes

Windows processes

Windows processes are programs running in the background that carry out system operations and execute programs or applications on a Windows operating system. Each process has its own memory space and allocated resources. In the Windows environment, various activities such as account logins, file access, memory allocation, running DLLs, and application execution are tied to processes.

Processes run in memory with their associations, and you can view the running processes on a live system using tools like:

Command-line: Tasklist tool

Graphical User Interface (GUI): Task Manager or Process Hacker

brief about each column of task manager :

  • Type - Each process falls into 1 of 3 categories (Apps, Background process, or Windows process).
  • Publisher - Think of this column as the name of the author of the program/file.
  • PID - This is known as the process identifier number. Windows assigns a unique process identifier each time a program starts. If the same program has multiple running processes, each will have its unique process identifier (PID).
  • Process name - This is the file name of the process. In the above image, the file name for Task Manager is Taskmrg.exe.
  • Command line - The full command used to launch the process.
  • CPU - The amount of CPU (processing power) the process uses.
  • Memory - The amount of physical working memory utilized by the process.

Task Manager is a utility you should be comfortable using, whether you're troubleshooting or performing analysis on the endpoint.

Let's move to the Details tab. This view provides some core processes . Sort the PID column so that the PIDs are in ascending order.

Add some additional columns to see more information about these processes. Good columns to add are Image path name and Command line.

These two columns can quickly alert an analyst of any outliers with a given process. Key Attributes of a Windows Process:

  1. Process Name: Name of the executed program.
  2. Process ID (PID): Unique dynamic identifier for each process.
  3. Process Path: File location from which the process is executed.
  4. Username: Indicates the user context under which the process is running (affects security privileges).
  5. Command-Line Arguments: Specifies options or switches for the process.
  6. Parent Process: The process that spawned the current process.

Windows Process Types

Windows processes are broadly classified into two categories:

  1. Standard Windows Processes: Developed by Microsoft, these processes support core system operations like boot, login, and services.
  2. Non-Standard Windows Processes: Custom or third-party processes not included in default Windows installations. These may be legitimate (custom software) or malicious (malware).

Common Standard Windows Processes

Understanding standard Windows processes and their behavior is crucial for Security Operations Center (SOC) analysts, threat hunters, and incident responders. Adversaries often mimic, inject into, or hijack these processes to avoid detection. Recognizing legitimate process behavior helps distinguish normal activity from anomalies that may indicate malware, persistence mechanisms, or lateral movement.

System

  • Process Name: System
  • Path: N/A (kernel-level)
  • Username: SYSTEM
  • Instances: One
  • Parent Process: N/A (spawned by the kernel)
  • Normal Behavior: Handles kernel-level tasks, hardware abstraction, and core system scheduling.
  • Red Flags:
    • Should never have a visible GUI or outbound network activity.
    • Unexpected termination or modification may indicate a rootkit or driver exploit.

Session Manager Subsystem (smss.exe)

  • Path: %SystemRoot%\System32\smss.exe
  • Username: SYSTEM
  • Instances: One primary + one per user session
  • Parent Process: System
  • Normal Behavior: Initializes user sessions and environment variables, launches csrss.exe, winlogon.exe, and wininit.exe.
  • Red Flags:
    • Running from incorrect path (e.g., C:\Temp\smss.exe)
    • Running persistently outside boot phase

Client/Server Runtime Subsystem (csrss.exe)

  • Path: %SystemRoot%\System32\csrss.exe
  • Username: SYSTEM
  • Instances: One per active session
  • Parent Process: smss.exe
  • Normal Behavior: Manages user-mode side of Win32 subsystem (console windows, threads).
  • Red Flags:
    • More than one per session
    • Running from unusual directory
    • Engaging in network communication

Windows Initialization (wininit.exe)

  • Path: %SystemRoot%\System32\wininit.exe
  • Username: SYSTEM
  • Instances: One
  • Parent Process: smss.exe
  • Normal Behavior: Initializes essential subsystems during boot: starts services.exe and lsass.exe.
  • Red Flags:
    • Missing or altered during boot = system integrity issues
    • Rarely abused, but monitor for name spoofing

Windows Logon (winlogon.exe)

  • Path: %SystemRoot%\System32\winlogon.exe
  • Username: SYSTEM
  • Instances: One per interactive user session
  • Parent Process: smss.exe
  • Normal Behavior: Handles user logon/logoff, lock screen, and credential input.
  • Red Flags:
    • Running under a different user
    • Spawned from abnormal parent
    • Being injected into by unknown processes

Local Security Authority Subsystem (lsass.exe)

  • Path: %SystemRoot%\System32\lsass.exe
  • Username: SYSTEM
  • Instances: One
  • Parent Process: wininit.exe
  • Normal Behavior: Manages authentication, access tokens, and Active Directory communication.
  • Red Flags:
    • Targeted for credential dumping (e.g., Mimikatz)
    • Dump file creation or suspicious memory access
    • Unusual outbound connections

Service Control Manager (services.exe)

  • Path: %SystemRoot%\System32\services.exe
  • Username: SYSTEM
  • Instances: One
  • Parent Process: wininit.exe
  • Normal Behavior: Responsible for starting/stopping system services.
  • Red Flags:
    • Anomalous child processes (e.g., cmd.exe, powershell.exe)
    • Services registered with unusual binaries

Service Host (svchost.exe)

  • Path: %SystemRoot%\System32\svchost.exe
  • Username: SYSTEM, LOCAL SERVICE, or NETWORK SERVICE
  • Instances: Multiple (based on hosted services via k switch)
  • Parent Process: services.exe
  • Normal Behavior: Container process for Windows services, grouped by functionality.
  • Red Flags:
    • Running from incorrect path
    • Hosting unexpected or malicious DLLs
    • Engaging in network activity unrelated to hosted services

Runtime Broker (RuntimeBroker.exe)

  • Path: %SystemRoot%\System32\RuntimeBroker.exe
  • Username: Logged-in user
  • Instances: One or more
  • Parent Process: svchost.exe
  • Normal Behavior: Manages permissions for UWP apps (Windows Store apps).
  • Red Flags:
    • Running without any UWP apps open
    • Consistent high CPU/memory usage (may indicate abuse)
    • Spawned by unknown parent

Windows Explorer (explorer.exe)

  • Path: %SystemRoot%\explorer.exe
  • Username: Logged-in user
  • Instances: One per interactive session
  • Parent Process: winlogon.exe or user-launched
  • Normal Behavior: Handles desktop, taskbar, and file browser UI.
  • Red Flags:
    • Running multiple times for a single user
    • Executing child processes like powershell.exe, cmd.exe, or rundll32.exe
    • Running from an unexpected path (e.g., %AppData%)

Note:

Baseline is key: Learn what’s normal for your environment (e.g., number of svchost.exe instances, typical user behavior).

Paths matter: Legitimate processes running from the wrong path are often signs of masquerading.

Windows Process Tracking Events

Microsoft Windows allows tracking of process creation and termination activities through Event IDs:

  • Event ID 4688: Logs every process creation activity.
  • Event ID 4689: Logs every process exit activity.

These events are critical for incident response, SOC analysis, and threat hunting, helping detect and investigate anomalies in process behavior.

Event ID 4688: A New Process Has Been Created

Details Captured:

  1. Creator Subject: Information about the user and login session that initiated the process, including the parent process owner.
  2. Target Subject: If the new process owner matches the creator, fields are empty; otherwise, details of the new process owner are recorded.
  3. Process Information:
    • New Process ID: Tracks process activities and correlates events.
    • New Process Name: Full path of the created process.
    • Token Elevation Type: UAC-related token, with three types:
      • Type 1: Full token, no privilege removal.
      • Type 2: Elevated token, administrative privileges retained.
      • Type 3: Limited token, administrative privileges removed.
    • Mandatory Label: Indicates process integrity levels (e.g., low, medium, high, system).
    • Creator Process ID/Name: Tracks parent process activities and relationships.
    • Process Command Line: Captures the arguments used; crucial for identifying misuse of processes (e.g., cmd.exe).

Note: Logging command-line arguments requires enabling group policies:

  • Administrative Templates > System > Audit Process Creation > Include command line in process creation events.

Event ID 4689: A Process Has Exited

Details Captured:

  1. Subject: User and login session under which the process was running.
  2. Process Information: Exited process's ID, name, and full path.

Event ID 4689 enables correlation with Event ID 4688 (via Process ID) to determine process execution duration.

Additional Investigation Techniques

  • Event Log Correlation

Cross-reference process creation logs (e.g., Event ID 4688, Sysmon Event ID 1) with:

    • File creation/modification (Event ID 11)
    • Network connections (Sysmon Event ID 3)
    • Registry modifications (Event ID 13)
  • Automation & Monitoring Tools

Use telemetry and behavioral detection platforms to monitor and alert on deviations:

    • Sysmon: Granular visibility into process and system activity.
    • HELK / Elastic SIEM: Enable centralized log aggregation and detection rules.
    • Sigma Rules: Standardized detection signatures that can be translated to SIEM queries.

Common Evasion and Abuse Techniques

  1. Masquerading (Hiding in Plain Sight)

Attackers may rename binaries to resemble legitimate system processes.

    • Example: svch0st.exe, expl0rer.exe, or lsas.exe
  • Living Off the Land Binaries (LOLBins)

Use of trusted native tools to execute malicious tasks without dropping new binaries.

    • Common examples: powershell.exe, cmd.exe, rundll32.exe, wscript.exe, regsvr32.exe
  • Unusual Parent-Child Relationships

Pay attention to suspicious or unauthorized spawning of processes.

    • Example: winword.exe spawning cmd.exe or powershell.exe
    • Abnormal: svchost.exe starting a user interface binary like explorer.exe
  • Suspicious Execution Paths

Even when the binary name is correct, location matters.

    • Legitimate: C:\Windows\System32\svchost.exe
    • Suspicious: C:\Users\Public\svchost.exe
