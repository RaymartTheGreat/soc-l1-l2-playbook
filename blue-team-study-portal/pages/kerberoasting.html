Original Notes

Kerberoasting

Kerberoasting is an attack targeting service accounts within Active Directory by exploiting Service Principal Names (SPNs). In this attack, an adversary requests Kerberos service tickets, extracts the encrypted credential data, and then attempts to crack the ticket offline to recover the plaintext password.

  • Service Principal Name (SPN)

    • is a unique identifier for a service in Active Directory (AD). It is used by Kerberos authentication to associate a service with a specific account (usually a service account)

How SPNs Work:

    • When a client needs to authenticate to a service (e.g., SQL Server, web applications, or file shares), it uses the SPN to request a Kerberos ticket.
    • The Domain Controller (DC) checks the SPN to find which account is running that service.
    • The Kerberos Ticket Granting Service (TGS) then issues a service ticket encrypted with the service account’s password hash.

MITRE ATT&amp;CK ID: T1558.003

1- How Kerberoasting Works

a. Service Ticket Request

  • Identify SPNs:

The attacker searches for service accounts with associated SPNs. These SPNs uniquely identify network services (e.g., SQL Server, Exchange).

Example Command (PowerShell):

``jsx Get-ADUser -Filter {ServicePrincipalName -ne "$null"} -Properties ServicePrincipalName ``

  • Ticket Request:

For each identified SPN, the attacker requests a Kerberos service ticket from the Ticket Granting Service (TGS). The issued ticket is encrypted using the service account’s password hash.

User with SPN found

b. Ticket Retrieval and Offline Cracking

  • Ticket Extraction:

Tools such as Rubeus or Mimikatz are used to extract the service ticket from the target system’s memory.

Example Command (Rubeus):

``python Rubeus.exe kerberoast ``

  • Offline Cracking:

The attacker extracts the encrypted hash from the ticket and employs offline cracking tools like Hashcat to retrieve the plaintext password.

2- Attack Impact

  • Privilege Escalation:

Cracking a service account’s password may allow an attacker to gain high-level privileges, especially if the account is misconfigured with excessive rights.

  • Lateral Movement:

With the compromised service account credentials, attackers can move laterally within the network to access additional systems and data.

  • Persistence:

Weak or long-lived passwords increase the risk of ongoing unauthorized access to critical resources.

3- Key Tools and Techniques

  • Rubeus:

Used for both requesting and harvesting Kerberos tickets.

  • PowerShell:

Employed for SPN enumeration and initial ticket requests.

  • Mimikatz:

Can be used to dump tickets from memory and assist in offline extraction.

  • Offline Cracking Tools:

Tools like Hashcat are used to perform brute-force or dictionary attacks on the captured ticket hash.

4- Mitigation Strategies

  • Use Strong, Complex Passwords:

Ensure that service account passwords are long and complex to resist offline cracking efforts.

  • Implement Managed Service Accounts (MSAs):

MSAs reduce the risk of weak or stale passwords and automate password management.

  • Enforce Kerberos Pre-Authentication:

Require valid pre-authentication data to limit unauthorized ticket requests.

  • Monitor TGS Requests:

Use SIEM systems to detect abnormal volumes of TGS requests, particularly for high-value service accounts.

  • Restrict Service Account Privileges:

Limit each service account to only the permissions necessary for its function.

  • Network Segmentation and Service Account Monitoring:

Segment your network and continuously audit service account activity to quickly identify anomalous behavior.

5- Detection and Defensive Strategies

Detection

  • Service Ticket Requests (Event ID 4769)

Focus on TGS requests that use RC4 encryption (etype 0x17), since these are commonly targeted for offline cracking.

  • Filtering Out Noise:

Exclude requests for the krbtgt account, machine accounts (ending in $), and well-known noisy service accounts (e.g., sqlsvc, generic svc-* accounts). This reduces false positives.

  • Abnormal Patterns:

Look for spikes in ticket requests against service accounts or repeated RC4-based TGS requests from a single host.

  • Example Detection Query:

``jsx (event.code: 4769 OR event.code: 4768) AND winlog.event_data.TicketEncryptionType: "0x17" AND NOT winlog.event_data.ServiceName: "krbtgt" AND NOT winlog.event_data.TargetUserName: ("sqlsvc" OR "svc-" OR $*) ``

This query isolates RC4-HMAC TGS requests while filtering out noise, making it easier to spot Kerberoasting attempts.

Defensive Strategies

  • Disable RC4 Where Possible:

Enforce stronger encryption types in Kerberos (AES-based), reducing the usefulness of Kerberoasting.

  • Service Account Hygiene:

Use long, complex passwords for service accounts, and rotate them regularly. Avoid giving unnecessary privileges to these accounts.

  • Monitoring and Alerting:

Configure SIEM rules to detect unusual volumes of TGS requests against service accounts, especially if they all use RC4.

6- Summary

  • Protocol Targeted: Kerberos
  • Authentication Material: Service tickets (encrypted with the service account’s password hash)
  • Scope of Attack: Credential compromise leading to lateral movement, privilege escalation, and persistent access in a Kerberos-based environment
  • Key Tools: Rubeus, PowerShell, Mimikatz, offline cracking tools (e.g., Hashcat)
