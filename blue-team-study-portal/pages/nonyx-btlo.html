Notes/Tips

  • SSDT (System Service Descriptor Table) is a data structure in the Windows operating system that manages system calls. It holds pointers to functions responsible for handling core system services such as file operations, memory management, and process control. Through the SSDT, user-mode applications can request kernel-mode services via system calls.
  • The modscan plugin identifies LDRDATATABLEENTRY structures by scanning physical memory for pool tags. This allows it to detect previously unloaded drivers as well as drivers that have been hidden or unlinked by rootkits.

Q2) What dump file in the malfind output directory corresponds to the memory address identified for code injection? (Format: Output File Name)

Adding --dump-dir to dump the malfind output (dump his memory address from the memory dump) and the name is the answer of the question

Q3) Which full filename path is referenced in the strings output of the memory section identified by malfind as containing a portable executable (PE32/MZ header)? (Format: Filename Path)

C:\WINDOWS\system32\drivers\str.sys

I used strings commands and scroll down till i saw the full path

Q4) How many functions were hooked and by which module after running the ssdt plugin and filtering out legitimate SSDT entries using egrep -v '(ntoskrnl|win32k)'? (Format: XX, Module)

14, 00004A2A

using ssdt | egrep -v '(ntoskrnl|win32k)’ this command used to list SSDT (System Service Descriptor Table) and filtering out legitimate SSDT then we can see the 14 functions hooked by 00004D2A module

SSDT (System Service Descriptor Table) is a data structure in the Windows operating system that manages system calls. It holds pointers to functions responsible for handling core system services such as file operations, memory management, and process control. Through the SSDT, user-mode applications can request kernel-mode services via system calls.

Q5) Using the modules (or modscan) plugin to identify the hooking driver from the ssdt output, what is the base address for the module found in Q4? (Format: Base Address)

0xff0d1000

as we already know the module we use modscan | grep "00004A2A”

note it didn’t work for me so i have to dump the output first then use grep

The modscan plugin identifies LDR_DATA_TABLE_ENTRY structures by scanning physical memory for pool tags. This allows it to detect previously unloaded drivers as well as drivers that have been hidden or unlinked by rootkits.

we use the command moddump -b 0xff0d1000 —dump -dir=output to dump the driver file then use sh256sum to get the answer
