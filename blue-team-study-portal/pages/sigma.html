Original Notes

Sigma

Sigma is an open-source, generic signature format for log-based detection. It uses YAML to define structured rules, enabling security analysts to detect threats across various SIEMs. Sigma rules specify log sources, conditions, and metadata, making them adaptable to different query languages. Essentially, Sigma standardizes log analysis, much like Snort does for network traffic and YARA for files.

Key Objectives

  • Share Detection Methods: Enable sharing of signatures, IOCs, and Yara rules.
  • Avoid Vendor Lock-In: Write SIEM searches that are not tied to a single vendor.
  • Enhance Threat Intelligence: Facilitate sharing among threat intelligence communities.
  • Custom Rule Creation: Develop detection rules for specific malicious behaviors.

Sigma Rule Structure

Sigma rules are written in YAML and include several key components:

1- Rule Format

  • YAML Basics: (https://www.tutorialspoint.com/yaml/yaml_quick_guide.htm)
    • Case-Sensitive: YAML is case-sensitive.
    • File Extension: Use the .yml extension.
    • Indentation: Use spaces (not tabs).
    • Comments: Precede with the # character.
    • Key-Value Pairs: Use the colon (:) to separate keys and values.
    • Arrays: Denote list items with a dash ( - ).

2- Core Components of a Sigma Rule

  • Title: A short, clear name for the rule.
  • ID: A globally unique identifier in UUID format.
  • Status: Indicates the rule's maturity. Common statuses include:
    • Stable: Ready for production.
    • Test: Under trial.
    • Experimental: May be noisy or imprecise.
    • Deprecated: Replaced by newer rules.
    • Unsupported: Not usable in its current state.
  • Description: Provides detailed context about the malicious activity.
  • Logsource: Describes the origin of the log data, including:
    • Product: e.g., Windows, Apache.
    • Category: e.g., firewall, web, antivirus.
    • Service: Specific components, e.g., sshd on Linux.
  • Detection: The heart of the rule, divided into:
    • Selection: Fields and values to match (using lists for OR, maps for AND).
    • Condition: Logical expression that determines which events trigger an alert.
  • False Positives: Lists known benign scenarios that might trigger the rule.
  • Level: Indicates the severity (Informational, Low, Medium, High, Critical).
  • Tags: Categorizes the rule, often with references to MITRE ATT&CK tactics and techniques.
  • References: External links for further context.

Official Repository

The Sigma GitHub repository is the official hub for the project. It houses the source code, documentation, public rules, test cases, and conversion tools. This repository is not only a resource for downloading the latest Sigma rules but also serves as a collaborative platform where contributors can propose improvements and share custom rules.

Detection Syntax and Logic

Search Identifiers

  • Lists vs. Maps:
    • Lists: Use a hyphen to denote multiple values (logical OR).
    • Maps: Use key/value pairs for matching (logical AND).

Value Modifiers

  • Transformation Modifiers: Adjust how values are matched.
    • contains: Matches if the value appears anywhere in the field.
    • all: Requires all listed values to match.
    • endswith / startswith: Match values at the end or beginning.
    • base64: Used for Base64-encoded values.
  • Type Modifiers:
    • re: Indicates a regular expression match (currently the only supported type modifier).

Conditional Expressions

  • Logical operators (AND, OR, NOT) combine search identifiers.
  • Conditions can reference multiple identifiers and filters to fine-tune the detection.

Example Rules

WMI Event Subscription Rule

yaml
title: WMI Event Subscription
id: 0f06a3a5-6a09-413f-8743-e6cf35561297
status: test
description: Detects creation of WMI event subscription persistence method.
logsource:
   product: windows
   category: wmi_event
detection:
  selection:
    EventID:
      - 19
      - 20
      - 21
  condition: selection
falsepositives:
  - Exclude legitimate (vetted) use of WMI event subscription in your network
level: medium
tags:
  - attack.persistence
  - attack.t1546.003

AnyDesk Installation Detection Scenario

Scenario Overview

An adversary might silently install AnyDesk normally a legitimate remote tool to gain unauthorized access. The rule detects such installations by looking for specific installation commands and context in Windows process creation logs.

Steps to Create the Rule:

  1. Intel Analysis:
    • Identify key elements: download source, destination file, installation commands (e.g., -install, -start-with-win), registry edits, and persistence mechanisms.
  2. Rule Identification:
    • Title: AnyDesk Installation
    • Status: Experimental (for internal testing)
    • Description: Detects potential AnyDesk installation used for remote access.
  3. Log Source:
    • Target Windows logs, specifically process creation logs.
  4. Detection:
    • CommandLine: Must contain both -install and -start-with-win.
    • CurrentDirectory: Must include C:\ProgramData\AnyDesk.exe.
    • Condition: All specified criteria must be met.
  5. Metadata:
    • False Positives: Legitimate deployments.
    • Level: High
    • References: e.g., TheDFIRReport Tweet
    • Tags: attack.command_and_control, attack.t1219
yaml
title: AnyDesk Installation
status: experimental
description: AnyDesk Remote Desktop installation can be used by an attacker to gain remote access.
logsource:
    category: process_creation
    product: windows
detection:
    selection:
        CommandLine|contains|all:
            - '--install'
            - '--start-with-win'
        CurrentDirectory|contains:
            - 'C:\ProgramData\AnyDesk.exe'
    condition: selection
falsepositives:
    - Legitimate deployment of AnyDesk
level: high
references:
    - https://twitter.com/TheDFIRReport/status/1423361119926816776?s=20
tags:
    - attack.command_and_control
    - attack.t1219

Sigma Rule Conversion Tools

Sigmac

  • Purpose: Converts Sigma rules into queries for various SIEM backends (e.g., Elasticsearch, Splunk).
  • Usage:
    • Convert rules via command line (e.g., for Splunk:

``python python3.9 sigmac -t splunk -c splunk-windows Process_Creation_AnyDesk_Installation.yml `). ``

  • Note: Sigmac is scheduled to be deprecated in favor of sigma-cli.

Uncoder.io

  • Purpose: An online platform for converting Sigma rules across multiple SIEM/EDR platforms.
  • Usage: Paste the rule, select your target SIEM, and obtain the converted query (supporting outputs such as Elastic Stack Query, QRadar, and Splunk).
