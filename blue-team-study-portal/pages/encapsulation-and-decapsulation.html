Original Notes

Encapsulation and Decapsulation

Encapsulation

Encapsulation is the process where each protocol layer “wraps” the data it receives with its own header (and sometimes a trailer) before handing it to the layer below. This modular approach lets each layer focus solely on its responsibility—port addressing, routing, framing, error checks, etc. without worrying about the rest of the stack.

Protocol Data Units (PDUs)

LayerPDU NameWhat’s Added
ApplicationDataApplication‑specific formatting (e.g., HTTP headers)
TransportSegment (TCP)Datagram (UDP)Source/destination ports,sequence numbers, flags
NetworkPacketSource/destination IP addresses,TTL, protocol field
Data LinkFrameSource/destination MACs,frame check sequence (FCS) trailer

Step‑by‑Step Encapsulation

  1. Application Layer
    • Your application (web browser, chat client, etc.) generates the raw payload and passes it down.
  2. Transport Layer
    • TCP adds a header with ports, sequence/ACK numbers, flags, window size, checksum → TCP segment.
    • UDP adds ports, length, checksum → UDP datagram.
  3. Network Layer
    • The IP layer prepends source and destination IP addresses, TTL, fragmentation info → IP packet.
  4. Data Link Layer
    • Ethernet/Wi‑Fi wraps the packet with source/destination MAC addresses and a trailer with FCS for error detection → Frame.
  5. Physical Layer
    • The frame is converted to electrical, optical, or radio signals and transmitted over the medium.
Note: Frames often include both a header (addresses, type) and a trailer (error‑check sequence).

Decapsulation

On the receiving side, each layer strips off its header/trailer and passes the remaining PDU up to the layer above:

  • Frame → Packet: Data Link checks FCS, removes header/trailer
  • Packet → Segment/Datagram: Network verifies IP checksum, removes IP header
  • Segment/Datagram → Data: Transport verifies ports, sequence/ACK, checksum
  • Data → Application: Application processes the payload

Example: Fetching a Web Page

  1. App Layer: Browser builds an HTTPS GET request.
  2. Transport: TCP performs three‑way handshake, then wraps the GET in segments.
  3. Network: IP adds your PC’s IP and the server’s IP to each segment.
  4. Data Link: Your NIC frames each packet with MACs and FCS and sends it to the router.
  5. Routers: Strip/link‑layer header, inspect IP, reframe, forward until the server.
  6. Reverse Path: Server’s response travels back, with each node decapsulating and re‑encapsulating until your browser finally receives raw data to render the page.

Why It Matters

  • Isolation of functions: Easier troubleshooting—e.g., a broken MAC header points to the link layer, not routing.
  • Interoperability: New protocols fit neatly into one layer without rewriting the entire stack.
  • Security & QoS: Middleboxes (firewalls, QoS shapers) act on specific headers without touching application data.
