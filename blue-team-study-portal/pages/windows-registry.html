Original Notes

Windows Registry

1- What Is the Windows Registry?

The Windows Registry is a hierarchical database critical to Windows operation, storing OS, hardware, and application configurations in a tree of keys and values. The registry functions similarly to a file system:

  • Keys & Subkeys (folders) and Values (files) form the basic structure for storing settings.
  • Lists of Subkeys, Keys with Multiple Values, and Multi‑Line Values represent registry lists in different forms.

Registry terminology example

List of values in the Run Key

Understanding these building blocks is essential for spotting how attackers abuse registry components for persistence (e.g., Run keys, Scheduled Tasks, ShellBags).

2- Registry Structure and Terminology

2-1 Root Keys (Predefined Hives)

Hive (Abbrev.)Description
HKEY_CURRENT_USER (HKCU)Contains the configuration for the user currently logged on
HKEY_USERS (HKU)Holds all actively loaded user profile hives. Each user’s NTUSER.DAT is mounted here under their security identifier (SID).
HKEY_LOCAL_MACHINE (HKLM)Stores machine‑wide configuration for all users: hardware drivers, services, software installs, security policies, and lower‑level components. On‑disk hives SYSTEM, SOFTWARE, SAM, and SECURITY all mount under HKLM.
HKEY_CLASSES_ROOT (HKCR)File type associations & COM registrations
HKEY_CURRENT_CONFIG (HKCC)Contains information about the hardware profile used at system startup, drawn from HKLM\SYSTEM\CurrentControlSet\Hardware Profiles.

The Windows Registry lives in hive files under %SystemRoot%\System32\config and in each user’s profile: System Hives (%SystemRoot%\System32\config)

  • SAM: Security Account Manager. Stores security information related to user accounts and security policies. In the Live System, this Hive is mapped into HKLM\Local_Machine\SAM.
  • SECURITY: Holds security-related configuration data, including user authentication and permissions.
  • SYSTEM: System configuration. Stores system-related configuration data, including hardware, device drivers, and startup settings.
  • SOFTWARE: Contains configuration information for installed software and system-wide settings.
  • DEFAULT: Acts as a template for creating new user profiles, providing default settings for user-specific configuration.

Additionally, there are two main user-related Hives :

  • NTUSER.DAT:
    • User-specific settings for each user profile. Contains information about user preferences and specific configurations.
    • Located at %USERPROFILE%\NTUSER.dat
    • Mapped into HKCU in the Live Registry.
  • USRCLASS.DAT:
    • User-specific class settings. Stores information about Explorer's settings and interaction.
    • Located at %USERPROFILE%\AppData\Local\Microsoft\Windows\UsrClass.dat
    • Mapped into HKCU\Software\Class.

Execution Metadata

  • C:\Windows\AppCompat\Programs\Amcache.hve → Amcache (file execution history)

2-2 Hive Mounting and Transaction Logs

  • When a hive is loaded, its LOG1 and LOG2 transaction logs replay at load time to present a consistent view.
  • For analysis, tools often create a “clean” hive by merging logs, yielding filename_clean.
  • You can optionally load both clean and “dirty” (original) hives side by side for comparison.
  • \Windows\System32\config: contains DEFAULT, SAM, SECURITY, SOFTWARE, SYSTEM hives.
  • %USERPROFILE%\NTUSER.DAT and USRCLASS.DAT: per-user settings and classes.
  • Amcache.hve in AppCompat folder: execution metadata (file path, hash, timestamps).

Dirty Hives and Transaction Logs

Transaction logs are records of changes made to a database over time, while dirty Hives are parts of the Windows registry that haven't been properly closed after changes. Both are important in forensics for understanding system activity and potential security issues.

Sometimes, when your computer shuts down unexpectedly, or the Registry gets messed up, the Hive files can become "dirty," meaning they're inconsistent. To prevent corruption, Windows uses transaction logs. These logs track changes to the registry Hives, allowing Windows to roll back changes if something goes wrong.

The transaction logs are stored in the C:\Windows\system32\config directory alongside the Hive files. They have names like SYSTEM.LOG1, SYSTEM.LOG2, and so on.

Transaction logs are usually hidden files. One of the ways to view them is to open the terminal and run the command dir /a to display all the files present in the folder, including the hidden files

4- Acquisition Methods with KAPE and Traditional Approaches

4-1 Cold (Offline) Acquisition

  1. Attach suspect disk via write‑blocker to a forensic workstation.
  2. Image the disk with FTK Imager or dd to create an E01/RAW file.
  3. Mount the forensic image read‑only and extract hive files from system and user directories.

4-2 Live Acquisition via KAPE (Triaging & Parsing)

  1. Open KAPE (GUI or CLI).
  2. Select RegistryHives target; specify source and output directories.
  3. Choose !EZParser module to parse and normalize registry artifacts.
  4. Run KAPE: it collects hives, replays transaction logs, and generates CSV files by category.
  5. Review CSV outputs (e.g., KnownNetworks.csv, USBSTOR.csv) in your analysis tool.

4-3 Live Acquisition via reg save / FTK Imager

  • Use reg save HKLM\SYSTEM system.hive to export the live SYSTEM hive; repeat for other hives.
  • FTK Imager can likewise export hives without command line.

5- In‑Depth Tool Walkthroughs

5-1 Registry Explorer

Registry Explorer loads hives and transaction logs, provides a bookmarks pane of key artifacts, and allows deep inspection.

Steps: Load hive → integrate LOG1/LOG2 → view Available Bookmarks → navigate or export.

5-2 RECmd (CLI)

RECmd is a command‑line utility for recursive registry hive analysis. It can:

  • Traverse directories or individual hive files, automatically replay transaction logs and load multiple hives.
  • Extract specified keys or values, or search for strings in key names, value names, or data.
  • Support searches in BASE64, Unicode, ASCII, or via regular expressions.
  • Recover deleted keys and values if remnants remain in the hive.
  • Export results in CSV or JSON for easy ingestion into analysis tools.

Example: Search for the keyword ‘administrator’ in the SAM hive and output CSV:

powershell
C:\Users\Administrator\Desktop\EZTools\RECmd>RECmd.exe -f c:\Users\Administrator\Desktop\collection\c\windows\System32\config\SAM --sk administrator

5-3 RegRipper

RegRipper uses plugins to parse specific artifacts from a clean hive.

bash
rr.exe -H C:\hives\NTUSER.DAT -o ntuser_report.txt
rip.exe -r C:\hives\NTUSER.DAT -p userassist > userassist.txt

Note: clean hive first via Registry Explorer to include transaction logs.

5-4 KAPE + !EZParser

One‑step collection and parsing:

powershell
kape.exe --tsource C:\ --tdest D:\out --target RegistryHives --module !EZParser

Outputs normalized CSVs for all key artifacts, ideal for rapid triage.
