ProxyLogon Lab (CyberDefenders)

Scenario

In late February 2021, a threat actor known as HAFNIUM exploited a new vulnerability in Microsoft Exchange known as ProxyLogon CVE-2021-26855. As a SOC analyst in an enterprise utilizing GrayLog as the SIEM solution, your responsibility is to conduct a thorough investigation within the company's network to ensure the absence of any malicious activities. Your objective is to proactively hunt for potential threats and mitigate any risks that may compromise the network's security.

https://cyberdefenders.org/blueteam-ctf-challenges/proxylogon/

CVE-2021-26855 & SSRF Overview

After searching for CVE-2021-26855, I found that it is a Server-Side Request Forgery (SSRF) vulnerability in Microsoft Exchange Server. This vulnerability allows unauthenticated attackers to send arbitrary HTTP requests and authenticate as the Exchange server itself, which is used by attackers to gain initial access to Exchange servers.

What is SSRF?

A web security vulnerability that allows an attacker to force a server to make unauthorized requests to internal or external resources.

How SSRF Works:

A vulnerable web application accepts user input (URL, IP, hostnames) and fetches data from it without proper validation. If the attacker controls the input, they can force the server to send requests to:

  • Internal services: e.g., http://127.0.0.1/admin or http://169.254.169.254/latest/meta-data/
  • Other machines in the same network: e.g., http://10.0.0.5:8080/
  • Malicious servers: e.g., http://attacker.com/steal-data

Attackers can access internal systems, steal cloud credentials, scan the internal network, bypass security controls, and exploit other vulnerabilities.

Lab Environment & Sources

For this lab, GrayLog was used as the SIEM tool (my first time using it and hopefully my last). I have this sources:

  • Winlogbeat – Windows Event Logs Collector: Collects and ships Windows Event Logs

(security, system, application logs).

  • Metricbeat – System & Application Metrics Collector: Collects system and service metrics

(CPU, memory, disk, network, application performance).

  • Packetbeat – Network Traffic Monitor: Captures network traffic

(packets, flows, HTTP, DNS, TLS, etc.) and sends it to Elasticsearch.

  • Filebeat – Log File Collector: Collects and ships log files from applications, security tools, and services.

The main focus was on Packetbeat because, from the information available, attackers send arbitrary HTTP requests.

Questions & Answers

1. What is the tool used by the attacker to identify services running on the victim's machine?

I filtered the logs using Filebeat to find logs coming from security tools such as IDS. I found Suricata logs and obtained the attacker IP 143.243.123.131 and the tool used for reconnaissance was nmap.

2. What is the browser the user spoofed to exploit the vulnerability?

I switched to Packetbeat and queried with the attacker IP. By examining the user-agent, I determined the spoofed browser was FireFox/86.0.

3. What is the local file path of the shell dropped by the exploit?

Using Packetbeat and searching with the attacker IP, I located shell.aspx. The complete local file path is:

C:\Program Files\Microsoft\Exchange Server\V15\FrontEnd\HttpProxy\owa\auth\shell.aspx

4. What is the post-request parameter used to execute commands on the server?

Using the query beats_type:packetbeat AND packetbeat_method:post AND shell.aspx, I identified the parameter used is data.

5. What is the web shell used by the attacker as a backdoor?

At the end of an encoded URL, I found the web shell file: aspxspy.aspx.

aspxspy

6. What is the “MITRE” id of the technique used by the attacker to gain persistence?

By searching for “mitre adding domain users persistence,” the answer is:

T1136.002

7. The attacker tried to evade defenses by renaming the web shell. What is the name of the web shell file after rename?

From our previous query that included POST requests, the attacker saved the aspxspy.aspx file as out.aspx and later renamed it using:

mv . out.aspx log.aspx

log.aspx

8. What is the name of the service the attacker tried to enable on the victim's machine?

From the scanning activity (port 5985) and the web shell commands (e.g., powershell.exe -command Set-Service WinRM -StartMode Automatic), the service is:

WinRM

9. The attacker injected a shellcode in the memory to gain a reverse shell. What is the md5 hash of the shellcode?

After decoding a heavily obfuscated PowerShell command (involving Base64 decoding, character replacements, unescaping Unicode, and gunzipping the result), the calculated md5 hash of the shellcode :

0ce28cfdfc11d8d05374cb0559a761de

10. What is the filename of the tool used by the attacker to move through the network?

From the logs, I found a GitHub repository from ThreatActor13337 at:

https://raw.githubusercontent.com/ThreatActor13337/Exchange60/main/procdump64.exe

Upon further inspection of the repository, I discovered the tool psexec64.exe.

11. The attacker used two methods to access credentials on the victim's machine. What is the MITRE ID of the second method used by the attacker? By searching for the APT group responsible (HAFNIUM) and correlating their techniques, I found that the second method—accessing the Active Directory domain database (NTDS.dit)—corresponds to MITRE ATT&CK technique T1003.004.
