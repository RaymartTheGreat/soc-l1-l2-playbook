AD Hardening

Quick note: how to edit GPOs

Open Group Policy Management Editor on a domain-joined management host (or the lab VM) to change domain or local policies:

java
Group Policy Management Console → [select GPO] → Edit

Most GPO paths used below live under:

java
Computer Configuration → Policies → Windows Settings → Security Settings → (Local Policies | Account Policies | Advanced Audit Policy Configuration)

1- Disable LAN Manager (LM) hashes

Why: LM hashes are weak and trivially cracked; storing them increases credential risk.

GPO path / setting:

java
Computer Configuration → Policies → Windows Settings → Security Settings → Local Policies → Security Options
→ Network security: Do not store LAN Manager hash value on next password change → Enable (Define this policy)

Action: Enable. Force a password change for all accounts after enabling.

2- SMB signing (prevent SMB MiTM / tampering)

Why: Ensures integrity of SMB communications; helps detect MiTM modifications.

GPO path / settings:

java
Computer Configuration → Policies → Windows Settings → Security Settings → Local Policies → Security Options
→ Microsoft network server: Digitally sign communications (always) → Enable
→ Microsoft network client: Digitally sign communications (if server agrees) → Enable (or Always if compatible)

Action: Start with servers (always) and roll to clients after testing.

3- LDAP signing & encryption

Why: Prevents replay/MiTM of LDAP requests and plaintext credential exposure. Prefer LDAPS (TLS) or Require signing.

GPO path / setting:

java
Computer Configuration → Policies → Windows Settings → Security Settings → Local Policies → Security Options
→ Domain controller: LDAP server signing requirements → Require signing

Action: Require signing and deploy LDAPS (certificate on DCs). Test client compatibility.

4- Password management & rotation

Approaches (pros/cons):

  1. Scripted rotation // automated but requires script maintenance.
  2. MFA + less frequent rotation // reduces reliance on password-only auth.
  3. gMSA / Managed accounts // Microsoft solution for service account auto-rotation.

Service accounts: Use gMSAs / Managed Service Accounts where possible to avoid static service-account passwords.

5- Password policy (recommended baseline)

GPO path:

java
Computer Configuration → Policies → Windows Settings → Security Settings → Account Policies → Password Policy

Recommended settings (example):

  • Enforce password history: 10–15
  • Minimum password length: 12 (10–14 acceptable depending on environment)
  • Complexity requirements: Enabled (upper, lower, digits, symbols; exclude account name)
  • Maximum password age: align with org policy (combine with MFA and gMSA for service accounts)
  • Account lockout policy: use sensible thresholds to mitigate brute force while avoiding DoS

6- Least privilege & account design

Design rules:

  • Use standard user accounts for daily activities.
  • Separate privileged accounts (administrators) from regular accounts; restrict, monitor, and require MFA.
  • Avoid shared accounts; where unavoidable use controlled, time-limited solutions.
  • Use role-based access and fine-grained delegation (RBAC) for administrative tasks.

7- Tiered Access Model (AD TAM)

Purpose: Separate assets and prevent privileged credentials from crossing boundaries. Typical tiers:

  • Tier 0: Domain Controllers, AD Forest/Domain admins, other high-value AD components.
  • Tier 1: Servers and applications (domain-joined app servers).
  • Tier 2: End-user workstations and low-privilege systems.

Implementation tip: Enforce via ACLs, jump hosts, dedicated admin workstations, and GPOs. Prevent credential reuse across tiers.

8- Auditing & monitoring

Audit areas: usage, privilege, and configuration changes.

GPO path (start here):

java
Computer Configuration → Policies → Windows Settings → Security Settings → Advanced Audit Policy Configuration

Actionable: Enable detailed object-access, account-management, privileged-use and logon auditing. Forward logs to a centralized SIEM and alert on suspicious patterns (e.g., unusual TGS requests, sudden privilege escalations).

9- Microsoft Security Compliance Toolkit (MSCT) & Policy Analyzer

Why: Microsoft baseline policies and PolicyAnalyzer speed correct configuration and detect conflicting/redundant GPOs.

Workflow: Download official baselines → import/apply in lab → use PolicyAnalyzer to compare GPOs and detect deviations. Only download from Microsoft. Test baselines before wide deployment.
