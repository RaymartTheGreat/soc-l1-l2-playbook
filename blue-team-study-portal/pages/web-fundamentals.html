Original Notes

Web Fundamentals

Before you dig into web servers, HTTP, cookies, or other web concepts, you need two things firmly understood:

  1. How the internet names and locates services (DNS & URL anatomy) — without that, you can’t reason about hosts, subdomains, or why assets load from a CDN.
  2. What components make up a web application (frontend, backend, DB, CDN, load balancer, WAF) — this helps you reason about where to look when things break or when you hunt for malicious activity.

Think of the web as multiple layers stacked on each other. Learn the stack from names → routing → transport → application → logs/observability. That’s the order we’ll follow.

1- URL anatomy

A URL is how humans and browsers ask for a resource. Break it down and understand which parts are client-side vs server-side trust boundaries.

Example URL

python
https://alice:pw123@shop.acme.local:8443/products/view.html?q=socks#reviews

Parts:

  • https — scheme (protocol). Determines the transport and whether TLS is used. Security: prefer https always.
  • alice:pw123@ — userinfo (deprecated for browsers; never send secrets in URLs). Treat as untrusted.
  • shop.acme.local — host (domain or IP). Controls virtual-host selection — maps via DNS to an IP.
  • :8443 — port (default 443 for HTTPS). Nonstandard ports are allowed and may alter firewall paths.
  • /products/view.html — path (resource locator). Server maps this path to a file or a backend handler.
  • ?q=socks — query string (client-controlled input; always treat as untrusted and sanitize).
  • #reviews — fragment (client-side only; not sent to server).

Why each matters for security

  • Query strings often carry user input (SQLi / XSS vectors). Never trust them.
  • Host + port determine which virtual host and which certificates are used.
  • Fragments are irrelevant to server logs (they’re client-only), but useful for understanding what the browser will render.

2- Before web: DNS

If web is the application layer, DNS is the naming & routing layer beneath it. Start here — a wrong DNS understanding leads to bad triage, missed subdomains, and confusing origin IPs.

What DNS actually does

DNS answers one question: “What IP(s) or resource records are associated with this name?” When a browser loads shop.acme.local, it asks DNS: “what IP(s) should I connect to?” The resolver returns records (A, AAAA, CNAME, etc.). The browser then connects to the IP and sends an HTTP request with the Host header set to shop.acme.local.

Domain hierarchy

Domain names are read right-to-left for hierarchy:

  • . — root (implicit)
  • local — TLD (top-level domain)
  • acme — second-level domain
  • shop — subdomain

So shop.acme.local is a subdomain of acme.local. The authority for each label is delegated: root → TLD → second-level → authoritative name servers.

Practical example: if shop.acme.local is a CNAME to cdn-3.edge.example.net, the chain matters: requests resolve to cdn-3.edge.example.net → then that name resolves to one or more A/AAAA addresses. A long CNAME chain can reduce clarity of who hosts the asset and may hide origin IPs.

Important DNS record types

  • A — IPv4 address.
  • AAAA — IPv6 address.
  • CNAME — alias: name points to another name (not directly to an IP). Useful for multi-host setups but avoid for zone apex.
  • MX — mail exchangers.
  • TXT — text blobs (SPF, DKIM, verification tokens). Danger: people sometimes put secrets here by mistake — check TXT for leaked tokens.
  • NS — authoritative name servers for the domain.
  • SOA — start of authority; contains serial & refresh timings.

dig example

bash
dig +nocmd +noall +answer shop.acme.local A

Sample output:

text
shop.acme.local.         300  IN  CNAME  cdn-3.edge.example.net.
cdn-3.edge.example.net.   60  IN  A      198.51.100.24

Interpretation:

  • shop.acme.local is a CNAME pointing to cdn-3.edge.example.net.
  • The CDN hostname resolves to IP 198.51.100.24.
  • TTLs (300, 60) show caching lifetimes / short TTL means answers change often; long TTL caches longer.

Why this matters for incident response

  • If a subdomain points to a third-party CDN, the origin server may be hidden. Investigate the CNAME chain to find origin or configuration weaknesses.
  • Malicious actors sometimes add TXT or CNAME records to route traffic or validate domains check NS and SOA history where possible.

Common DNS pitfalls & triage tips

  • Hidden origin: CDN or cloud provider hides the origin IP. Look up the CNAME chain and historical DNS if needed.
  • Subdomain takeover: CNAME points to a service that is no longer configured. If the target is unclaimed, an attacker can claim it and serve content.
  • Leaked tokens in TXT: scan TXT records for API keys or verification tokens.
  • Delegation mistakes: misconfigured NS records can delegate control to an attacker-controlled nameserver.

3- How web servers work

What is a web server?

A web server is software that:

  1. Listens on a network socket (port 80/443) for incoming TCP/TLS connections.
  2. Accepts HTTP requests, maps requested paths to local resources or backend handlers.
  3. Sends HTTP responses back to the client.

Common web server software:

  • Apache — feature-rich, modular.
  • Nginx — high-performance, event-driven; often used as a reverse proxy.
  • IIS — Microsoft’s web server on Windows.
  • Node.js (Express, Fastify) — application runtime acting as a web server.

Root directory example (Linux): /var/www/html/index.html corresponds to http://example.com/index.htmlWindows IIS default: C:\inetpub\wwwroot\

Virtual hosts (vhosts)

A single server IP can serve many domain names. The server checks the Host header and maps it to the corresponding vhost config. Each vhost has its own document root, SSL cert, logging, and configuration. Misconfigured default vhosts can leak test pages or debug endpoints.

Example:

  • one.com → /var/www/one
  • two.com → /var/www/two

    • If the Host doesn’t match, the server returns the default vhost content.

How dynamic content is served

Static files are served directly from disk. Dynamic requests hit:

  • CGI/FCGI backends (e.g., PHP-FPM), or
  • Application servers (Node.js, Python WSGI), or
  • Reverse proxy forwards requests to upstream services.

Dynamic backends often run as different processes and log separately — check both web server logs and application logs during investigations.

4- Supporting infrastructure: CDN, Load Balancer, WAF, Databases

CDN (Content Delivery Network)

External image reference: A graphical representation of a Content Delivery Network (CDN) in which the central server distributes data to edge servers around the world, which then distribute data to the users.

Function: cache & serve static assets (images, JS, CSS) from edge nodes close to users. Benefits: reduces latency, offloads origin, and provides DDoS mitigation. Security/triage notes:

  • CDN hides origin IP / makes finding the real server harder.
  • CDN configurations (caching rules, rewrite rules) can leak sensitive assets if misconfigured.
  • Check CNAMEs and CDN-origin configuration to confirm origin security.

Load Balancer

Function: distribute incoming traffic to multiple backend servers. Types: Layer 4 (TCP) or Layer 7 (HTTP-aware). Why it matters: health checks, sticky sessions, and SSL termination can be on the LB / this affects where TLS is terminated and where to inspect traffic. Misconfigured LBs can route traffic to stale or unpatched backends.

WAF (Web Application Firewall)

External image reference: The Cloudflare CDN dashboard with the Security panel selected shows the total number of requests the website received in the past 24 hours and all mitigated or blocked requests.

Function: analyzes HTTP traffic for common attacks (SQLi, XSS, LFI) and rate-limits suspicious behavior. How it’s placed: typically between LB/CDN and origin. Limitations: WAF rules can be noisy (false positives) and bypassable. Don’t rely on a WAF as the only security control / fix the root cause in the application.

Databases

Function: persistent storage for application data. RDBMS (MySQL, Postgres, MSSQL) and NoSQL (MongoDB). Security notes:

  • SQL injection targets databases directly - parameterized queries are essential.
  • Publicly exposed DB ports are high-severity findings.
  • Backups, replicas, and failover nodes may have separate access controls / enumerate all.

5- Frontend vs Backend

Frontend (client-side)

  • Technologies: HTML, CSS, JavaScript.
  • What it does: renders UI, executes client-side logic, and makes API calls to the backend.
  • Security concerns: sensitive data in JS, plaintext secrets embedded in HTML/JS, DOM-based XSS, insecure use of eval() or innerHTML.

Backend (server-side)

  • Technologies: Node.js, Python, Ruby, PHP, .NET, Java, etc.
  • What it does: business logic, authentication, database access, rendering dynamic pages, issuing tokens/cookies.
  • Security concerns: SQLi, insecure deserialization, RCE via vulnerable libraries, privilege escalation via broken auth.

Where issues live: frontend vulnerabilities expose the user; backend vulnerabilities expose server/data. Both matter for defenders.

6- HTML / CSS / JavaScript

  • HTML — structure and semantics of a page. Elements, attributes, forms (<form>), inputs.
  • CSS — presentation. Controls layout; can leak info via :visited timing in older browsers (rare now).
  • JavaScript — behavior. Detect it in the DOM, watch for inline scripts, and for client-side logic that builds requests (useful for hunting).

Example: insecure pattern

html
<!-- Unsafe: directly inserting user input into the DOM --><div id="comment"></div><script>  document.getElementById('comment').innerHTML = location.search; // dangerous</script>

This allows DOM-based XSS if location.search contains "><script>…</script>.

7- Logging

Logs are the single most valuable asset for incident response. Learn formats and fields.

Common Log Format (combined) — fields:

python
remote_addr - remote_user [time_local] "request_line" status bytes_sent "referer" "user_agent"

Realistic example (benign session):

python
10.10.10.100 - - [08/Sep/2025:12:34:56 +0300] "GET /index.html HTTP/1.1" 200 1024 "https://example.com/" "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/116.0.0.0 Safari/537.36"
10.10.10.100 - - [08/Sep/2025:12:35:02 +0300] "GET /login.html HTTP/1.1" 200 2048 "https://example.com/index.html" "Mozilla/5.0 ..."
10.10.10.100 - - [08/Sep/2025:12:35:20 +0300] "POST /login HTTP/1.1" 302 512 "https://example.com/login.html" "Mozilla/5.0 ..."
10.10.10.100 - - [08/Sep/2025:12:35:25 +0300] "GET /myaccount.html HTTP/1.1" 200 5172 "https://example.com/" "Mozilla/5.0 ..."

How to use these fields:

  • remote_addr — identify client IP and geo/ASN.
  • time_local — timeline reconstruction; correlate with firewall and app logs.
  • request_line — HTTP method, path, and protocol.
  • status — 200/302/401/403/404/500 — use to filter anomalies.
  • bytes_sent — detect exfil (sudden large downloads).
  • referer & user_agent — fingerprint sessions and spot unusual clients.

Hunting example: repeated POST /login from many IPs with 401/403 responses → suspect credential stuffing or brute force; watch for account lockouts or unusual successful logins.

8- HTTP internals: requests, responses, methods and status codes

HTTP request structure

A client request has three parts:

  1. Start line — METHOD path HTTP/x.x (e.g., GET /index.html HTTP/1.1).
  2. Headers — Host, User-Agent, Accept, Cookie, Content-Type, etc.
  3. Body — optional; used by POST/PUT/PATCH.

External image reference: A diagram visualizing how cookies are introduced in http requests to allow storing user information

Example request

python
POST /login HTTP/1.1
Host: app.example.com
User-Agent: curl/8.2
Accept: application/json
Content-Type: application/json
Content-Length: 53
Cookie: session=abc123

{"username":"alice","password":"p@ssw0rd"}

HTTP response structure

Responses: HTTP/x.x status_code reason + headers + body.

Example response

python
HTTP/1.1 302 Found
Location: /dashboard
Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Lax
Content-Length: 0
html
HTTP/1.1 200 OK             ---> Status Line
Date: Mon, 09 Sep 2025 10:30:00 GMT
Server: OmarServer/1.0      ---> Response Header 
Content-Type: text/html
Content-Length: 85
Connection: close

<html>                     ---> Response Body
  <head>
    <title>Welcome to Omar's Site</title>
  </head>
  <body>
    <h1>Hello, Omar!</h1>
    <p>This is a sample HTTP response.</p>
  </body>
</html>
  • Status Line: HTTP/1.1 200 OK → protocol version, status code, status message.
  • Response Headers: Provide metadata about the response (like server, content type, length, etc.).
  • Response Body: The actual content sent to the client (HTML in this case).

Methods

Status codes

  • 1xx — informational.
  • 2xx — success (200, 201).
  • 3xx — redirects (301, 302, 307, 308).
  • 4xx — client errors (400, 401, 403, 404, 405, 429).
  • 5xx — server errors (500, 502, 503, 504).

Threat indicator example: repeated POST /api/transfer 500 with unusual payloads → may indicate an input causing server-side exceptions (possible injection).

9- Headers & Cookies

Request headers

  • Host — vhost selection; mismatch with TLS CN/SAN causes browser warnings/errors.
  • User-Agent — spoofable; useful for filtering.
  • Referer — origin of the request; can be stripped.
  • Accept, Accept-Encoding, Accept-Language — content negotiation.
  • Authorization — sensitive (bearer tokens/basic auth).

Response headers to care about

  • Content-Type — how to parse the body.
  • Set-Cookie — set cookies; flags below matter.
  • Cache-Control, Expires — caching behavior.
  • Location — redirects.

Cookies

Set-Cookie example:

python
Set-Cookie: name=value; Expires=Wed, 09 Jun 2026 10:18:14 GMT; Path=/; Domain=.example.com; Secure; HttpOnly; SameSite=Strict

Flags:

  • HttpOnly — prevents JS access via document.cookie.
  • Secure — send only over TLS.
  • SameSite — Strict | Lax | None — controls cross-site sending. None must be paired with Secure.
  • Domain/Path — scope of cookie. Prefer host-only cookies where possible.
  • Expires / Max-Age — lifetime.

Cookie anti-pattern: embedding user ID or role in cookie plaintext — always avoid; use server-side session mapping.

Debugging headers & cookies

  • Browser DevTools → Network → request → Headers/Cookies tabs.
  • curl example:
bash
curl -v -i https://app.example.com/login -d '{"u":"a","p":"b"}' -H 'Content-Type: application/json'
  • v prints TLS handshake and verbose info; i includes response headers.

10- Security headers

Strict-Transport-Security (HSTS)

python
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

Forces browsers to use HTTPS. Test before preload.

Content-Security-Policy (CSP) / example:

python
Content-Security-Policy: default-src 'self'; script-src 'self' https://cdn.example.com; style-src 'self' 'unsafe-inline'; img-src 'self' data: https://images.examplecdn.com; object-src 'none';
  • Start with report-only:
python
Content-Security-Policy-Report-Only: <policy>; report-uri /csp-report-endpoint

Browser console shows blocked resources; reports are POSTed to the endpoint if configured.

X-Frame-Options / frame-ancestors

python
X-Frame-Options: DENY

Or modern approach:

python
Content-Security-Policy: frame-ancestors 'self';
  1. X-Content-Type-Options: nosniff — prevent MIME sniffing.
  2. Referrer-Policy
python
Referrer-Policy: no-referrer-when-downgrade

Or stricter: strict-origin-when-cross-origin or no-referrer.

  1. Permissions-Policy — control browser features (geolocation, camera).
  2. Expect-CT — certificate transparency; advanced.

11- TLS / TLS handshake & certificates

Why TLS matters

TLS provides confidentiality, integrity, and server authentication. Without TLS, eavesdropping and MitM attacks are trivial.

TLS 1.2 handshake (simplified)

  1. ClientHello (version, ciphers, random, SNI).
  2. ServerHello (chosen cipher, random).
  3. Certificate (server chain).
  4. ServerKeyExchange (if needed).
  5. ServerHelloDone.
  6. ClientKeyExchange (pre-master secret).
  7. ChangeCipherSpec + Finished (switch to encrypted channel).

After this, data is encrypted.

TLS 1.3 differences

  • Fewer round-trips (faster).
  • Forward secrecy by default.
  • 0-RTT option exists (resumption with replay risks).

Certificate concepts

Think of a certificate like an official ID card for a website. It proves that the website is who it says it is. Without it, your browser can’t be sure it’s talking to the real site / someone could be impersonating it.

  • CN / SAN — hostnames covered. Modern clients require SAN.
  • Chain of trust — leaf → intermediates → root.
  • OCSP / OCSP stapling — revocation checks; stapling improves latency/privacy.
  • Pinning — reduces CA risk but is hard to manage.

Inspect a certificate

python
openssl s_client -connect app.example.com:443 -servername app.example.com -showcerts

Check:

  • Valid From / To (expiry).
  • Issuer.
  • SANs include the expected host.
  • Chain completeness (missing intermediates cause client errors).

12- HTTP/2 & HTTP/3

HTTP/2

  • Binary framing and multiplexing: multiple requests/responses on a single TCP connection.
  • Header compression (HPACK) — smaller headers but possible info-leak if bug exists.
  • Server push — server can proactively send resources. Misconfiguration can cause cache issues.
  • Observability: intermediaries and WAFs must parse HTTP/2; older tools may miss attacks.

HTTP/3 (QUIC)

  • Runs over QUIC (UDP); built-in TLS 1.3; connection migration.
  • Observability impact: network devices must support QUIC to inspect traffic; update WAF/CDN and SIEM parsing as needed.

13- WebSockets

  • What: a protocol for full-duplex communication between a client (like your browser) and a server. Unlike normal HTTP requests:
  • HTTP: Request → Response → Done (stateless, one-way at a time)
  • WebSocket: Continuous two-way connection; both client and server can send messages at any time
  • Use cases:
    • Chat apps (Slack, WhatsApp Web)
    • Live feeds (stock prices, sports scores)
    • Multiplayer games

14- Hardening checklist

Web Server & App

  • Remove default pages; disable directory listing.
  • Enforce least privilege for files/dirs; run app process as non-root.
  • Use parameterized queries and centralized input validation.
  • Implement robust error handling; do not expose stack traces.
  • Patch web server & frameworks promptly.

TLS & Certificates

  • Enforce TLS 1.2+; prefer TLS 1.3; disable weak ciphers.
  • Use HSTS (test in report-only first); enable OCSP stapling.
  • Monitor certificate expiry; automate renewals (Let’s Encrypt / ACME).

CDN & Load Balancer

  • Restrict origin access to known CDN edge IPs (prevent direct public origin access).
  • Ensure LB health checks verify full app health (not just TCP).
  • Avoid exposing management APIs on public-facing LB.

WAF & Rate Limiting

  • Deploy WAF with tuned rules; use block for high-confidence detections and monitor for suspicious patterns.
  • Implement per-IP and per-user rate limits for auth endpoints.

Cookies & Session Management

  • Set HttpOnly, Secure, SameSite appropriately.
  • Use short-lived session tokens with refresh workflow; revoke tokens on logout or suspicious activity.
  • Rotate session keys periodically.

Logging & Detection

  • Centralize logs (web, app, WAF, LB) to SIEM with synchronized time (NTP).
  • Log request bodies for high-suspicion events (redact PII) and keep appropriate retention policies.
  • Build dashboards: 404 spikes, 500s per endpoint, login failure rate, unusual user agents.

Databases & Backups

  • Do not expose DB ports publicly. Use private networks or SSH tunnels.
  • Enforce least privilege DB users; use prepared statements.
  • Encrypt backups and test restores.

DevOps & CI/CD

  • Scan dependencies for vulnerabilities (Snyk, Dependabot).
  • Use IaC with code review; avoid secrets in code or environment files. Use secret managers (Vault, AWS Secrets Manager).
