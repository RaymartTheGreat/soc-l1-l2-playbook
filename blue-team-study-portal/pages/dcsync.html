Original Notes

DCSync

DCSync is an attack that allows an adversary to simulate a domain controller by abusing the Directory Replication Service Remote Protocol (MS-DRSR). By doing so, the attacker can request and retrieve sensitive password data including NTLM hashes and the KRBTGT account hash from other domain controllers.

MITRE ATT&amp;CK ID: T1003.006

1- Understanding the Attack

  • Attack Vector:

The attacker leverages replication permissions (e.g., "Replicating Directory Changes") to request replication data from a domain controller.

  • Replication Permissions

Replication permissions in Active Directory, such as "Replicating Directory Changes," allow an account to retrieve updated directory data from domain controllers. This functionality is critical for synchronizing AD data across the network, but if misused, it can provide an attacker with access to sensitive information.

  • Mechanism:

By compromising an account with replication privileges, the attacker issues DCSync requests to obtain sensitive Active Directory (AD) data that is normally used for AD synchronization. This data can then be used for further attacks, such as crafting Golden Tickets or moving laterally within the network.

2- Attack Execution

a. Compromise an Account with Replication Permissions

  • Objective:

Obtain credentials for an account with the "Replicating Directory Changes" or "Replicating Directory Changes All" right.

  • Example:

An attacker might first use Pass-the-Hash to impersonate a privileged user with these rights.

b. Request Replication Data Using Mimikatz

  • Process:

Execute the DCSync command to retrieve password hashes and key material (e.g., for the KRBTGT account).

  • Example Command:

``powershell .\mimikatz.exe "lsadump::dcsync /user:DOMAIN\krbtgt" ``

  • Outcome:

The command returns NTLM hashes and key material, which can be used to craft a Golden Ticket.

c. Utilize the Retrieved Credentials

  • Result:

With the obtained data, the attacker can forge Golden Tickets and perform Pass-the-Ticket attacks, gaining unrestricted access across the AD domain.

3- Key Tools

  1. Mimikatz
    • Usage:

Executes the lsadump::dcsync command to extract replication data.

  1. Other Tools
    • Alternatives:

DSInternals PowerShell module or secretsdump.py can perform similar replication-based credential extractions.

4- Detection and Defensive Strategies

Detection

  • Replication Rights Abuse (Event ID 4662)

Monitor directory service object operations where the Properties field contains replication rights GUIDs:

    • 1131f6aa-9c07-11d1-f79f-00c04fc2dcd2 → DS-Replication-Get-Changes
    • 1131f6ad-9c07-11d1-f79f-00c04fc2dcd2 → DS-Replication-Get-Changes-All
    • 89e95b76-444d-4c62-991a-0facbeda640c → DS-Replication-Get-Changes-In-Filtered-Set

These appear only when replication permissions are being exercised—on non-DC accounts, that’s a red flag.

  • Suspicious Logons (Event ID 4624)

DCSync attempts often require NTLM/Kerberos logons. Spot logons with:

    • LogonType 3 (network logon)
    • AuthenticationPackageName NTLM from non-DC hosts
    • SubjectUserSid = S-1-0-0 or privileged accounts like Administrator being used abnormally
  • Correlation Checks:

Replication requests should always come from actual domain controllers. If you see Event ID 4662 with replication GUIDs followed by 4624 network logons from workstations or admin accounts, that’s likely a DCSync attempt.

  • Detection Queries :

``jsx event.code: 4662 AND winlog.event_data.Properties: ( "1131f6aa-9c07-11d1-f79f-00c04fc2dcd2" OR "1131f6ad-9c07-11d1-f79f-00c04fc2dcd2" OR "89e95b76-444d-4c62-991a-0facbeda640c" ) AND winlog.event_data.AccessMask: "0x100" AND NOT winlog.event_data.SubjectUserName: ("DC01$" OR "krbtgt") ``

``jsx event.code: 4624 AND winlog.event_data.LogonType: 3 AND winlog.event_data.AuthenticationPackageName: "NTLM" AND winlog.event_data.TargetUserName: "Administrator" AND winlog.event_data.SubjectUserSid: "S-1-0-0" AND NOT host.name: ("DC01" OR "DC02") ``

Mitigation Measures

  • Restrict Replication Rights:

Only Domain Controllers should hold replication permissions. Regularly audit for accounts granted DS-Replication-Get-Changes rights.

  • Monitor Replication Traffic:

Track replication requests and validate that they originate strictly from trusted DCs.

  • Privileged Access Management:

Apply strong controls and real-time monitoring for accounts with elevated AD privileges.

5- Summary

  • Protocol Targeted: Active Directory replication (MS-DRSR)
  • Authentication Material: NTLM hashes and KRBTGT account hash
  • Scope of Attack: Credential compromise enabling Golden Ticket creation, lateral movement, and unrestricted access in an AD environment
  • Key Tools: Mimikatz, DSInternals, secretsdump.py
