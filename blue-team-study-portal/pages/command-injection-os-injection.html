Original Notes

Command Injection / OS Injection

Command injection happens when untrusted data is passed into an OS shell (or any command interpreter) and the attacker is able to run system commands. The app runs those commands with the web service’s privileges — so impact ranges from data theft to full server takeover.

Tiny shell primer

  • Command separators / chaining: ; && || | `command $()`
  • Common utilities an attacker may try to call: ls, cat, echo, cp, rm, sh, bash, cmd.exe, powershell, curl, wget, nc
  • Danger indicators: use of ;, &&, backtick, $( inside user-supplied fields; long Base64 blobs; unexpected IPs/hosts in outgoing connections

These are the building blocks attackers use to chain commands and exfiltrate data.

How it works

Imagine a web app that copies an uploaded file name into a shell command like:

python
cp <user_filename> /tmp/

If a user supplies a filename containing a separator, e.g. file; ls; .txt, the shell sees multiple commands:

python
cp file
ls
.txt

ls runs on the server — even if its output isn’t returned to the user. With a crafted payload, an attacker can escalate to stronger actions (create files, open network connections, try to spawn shells).

What attackers can gain

  • Read local files (configs, secrets)
  • Modify or delete data (integrity/destruction)
  • Install backdoors or web shells (persistence)
  • Move laterally if network tools are available (scan, pivot)
  • Create outbound connections to exfiltrate data or receive commands

Detection

Where to look

  • Web access logs (query strings, paths)
  • Request headers (User-Agent, Referer) — attackers sometimes hide commands here
  • Application logs (parameters passed to system calls)
  • WAF / IPS alerts (command-like patterns)
  • Outbound logs (proxy, firewall, DNS) for suspicious callbacks or connections

What to search for (decoded)

  • Command separators / chaining tokens: ;, &&, ||, |, ``, $(`
  • Shell keywords / utilities in inputs: ls, cat, curl, wget, nc, bash, cmd.exe, powershell
  • Long encoded blobs or many \x / % sequences (possible binary or encoded payload)
  • Suspicious use of shell constructs in headers (User-Agent containing $(...) or backticks)
  • Outbound connections initiated shortly after suspicious input (DNS lookups, HTTP/HTTPS to external hosts)

Example HTTP request

text
GET / HTTP/1.1
Host: yourcompany.com
User-Agent: () { :; }; echo "NS:" $(</etc/passwd)
Accept: ...
Connection: close

Why this is suspicious / what happened:

  • User-Agent should be browser/agent info, not commands.
  • The string contains a shell-style payload that exploits a vulnerable bash-like interpreter (Shellshock).
  • The attacker expects the server to evaluate the header and run the command; successful execution can leak /etc/passwd into responses or logs.
  • Presence of echo and file reads (/etc/passwd) in a header is a clear IOC.

Quick checks to run after spotting suspicious payloads

  1. URL-decode and repeatedly decode parameters/headers; inspect for separators & shell tokens.
  2. Search logs for same IP / UA and similar patterns (other endpoints, POSTs).
  3. Check outbound logs (DNS / proxy) for new external domains contacted after the request timestamp.
  4. Inspect app code paths that call system(), exec(), or command shells — note where user input reaches them.
  5. If you find evidence of command execution (unexpected files, processes, or logs), isolate the host and preserve volatile data for IR.
  6. If stored input is found, check which users saw it (stored payload risk).
  7. Add temporary WAF rule to block exact suspicious tokens while devs fix the root cause.

Prevention / developer guidance

  • Never pass raw user input into shell commands / use safe APIs or language libraries to handle files/IO.
  • Validate and strictly whitelist filenames and parameters.
  • Run services with the least privileges required.
  • Avoid shell interpretation entirely for user data (don’t system() with concatenation).
  • Keep OS and interpreters (bash, sh) patched to avoid known issues (e.g., Shellshock).
  • Use containerization / sandboxing to limit blast radius.
