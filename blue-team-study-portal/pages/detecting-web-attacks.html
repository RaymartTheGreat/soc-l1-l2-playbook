Client-side vs Server-side attacksClient-side attacks exploit the user’s browser/device or their actions. The attacker runs malicious code in the user’s environment to steal sessions, credentials, or data that lives on the endpoint.0 child notesSQL Injection (SQLi)SQL Injection happens when an attacker puts SQL code into input fields (or headers, or URLs) and the application builds and runs a database query using that untrusted data. The result: the attacker can read or change data, bypass login logic, or sometimes run system commands via the database.0 child notesCross-Site Scripting (XSS)XSS is when an attacker injects malicious client-side code (usually JavaScript) into a web page that other users load. The browser executes that code in the context of the site, so the attacker can hijack sessions, steal cookies, redirect users, deface pages, or do other client-side abuse.0 child notesCommand Injection OS InjectionCommand injection happens when untrusted data is passed into an OS shell (or any command interpreter) and the attacker is able to run system commands. The app runs those commands with the web service’s privileges — so impact ranges from data theft to full server takeover.0 child notesInsecure Direct Object Reference (IDOR)IDOR is a type of Broken Access Control vulnerability. It happens when an application exposes a reference (like id=123) to an internal object without properly checking if the requester is authorized to access it.0 child notesCSRF (Cross-Site Request Forgery)CSRF is when an attacker makes a victim’s browser send an authenticated request (with the victim’s cookies/session) to a target site — causing the victim to perform actions they didn’t intend (change password, transfer funds, etc.).
