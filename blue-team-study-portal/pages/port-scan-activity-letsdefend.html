Port Scan Activity (LetsDefend)

Can you determine evidences of port scan activity?

https://app.letsdefend.io/challenge/port-scan-activity

1- What is the IP address scanning the environment?

10.42.42.253

we observe multiple SYN packets originating from 10.42.42.253 and targeting the same ports across different IP addresses. This pattern is characteristic of a TCP port scan.

  • In the first few packets, the attacker probes several hosts on port 80(HTTP).
  • Later (packets 11–15), the attacker repeats the same behavior but against port 389(LDAP).

This confirms that 10.42.42.253 is actively scanning the environment for open services.

2- What is the IP address found as a result of the scan?

10.42.42.50

To identify which host responded to the attacker’s scan, I applied the following Wireshark filter:

ip.dst == 10.42.42.253 && tcp.flags.syn == 1 && tcp.flags.ack == 1

This filter captures packets sent back to the attacker (10.42.42.253) where the flags contain both SYN and ACK. A SYN+ACK response is the standard reply from a host when the scanned port is open.

From the results, 10.42.42.50 responded

3- What is the MAC address of the Apple system it finds?

00:16:cb:92:6e:dc

While analyzing the capture, I identified NetBIOS Session Service packets being broadcast on the LAN. These packets revealed the device’s NetBIOS name.

Inspecting the Ethernet II header of these packets showed the source as Apple_92:6e:dc, which corresponds to the MAC address 00:16:cb:92:6e:dc.

A quicker method in Wireshark is to navigate to:

Statistics > Resolved Addresses

and search for entries containing Apple

4- What is the IP address of the detected Windows system?

10.42.42.50

To identify the Windows host, I used the following Wireshark filter:

ip.ttl == 128

I knew from a past lab that Windows systems typically have a default TTL of 128, but I confirmed this again with ChatGPT to be sure before applying it here.

Also From earlier analysis, we had four IP addresses of interest:

  • 10.42.42.25 → the Apple device
  • 10.42.42.253 → the attacker scanning the environment
  • 10.42.42.56 → another host that didn’t respond to the scan
  • 10.42.42.50 → the host that responded with a SYN+ACK to the attacker’s probe

Since 10.42.42.50 matched both conditions /// a TTL of 128 and a successful TCP handshake response this confirms it as the detected Windows system.
