Original Notes

Group Policy

In environments with large numbers of users and machines, enforcing consistent settings manually is inefficient and error-prone. Security risks such as unrestricted USB access, weak passwords, or unauthorized applications can easily go unnoticed if not centrally managed.

Group Policy allows administrators to define and enforce configuration rules across the network without interacting with each machine individually.

What Is Group Policy?

Group Policy consists of a set of rules that control user and computer configurations across Windows environments. These rules are stored in Group Policy Objects (GPOs) and can control everything from password policies to software restrictions.

When used with Active Directory, GPOs can be applied at scale across users, computers, and organizational units.

Example: Instead of manually disabling USB storage on every endpoint, a single GPO can be created and applied across the environment.

Types of Group Policy Objects

1. Local GPOs

  • Apply only to a single machine, even if it’s not part of a domain.
  • Useful for testing or small-scale administration.
  • Example: A laptop with a local policy restricting command-line tools.

2. Active Directory (Domain) GPOs

  • Stored in AD and applied across multiple systems.
  • Used to implement domain-wide or departmental policies.
  • Example: Enforcing screen lock timeout for all users in the Operations department.

3. Starter GPOs

  • Serve as pre-configured templates with basic settings.
  • Used to standardize initial GPO creation.
  • Not applied directly but used as a baseline for new GPOs.

GPO Application Order

The order in which policies are processed matters. The closer the policy is to the target object (user or machine), the more likely it is to take precedence—unless overridden.

Processing Hierarchy:

  1. Local policies (on the machine itself)
  2. Site-level GPOs
  3. Domain-level GPOs
  4. Organizational Unit (OU) GPOs
    • From parent OU down to nested OUs

Policy Conflict Resolution:

  • If a setting is defined in multiple places, the last one applied (OU-level, for example) will win.
  • However, if a GPO is enforced, it takes precedence over conflicting settings—even if it’s higher up in the hierarchy.

Creating and Linking a GPO

Let’s say you want to enable logging for command-line process creation on all systems in the Finance OU.

Steps:

  1. Open Group Policy Management Console from the server or via search.
  2. Right-click the domain or OU you want to target.
  3. Choose Create a GPO in this domain, give it a clear name like Enable-CmdAudit.
  4. After creation, right-click the GPO → Edit.
  5. In the Group Policy Editor, browse to the relevant path for command-line logging.
  6. Double-click the setting → Enable → Apply → OK.
  7. Optionally, Enforce the GPO if you want it to override lower-level policies.
  8. Link it to the appropriate OU or group using Link an existing GPO.

GPOs usually update every 90 minutes by default, but this can vary slightly.

Notes on Enforcement and Linking

  • Enforcing a GPO ensures it cannot be overridden by other GPOs further down the chain.
  • You can link the same GPO to multiple OUs for wider coverage.
  • GPO link order affects which policies take priority when multiple are linked at the same level.

Summary

Group Policy is one of the most powerful tools for centralized security and configuration in Windows environments. By structuring GPOs well and applying them thoughtfully, you can:

  • Enforce least privilege.
  • Reduce attack surface.
  • Maintain consistency across departments.
  • Respond faster to evolving threats.
