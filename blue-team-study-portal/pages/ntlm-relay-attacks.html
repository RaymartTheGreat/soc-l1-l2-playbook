Original Notes

NTLM Relay Attacks

MITRE ATT&amp;CK ID: T1557.001

1- Understanding NTLM Authentication

  • Challenge–Response Mechanism:
    1. Client sends a negotiate message to server.
    2. Server replies with a challenge (random nonce).
    3. Client responds with an NTLM response: HMAC‑MD5 of the nonce using the user’s NTLM hash as key.
    4. Server verifies by comparing against its own HMAC calculation.
  • Why It’s Relayed:

The NTLM response blob can be forwarded intact no need to know the underlying password or hash.

  • Common NTLM Usage:
    • SMB file shares
    • HTTP(S) authentication (e.g. IIS)
    • RPC/LDAP binds

2- Steps of an NTLM Relay Attack

a. Interception

  • Objective: Capture the NTLM handshake from a legitimate client.
  • Method: Host a rogue service (SMB, HTTP, LDAP) or poison name‑resolution (LLMNR/WPAD) so clients authenticate to you.

b. Relaying

  • Objective: Forward the captured NTLM handshake to a target service that accepts NTLM.
  • Command Example (Impacket):

``jsx ntlmrelayx.py --smb2support -t smb://dc01.corp.local ``

  • Notes:
    • -smb2support ensures compatibility with newer SMB versions.
    • The target can be SMB, LDAP, HTTP, etc.—any service trusting NTLM.

c. Access

  • Outcome: If the target accepts the relayed credentials, you obtain an authenticated session as that user.
  • Post‑Exploitation: Dump hashes, modify AD objects, dump the SAM, or pivot further.

3- Key Tools for NTLM Relay

  • Impacket ntlmrelayx.py – Captures NTLM handshakes and relays them to specified targets.
  • Responder / Inveigh – Poison LLMNR/WPAD to capture authentication requests.
  • CrackMapExec – Combines poisoning and relaying in one tool.
  • Metasploit Framework – Modules for capturing and relaying NTLM sessions.

4- Example Scenario

  1. Attacker Host A runs Responder to poison LLMNR/WPAD.
  2. Victim Host B fails DNS for \\FILESRV → broadcasts LLMNR → connects to Host A.
  3. Responder captures NTLMv2 handshake.
  4. Attacker runs:

``bash ntlmrelayx.py --smb2support -t smb://dc01.corp.local ``

  1. Relay succeeds: Attacker gains an SMB session on DC01 as the victim user.
  2. Attacker uses that session to dump the domain’s SAM or create new domain‑admin users.

5- Detection and Mitigation

Detection

  • Event ID 4624 (Successful Logon):

Look for NTLM network logons (Logon Type 3) on domain controllers from unexpected source IPs.

  • Event ID 4776 (NTLM Authentication):

Monitor repeated NTLM authentications without preceding Kerberos.

  • Network IDS:

Alerts on SMB/LDAP binds to DCs from unknown hosts or subnets.

Mitigation

  • Enforce SMB Signing via GPO.
  • Restrict NTLM using “Network security: Restrict NTLM” policies.
  • Network Segmentation to protect critical servers.
  • Harden Name Resolution: Disable LLMNR and NetBIOS where not needed.

6- Summary

  • Protocol Targeted: NTLM challenge–response
  • Attack Material: Captured NTLM response blobs (no cracking required)
  • Scope: Lateral movement, privilege escalation, domain compromise
  • Key Defenses: SMB signing, NTLM restrictions, network segmentation, LLMNR/NetBIOS hardening
