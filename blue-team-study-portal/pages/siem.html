Overview

SIEM (Security Information and Event Management) is a cybersecurity solution that collects, aggregates, normalizes, and analyzes logs and event data from across an organization’s IT infrastructure to detect threats, support incident response, and facilitate forensic investigations.

SIEM

Log Sources

Host‑Centric Logs

  • Windows Event Logs (e.g., login attempts, file access)
  • Sysmon (process execution, registry changes)
  • Osquery
  • Examples:
    • User accessing or modifying a file
    • Process execution (Event ID 4688)
    • Registry key creation or deletion
    • PowerShell command execution

Network‑Centric Logs

  • Protocols: SSH, VPN, HTTP/S, FTP, SMB
  • Examples:
    • SSH and VPN connections
    • Web traffic and HTTP requests
    • File transfers via FTP or network shares
    • Outbound data transfer (potential exfiltration)

SIEM Architecture and Workflow

External image reference: https://d2y9h8w1ydnujs.cloudfront.net/uploads/content/images/156e333389c9e8c20566023ee01d3e5e43fd6ce76d19ca7085a9d9c1587c54acf54fd32c74a920e81644df3b3094.png

  1. Data Collection: Agents, port forwarding, or APIs ingest logs in real time.
    • Time Pitfalls

One common challenge analysts face when working with a SIEM is dealing with time differences across log sources.

Logs may come from systems in different time zones. Some might use UTC, others local time, and some might not include any timezone information at all.

It is important to remember that the time shown in your SIEM might not match your local time.

For example, if you are in UTC-2 and the SIEM is set to UTC+2, an event that happens at 5 PM local time could appear as 9 PM in the SIEM.

This does not mean the log was delayed in ingestion. It only reflects the timezone difference.

Always check which time zone each log source uses before analyzing events.

Being aware of time differences helps maintain accurate timelines and prevents confusion during investigations.

  1. Normalization: Convert diverse log formats into a consistent schema.
    • Log normalization
      • Normalisation = transform diverse log formats into a single schema so fields are consistent and searchable.
      • Purpose: simplify searches, reduce parsing overhead, and make correlation rules reliable.
      • Typical steps: parse raw message, map fields to standard names, convert types (timestamps, IPs), and add metadata (source, device role).
      • Use or document the standard schema you adopt (examples: CEF, LEEF, ECS) so mappings stay consistent across sources.
      • Without normalisation each new log source requires custom parsing and rule adjustments, which slows detection and triage.
  2. Storage: Centralized, scalable repository for short‑ and long‑term retention.
  3. Correlation & Analytics:
    • Rule‑based detection (e.g., threshold, pattern matching)
    • Statistical and behavioral analytics
  4. Alerting & Notification: Generate alerts for suspicious or anomalous events.
  5. Dashboard & Reporting: Visualize key metrics, trends, and incidents.
  6. Investigation & Response: Triage alerts, perform root‑cause analysis, and remediate.

Core Capabilities

  • Real‑Time Monitoring: Continuous log ingestion and analysis
  • Event Correlation: Link events across hosts and network devices
  • Alerting: Trigger on predefined rules or dynamic thresholds
  • Threat Hunting: Query raw logs for Indicators of Compromise (IoCs)
  • Forensics: Historical log retention for incident investigation
  • Compliance: Reporting templates for HIPAA, PCI‑DSS, SOX, etc.
  • Visualization: Customizable dashboards, charts, and heat maps

Dashboards and Reporting

Dashboards provide a high‑level view of security posture:

  • Alert Summary: Recent triggers, severity, and status
  • Login Activity: Failed vs. successful attempts over time
  • Event Volume: Logs ingested per source and time interval
  • Top Indicators: Frequently triggered rules, top IPs/domains
  • System Health: Data pipeline status, agent connectivity

Rule Creation Steps:

  1. Identify relevant Event IDs and log sources
  2. Define threshold or pattern (e.g., time window, count)
  3. Map field names (e.g., EventID, NewProcessName)
  4. Test rules against historical data
  5. Tune alert thresholds to minimize false positives

SIEM Rules: Out of the Box vs Custom

SIEM rules form the backbone of proactive threat detection. There are two main categories:

  • Vendor Provided (Out of the Box) Rules: Prebuilt by the SIEM vendor to cover generic attack scenarios and common suspicious patterns (e.g., brute‑force login attempts, known malware hashes, port scans).
  • Custom (Human Written) Rules: Developed by in‑house defenders who tailor queries to the organization’s unique environment and normal operational baselines.

What Are SIEM Rules?

They are search queries—running continuously (real‑time) or on a schedule—that inspect incoming log data. When a query matches one or more events, actions can be triggered such as:

  • Generating an alert in the dashboard
  • Sending an email/SMS to the security team
  • Writing the record to a dedicated incident index or ticketing system

Examples of Rule Functionality

CategoryMonitored Activity
AuthenticationFailed logons; logins to disabled accounts; use of high‑privilege accounts; SID changes
Process ExecutionExecution from suspicious paths (temp folders, browser cache); parent/child anomalies (WinWord → cmd.exe); known‑bad file hashes
Network ActivityPort scans; service enumeration; host discovery; unusual outbound volume

False‑Positive Reduction & Tuning

Rules tuned too loosely generate noise. Techniques to reduce false positives:

  1. Thresholds & Time Windows: e.g., only alert on ≥10 failed logins within 10 minutes to catch brute‑force attacks.
  2. Value Exclusions: Exclude known benign sources—e.g., internal vulnerability scanner IPs—to prevent non‑malicious scans from firing alerts.
  3. Conditional Filters: Refine queries by asset group, user role, or business hours.
  4. Rule Testing: Validate against historical logs before production rollout.

Writing Search Queries & Alerts

  • Familiarize yourself with the SIEM’s search language (e.g., Splunk SPL, Elastic DSL).
  • Identify key fields (EventID, src_ip, user, process_path) and their normalized names.
  • Start with a broad query, then iteratively narrow scope (add filters or thresholds).
  • Convert queries into alerts by specifying trigger conditions and notification channels.

Alert Investigation and Triage

When an alert triggers:

  1. Review Rule Details: Which conditions were met?
  2. Examine Associated Events: Timeline and related logs
  3. Determine Validity:
    • False Positive: Tune or disable rule
    • True Positive: Escalate
  4. Incident Response:
    • Query asset owner
    • Contain or isolate affected host
    • Block malicious IPs or domains
  5. Post‑Incident: Document findings, improve rules

SIM vs SEM vs SIEM

  • Security Information Management (SIM):
    • Log collection, storage, and reporting
    • Historical analysis and compliance
  • Security Event Management (SEM):
    • Real‑time event monitoring and alerting
    • Incident detection and response
  • SIEM:
    • Combination of SIM and SEM capabilities
    • Unified platform for end‑to‑end log management and security analytics

Advantages and Disadvantages

Advantages:

  • Centralized log aggregation and normalization
  • Real‑time threat detection and alerting
  • Improved forensic capabilities
  • Compliance reporting and audit readiness
  • Scalability for large data volumes

Disadvantages:

  • High deployment and licensing costs
  • Complexity of integration with diverse log sources
  • Potential for false positives requiring tuning
  • Requires skilled personnel for optimization
