Master File Table (MFT)

Overview

  • The MFT is the heart of NTFS—a 1 KB file located in the root of every NTFS volume.
  • Role: Every time Windows creates, modifies, or deletes a file or folder, it writes or updates a corresponding 1 KB record in the MFT.
  • Forensics Importance: Even after deletion, MFT records can persist (flagged “not in use”), allowing recovery of metadata and, sometimes, data remnants.

Structure of an MFT Record

Each 1 024‑byte MFT record breaks down into:

  1. Record Header
    • Signature (“FILE”)
    • Entry Number: Unique reference for the file/folder.
    • Flags: Mark record “in use” or “deleted.”
    • Sequence & Log Sequence Numbers (LSNs): For journal consistency.
  2. Attributes (in variable order—but typically STDI → FILE_NAME → DATA)
    • $STANDARD_INFORMATION 0x10
      • Timestamps: creation, modification (write), MFT entry change, last access.
      • Other flags (read‑only, hidden, system, compressed, encrypted).
    • $FILE_NAME 0x30
      • Actual filename and parent directory reference.
      • Four timestamps (only three mirrored from STDI by Windows)—more tamper‑resistant.
    • $DATA 0x80
      • Resident: File content stored directly in the record (small files).
      • Non‑resident: Pointers (extent lists) to clusters elsewhere on the volume.
    • Other Optional Attributes
      • $SECURITY_DESCRIPTOR (permissions & ownership).
      • $OBJECT_ID, $REPARSE_POINT, $EA_INFORMATION, etc.
  3. Slack Space
    • Unused bytes at the end of the 1 KB record may contain remnants of previous records—another carving target.
  • Example Attribute Structure in a File’s MFT Record

``python MFT Record ├── $STANDARD_INFORMATION (0x10) │ ├── Created: 2024-06-01 10:00:00 │ ├── Modified: 2024-06-02 12:00:00 │ ├── MFT Modified: 2024-06-03 08:00:00 │ ├── Accessed: 2024-06-04 09:00:00 │ ├── $FILE_NAME (0x30) │ ├── Created: 2024-06-01 10:00:00 │ ├── Modified: 2024-06-02 12:00:00 │ ├── MFT Modified: 2024-06-03 08:00:00 │ ├── Accessed: 2024-06-05 07:00:00 │ ├── $DATA (0x80) │ ├── (Raw file content) ``

Note :

  • If $STANDARD_INFORMATION (SI) and $FILE_NAME (FN) timestamps match, the file likely hasn’t been manipulated.
  • If they differ, it may suggest:
    • Copying/moving the file
    • Timestamp alteration (timestomping)
    • Attacker activity

Forensic Benefits

  • Timeline Reconstruction
    • Creation, write, run‑list, and access times in STDI & FILE_NAME.
  • Deleted File Discovery
    • “In use” flag cleared—but metadata often remains intact until reused.
  • Cross‑Artifact Correlation
    • Compare with NTFS transaction logs ($LogFile) and USN Journal ($UsnJrnl) to verify metadata changes.

Tools for MFT Analysis

  • MFTECmd (Eric Zimmermann) — parses raw MFT records, exports to CSV/JSON.
  • Autopsy / Sleuth Kit — built‑in modules to extract MFT and timeline.
  • FTK Imager — raw hex‑view and export of $MFT file.

MFTECmd.exe

Example Command:

bash
 MFTECmd.exe -f "C:\Location\of\$MFT" --csv "C:\Location\to\save" --csvf name.csv

MACB Time

In the records, we see different timestamps associated with each record. What are they, and how are they different from each other? Well, they are classified as MACB time. MACB refers to the timestamps associated with file system metadata, which provide crucial information about a file's history. These timestamps are:

  • M - Modified: The time when the file's content was last changed.
  • A - Accessed: The time when the file was last read or opened.
  • C - Changed (Metadata): The time when the file's metadata (e.g., permissions, ownership) was last altered.
  • B - Birth (Creation): The time when the file was first created (available in NTFS and other advanced file systems).

Why is the MACB Timestamp Important?

Some of the key points to understand the importance of MACB timestamp are:

  • By analyzing MACB times, we can reconstruct events, such as when a file was created, accessed, modified, or when its metadata changed. This is critical in building a timeline of user or attacker activities.
  • Timestamps can reveal unusual behavior, such as files modified after a system compromise or files accessed during odd hours.
  • Investigators can match timestamps to log entries, user activity, or external events to identify patterns or an attacker's presence.
  • Attackers often modify timestamps to cover their tracks. One such attack is classified as a timestomping attack. We can examine the timestamp to identify the manipulation of the timestamps of files or directories.
