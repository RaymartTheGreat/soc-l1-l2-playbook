Original Notes

Server Message Block (SMB)

SMB is a network protocol used to provide shared access to files, folders, printers, and other resources over a network. It's primarily used in Windows environments, but Linux systems can interact with SMB using tools like Samba.

It operates over:

  • Port 445 (modern SMB — direct over TCP)
  • Port 139 (legacy NetBIOS over TCP/IP)

Common SMB Use Cases

  • Accessing shared folders across a network
  • File reading and writing from remote systems
  • Printing via network-shared printers
  • Administrative tasks like GPO delivery or log collection
  • Used internally by tools like PsExec and Remote Administration features

Authentication in SMB

Before access is granted, the client must authenticate with the server. There are two main authentication methods:

1. NTLM (NT LAN Manager)

  • Challenge-response mechanism
  • Common in environments without Active Directory or when Kerberos isn't available
  • Used as a fallback protocol

2. Kerberos

  • Ticket-based system for secure and scalable authentication
  • Used in domain environments with proper time synchronization
  • Preferred over NTLM due to enhanced security and performance

NTLM Authentication Flow in SMB

This is typically observed in packet captures (e.g., Wireshark). The steps are:

  1. Negotiate Message

Client announces supported features and NTLM version

  1. Challenge Message

Server sends a random value (nonce)

  1. Authenticate Message

Client encrypts the challenge using a hash of its password and returns the result

  1. Validation

Server verifies the response against its stored credentials. If it matches, access is granted.

NTLM Security Considerations

  • Vulnerable to pass-the-hash and relay attacks
  • Lacks mutual authentication
  • Avoided in modern enterprise environments where Kerberos is supported

Kerberos Authentication Flow in SMB

Kerberos is the default and more secure protocol in Active Directory domains. The process involves interaction with the Key Distribution Center (KDC) and goes as follows:

  1. TGT Request

Client requests a Ticket Granting Ticket from the KDC

  1. TGT Response

KDC returns a TGT, encrypted with a session key

  1. Service Ticket Request

Client asks for a service ticket for the SMB service (cifs/servername)

  1. Service Ticket Response

KDC issues a ticket specific to that SMB server

  1. Authentication to SMB Server

Client presents the ticket to the SMB server, which validates and grants access

SMB: NTLM vs Kerberos

FeatureNTLMKerberos
Protocol TypeChallenge-responseTicket-based
Mutual AuthenticationNoYes
DC Required at RuntimeYesOnly for ticket issuance
PerformanceSlower (multiple round-trips)Faster (cached tickets)
SecurityWeak (hash theft, relays)Stronger (encryption, replay defense)
EnvironmentWorkgroups, legacy systemsAD domain-joined environments

Where to Observe This in Wireshark

To analyze SMB authentication:

  1. Filter by smb or smb2
  2. Look for Negotiate Protocol Request/Response
  3. For NTLM: identify NTLMSSP_NEGOTIATE, NTLMSSP_CHALLENGE, and NTLMSSP_AUTH messages
  4. For Kerberos: check for AS-REQ, AS-REP, TGS-REQ, and TGS-REP messages prior to SMB session
