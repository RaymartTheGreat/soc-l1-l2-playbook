Original Notes

Kerberos authentication

Kerberos is a secure authentication protocol used in Windows environments. It uses a ticket system and strong encryption to verify identities, even on open networks. In an Active Directory setup, the Domain Controller acts as the Key Distribution Center (KDC) with two main parts: the Authentication Service (AS) and the Ticket Granting Service (TGS), which issue tickets to confirm identities.

Key Components

  • Client: The user or computer trying to authenticate.
  • Domain Controller (KDC): The server hosting the AD database and handling authentication requests. It includes:
    • Authentication Service (AS): Issues the initial Ticket Granting Ticket (TGT).
    • Ticket Granting Service (TGS): Issues service-specific tickets based on the TGT.

Ticket Granting Server (TGS)

  • Ticket Granting Ticket (TGT): A temporary credential that allows the client to request service tickets without re-sending its password.
  • Service Ticket: A ticket that allows the client to authenticate to a specific service (e.g., file server, SQL server).

  • Tickets: is a credential that a client presents to an application server (whether it's the KDC or another service) to prove its authenticity. These tickets are issued by the KDC—specifically, the Authentication Server (AS) and the Ticket Granting Server (TGS)—and are encrypted using a secret key belonging to the intended service.

Because this shared secret is known only to the KDC and the service provider, the client has no access to the ticket’s contents and cannot modify them in any way.

Example on information included in the ticket

Principal Client

Principal Client: Refers to the requesting user's principal identity.

Principal Service

Principal Service: Represents the principal identity of the service for which the request is intended.

Client IP Address

The IP address of the client machine where the ticket is valid for use, included in the IP_list.

Timestamp

Timestamp: The date and time of the request in timestamp format.

Ticket Lifetime

Lifetime: Specifies the maximum validity duration of the ticket.

Session Key (SK Service)

SK Service: The session key, which plays a crucial role in authentication (explained further in section 4.4).

Why Ticket Lifetime Matters

Each ticket has a default validity of 10 hours (600 minutes). This limitation is crucial because, once issued, the authentication server has no direct control over the ticket. While administrators can restrict users from obtaining new tickets, they cannot revoke tickets that have already been issued. Therefore, setting a finite lifetime ensures security by minimizing potential misuse over time.

KRBTGT Service Account

KRBTGT is a built-in service account present in every Active Directory domain that functions as the Key Distribution Center (KDC) account for Kerberos authentication. It is used exclusively to encrypt the Ticket Granting Ticket (TGT) with a key derived from its password, ensuring that only authenticated users can access network resources.

If an attacker gains access to this password, they could forge “Golden Tickets” to impersonate legitimate users.

Golden Tickets

  • By compromising or resetting the KRBTGT password hash, attackers can generate forged Ticket Granting Tickets (TGTs), effectively allowing them to impersonate any user in the domain.
  • A successful Golden Ticket attack using the KRBTGT account can grant near-unlimited access across the network, making it one of the most coveted targets in an AD compromise.
  • Once the KRBTGT account is compromised, detection is challenging because the forged tickets are cryptographically valid from the perspective of the domain controllers.

To enhance security, the KRBTGT account is disabled for direct logins, minimizing its exposure. Its design as a non-interactive account emphasizes the need for a strong, well-protected password, making it a critical component in maintaining the integrity of the entire Kerberos authentication process.

Step-by-Step Authentication Process

1- Client Authentication Request (AS_REQ)

  • Action: The client initiates the process by sending an AS_REQ message to the KDC.
  • Content: The request contains the client’s identity and a request for a TGT.
  • Encryption: The request is sent in plaintext, but sensitive parts of the response will be encrypted.

2- KDC Verifies Credentials and Issues TGT (KRB_AS_REP)

  • Action: The KDC’s Authentication Service verifies the client’s identity by checking the AD database.
  • Response: If the credentials are valid:
    • The KDC issues a Ticket Granting Ticket (TGT).
    • A session key is generated.
  • Encryption: The TGT is encrypted using a key known only to the KDC (specifically for the TGS), and the session key is encrypted with a key derived from the client’s password.

3- Client Decrypts the AS Response

  • Action: The client uses its password (or a key derived from it) to decrypt the session key from the AS response.
  • Outcome: With the session key, the client can now use the TGT for further requests without sending the password again.

4- Service Ticket Request (TGS_REQ)

  • Action: When the client wants to access a specific service, it sends a TGS_REQ message to the KDC’s Ticket Granting Service.
  • Content: This request includes:
    • The previously obtained TGT.
    • An authenticator (a timestamped message encrypted with the session key).
    • The identity of the target service.

5- KDC Issues the Service Ticket (TGS_REP)

  • Action: The TGS validates the TGT and the authenticator.
  • Response: If valid, the TGS issues a service ticket for the requested service.
  • Encryption: The service ticket is encrypted with the service’s secret key (known only to the KDC and the service).

6- Client Presents the Service Ticket(AP_REQ)

  • Action: The client sends an AP_REQ (Application Request) message to the target service.
  • Content: This message contains the service ticket and a new authenticator proving the client’s identity.
  • Outcome: The service can now verify the authenticity of the ticket.

7- Service Validates the Ticket and Establishes a Session

  • Action: The target service decrypts the service ticket using its secret key.
  • Outcome: If the ticket is valid, the service grants access to the client and may establish a secure session (using the provided session key) for subsequent communication.

    S-->>C: AP_REP (Optional: confirmation, establish secure session)

Summary

  1. AS Request: The client requests a TGT from the KDC.
  2. AS Reply: The KDC verifies the client’s credentials and issues a TGT along with a session key.
  3. TGS Request: The client uses the TGT to request a service ticket for a particular service.
  4. TGS Reply: The KDC issues a service ticket encrypted for the target service.
  5. AP Request: The client presents the service ticket to the target service.
  6. Service Validation: The service verifies the ticket and, if valid, grants access to the client.

This process ensures that passwords are not transmitted over the network after the initial exchange and that all communications can be securely encrypted, making Kerberos a robust authentication mechanism in AD environments.

How Attackers Can Exploit Kerberos

Despite its robust design, there are several potential vulnerabilities and attack vectors:

1- Brute Force/Password Guessing Attacks

  • Attack Vector:
    • Since the initial key used to encrypt the session key in the AS reply is derived from the client’s password, weak passwords can be exploited.
  • Impact:
    • An attacker who can guess or crack weak passwords may derive the encryption key and decrypt sensitive messages.
  • Mitigation:
    • Enforce strong password policies and use multi-factor authentication where possible.

2- Pass-the-Ticket Attacks

  • Attack Vector:
    • If an attacker gains access to a valid TGT or service ticket (for example, via memory scraping on a compromised machine), they can reuse it to authenticate without needing the user’s password.
  • Impact:
    • This can lead to lateral movement within a network.
  • Mitigation:
    • Regularly clear cached tickets, monitor for abnormal ticket usage, and use endpoint protection measures.

3- Kerberoasting

  • Attack Vector:
    • Attackers request service tickets for services with service accounts, then extract and attempt to crack the service account’s encrypted credentials offline.
  • Impact:
    • Once a service account password is compromised, attackers can access services and possibly escalate privileges.
  • Mitigation:
    • Use strong, complex passwords for service accounts, monitor for unusual service ticket requests, and restrict permissions.

4- Golden Ticket Attacks

  • Attack Vector:
    • If an attacker compromises the KDC’s secret (often the KRBTGT account in AD), they can forge valid TGTs, granting virtually unlimited access across the domain.
  • Impact:
    • This is one of the most severe compromises, as it allows attackers to impersonate any user.
  • Mitigation:
    • Protect the KRBTGT account with stringent security measures, monitor for anomalous ticket creation, and periodically reset the KRBTGT account’s password.

5- Silver Ticket Attacks

  • Attack Vector:
    • Instead of compromising the KDC, an attacker may target a specific service’s key to forge a service ticket (silver ticket).
  • Impact:
    • Allows attackers to gain unauthorized access to a particular service without detection by the KDC.
  • Mitigation:
    • Secure service account keys, ensure services are updated, and monitor for abnormal service ticket activity.

6- Replay Attacks

  • Attack Vector:
    • Attackers capture and resend valid authenticators or tickets.
  • Impact:
    • May allow repeated access if timestamps are not correctly enforced.
  • Mitigation:
    • Use strict time synchronization (e.g., via NTP) and short ticket lifetimes to limit the effectiveness of replays.
