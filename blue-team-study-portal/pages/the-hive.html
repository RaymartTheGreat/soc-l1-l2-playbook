Original Notes

The Hive

TheHive is an open-source Security Incident Response Platform (SIRP) designed to centralize incident investigations and case management for SOCs, CSIRTs and CERTs. It enables teams to collect alerts, create investigations (cases), track actions with tasks, enrich observables, and collaborate in real time.

Official resources:

  • Project website: https://thehive-project.org/
  • GitHub: https://github.com/TheHive-Project/TheHive

Core capabilities

  • Case management: Create cases representing incidents; attach tasks, observables (IP, domain, URL, hash, file), notes, and evidence. Cases can be templated to standardize workflows.
  • Tasking & templates: Break cases into tasks. Use templates to ensure repeatable, auditable response steps.
  • Collaboration: Multi-user, live-stream updates so analysts can work simultaneously on the same case.
  • Observable enrichment & analysis: Send observables to Cortex for automated analysis (analysers) and enrichment from threat intel sources.
  • Alert ingestion & triage: Import alerts from SIEMs, IDS, email, or feeder connectors. Triage alerts and escalate to cases when needed.
  • Active response (Responders): Execute actions via responders (e.g., block IPs) when enabled.
  • REST API / Automation: Full REST API for integration and automation (create cases, update observables, trigger analysers/responders) — useful for SOAR playbooks.
  • Dashboards & KPIs: Custom dashboards for case metrics, triage status, SLAs, and other KPIs.

Typical integrations

  • Cortex — analyser & responder engine for enrichment and active-response tasks.
  • MISP — share/import/export IOCs and events between TheHive and threat intel communities.
  • SIEMs / feeders — e.g., custom connectors, DigitalShadows2TH, ZeroFox2TH, email feeders.
  • External tools via API — firewalls, endpoint platforms, ticketing systems (ServiceNow/Jira) via automation.

Roles & permissions

TheHive ships with pre-configured profiles; admins can create custom profiles and fine-grained permissions. Common profiles:

  • admin — global administrative permissions (manage global config, orgs, etc.).
  • org-admin — manage organisation-level users/config and full case/task/observable management.
  • analyst — create/edit cases, tasks, observables; run analysers & responders.
  • read-only — view-only access to cases, tasks, and observables.

Key permission areas (examples): manageCase, manageObservable, manageAlert, manageUser, manageProfile, manageCaseTemplate, manageAnalyse, manageAction. Cortex-related permissions become relevant when the Cortex connector is enabled.

Notes

  • TheHive supports multiple organizations (useful for multi-tenant setups or segmentation between teams).
  • Create custom case fields, observable types, analyser templates, tags, and case templates to enforce SOC processes.
  • Cases and tasks provide an auditable trail (who did what and when) — important for compliance.
  • You can import TTPs to link cases and tasks to ATT&CK techniques for reporting and metrics.

Operational guidance

  • Start with templates: Build case and task templates for common incidents (phishing, malware, lateral movement) to speed triage and maintain consistency.
  • Enrich early: Configure Cortex analysers for automatic enrichment of common observable types (reputation, passive DNS, WHOIS, sandbox results).
  • Integrate feeders: Point relevant SIEM alerts into TheHive but tune filters to avoid alert flooding / convert only high-confidence alerts into cases or create a triage queue.
  • Use tags & custom fields: Tags and custom observable types make reporting and correlation easier later.
  • Define SLAs & dashboards: Track case age, time-to-triage, and closure metrics on dashboards to measure SOC performance.
  • Access control: Apply least privilege: analysts should have access to create/update cases but only org-admins should manage users and global config.

Example workflows

  1. Alert ingestion → triage → case: SIEM alert feeder creates an alert in TheHive → analyst triages → alert converted to case with associated observables and tasks.
  2. Observable enrichment: Analyst adds an IP observable → sends to Cortex analysers (reputation, passive DNS, sandbox) → results attach to observable and inform next steps.
  3. Active containment: Analyst identifies malicious IP → use responder to push block to firewall or endpoint tool via REST/API.

Benefits for a SOC

  • Single source of truth for incidents and evidence.
  • Faster, standardized investigations via templates and automation.
  • Better collaboration and continuity between shifts.
  • Easier reporting and compliance through auditable case records and dashboards.
