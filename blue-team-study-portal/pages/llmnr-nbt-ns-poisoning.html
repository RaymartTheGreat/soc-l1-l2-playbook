Original Notes

LLMNR / NBT-NS Poisoning

LLMNR (Link-Local Multicast Name Resolution) and NBT-NS (NetBIOS Name Service) are fallback name-resolution protocols used by Windows when DNS fails. Poisoning those requests means an attacker answers the broadcasted name lookup, impersonates the requested host, and causes the victim to attempt authentication—leaking NTLM/Negotiate authentication data (challenge/response) that can be captured for cracking or relaying.

MITRE ATT&amp;CK ID: T1557.001

1- Quick facts

  • Protocols / ports: LLMNR = UDP 5355, NBT-NS = UDP 137.
  • IPv4 multicast (LLMNR analogue): 224.0.0.252 — IPv6 LLMNR: ff02::1:3.
  • Common tools (observed in labs): Responder / Inveigh (listener/catcher), ntlmrelayx (relay), responder-style sniffers.
  • Data leaked: Username, domain, NTLMv2 challenge/response (NTProofStr + NTLMv2 response). No plaintext password is sent.

2- How the attack flows

  1. User tries \\DCC01\share (typo or DNS failure).
  2. Windows issues LLMNR/NBT-NS query: “Who is DCC01?” (broadcast).
  3. Attacker replies: “I am DCC01 — here’s my IP.”
  4. Victim connects to attacker and performs Windows authentication (SMB/HTTP/LDAP).
  5. Attacker captures NTLM challenge/response packets (NTProofStr + blob).
  6. Attacker either:
    • cracks the NTLMv2 hash offline, or
    • relays the auth to another service (SMB relay) to impersonate the user.

3- What you see in network captures (Wireshark / PCAP)

  • LLMNR queries: udp.port == 5355 or display name llmnr.
  • NBT-NS queries: udp.port == 137 or nbns.
  • NTLM/SMB handshake: smb2 && ntlmssp — three packets: NEGOTIATE → CHALLENGE → AUTH.
  • Fields to extract for cracking (for forensic analysis):
    • User and Domain: in NTLMSSP_NEGOTIATE / SMB2 Session Setup (Account / Domain).
    • ServerChallenge: in NTLMSSP_CHALLENGE.
    • NTProofStr (first 16 bytes of NTLMv2 response) and remainder of the NTLMv2 Response: in NTLMSSP_AUTH.
  • Wireshark filters (quick):

``python udp.port==5355 || udp.port==137 || smb2 || ntlmssp ``

OR

``python llmnr || nbns || smb2.session_setup || ntlmssp ``

4- Detection & hunting

  • LLMNR/NBNS traffic bursts from many hosts (broadcasts).
  • Unusual responder host answering many LLMNR/NBNS queries (not a known DC or file server).
  • SMB session setup to unexpected hosts immediately after LLMNR: client IP → unusual server IP with many NTLMSSP_AUTH messages.
  • IPv6 LLMNR: traffic to ff02::1:3.

5- Impact

  • Credential theft: NTLMv2 responses can be cracked offline (depends on password strength).
  • Credential relay: If SMB signing is not enforced, captured responses can be relayed to authenticate elsewhere.
  • Lateral movement: Stolen or relayed credentials enable access to other hosts/services.

6- Mitigation & hardening

  • Disable LLMNR and NetBIOS name resolution where possible via Group Policy.
  • Enforce SMB signing and require channel binding to prevent relay attacks.
  • Restrict NTLM: audit then progressively disable NTLM; use NTLM auditing first.
  • Harden passwords & enforce MFA — reduces risk from cracked hashes.
  • Whitelist/segment management traffic: prevent workstations from accepting inbound SMB except to allowed servers.
  • Network detection: monitor for new/unknown hosts answering LLMNR/NBNS and isolate suspicious responders.

7- Summary

  • Protocol Targeted: LLMNR / NBT-NS
  • Authentication Material: NTLMv2 challenge/response (NTProofStr + NTLMv2 blob)
  • Scope of Attack: Local network credential harvesting and relay → lateral movement / privilege escalation
  • Key Tools: Responder, Inveigh, ntlmrelayx
