Original Notes

Memory Forensics Playbook

Memory forensics is the process of analyzing a RAM image to reconstruct the system's runtime state at the time of capture. It surfaces ephemeral evidence — running processes, loaded modules, network sockets, decrypted content, and in-memory malware — that often does not persist to disk.

Why it matters :

  • Reveals in-memory-only malware and injected code.
  • Shows live network connections and command-lines that may be absent on disk.
  • Helps validate or refute hypotheses made from disk analysis.

Acquisition

Rules:

  • Capture RAM before reboot or shut down; volatility of memory means you lose IPv4/IPv6 sockets, process memory, and cryptographic keys on power loss.
  • Record the capture method, timestamps, and who performed the acquisition (chain of custody basics).
  • Prefer hardware-assisted or trusted OS-level acquisition tools on live systems when possible.

Common tools:

  • Windows: Belkasoft Live RAM, Magnet RAM Capture, FTK Imager (live), DumpIt.
  • Linux: dd from /dev/mem, LiME (Linux Memory Extractor).

Capture tips:

  • Note system uptime and active user sessions.
  • If possible, isolate host from network to prevent further comms, but avoid rebooting.

Preparing the image: profile & basic checks

First steps:

  1. Run image identification (e.g., Volatility imageinfo) to see suggested OS profiles and the capture timestamp.
  2. If imageinfo is ambiguous, run kdbgscan or equivalent to find the KDBG offset and an accurate profile.
  3. Record architecture (x86 vs x64), OS build, and profile used — mismatches break analysis.

Why the profile matters: parsing structures (EPROCESS, VAD, KDBG) requires the correct OS layout. Wrong profile → false negatives or plugin failures.

Short, repeatable workflow

  1. imageinfo → choose candidate profile.
  2. kdbgscan if ambiguous → finalize profile.
  3. pslist, pstree, psxview — enumerations.
  4. netscan / connscan — network artifacts tied to PIDs.
  5. malfind, psinfo, vads — look for code injection/hollowing.
  6. printkey / registry plugins — persistence keys.
  7. mftparser / filescan — MFT fragments & file remnants.
  8. Dump suspicious regions and run yara / static analysis offline.

Processes: what to collect and why

Minimal process artifact set:

  • PID, PPID, image name & full path, command line, start/exit time, token privileges, thread count, handles, loaded modules.

Useful plugins & purpose:

  • pslist — canonical process list.
  • pstree — parent-child relationships.
  • psxview — cross-check multiple enumeration sources to detect hidden processes.
  • psinfo / procdump — extract process memory, PEB/VAD details.
  • malfind — finds suspicious executable scans in process VADs (RWX memory, shellcode-like footprints).

Red flags:

  • System process names running from non-standard paths (e.g., C:\Users\...\lsass.exe).
  • Short-lived suspicious children of user apps (Office → mshta, powershell).
  • PIDs present in psscan but missing from pslist or vice versa.

Code injection & hiding techniques

Common signs of injection:

  • Memory regions with execute permission but lacking a valid PE header.
  • PAGE_EXECUTE_READWRITE regions containing shellcode-like sequences.
  • VAD entries that don’t map to a file on disk or have MEM_PRIVATE with executable permissions.

Checks to run:

  • Compare pslist vs psxview and psscan.
  • Run malfind / envars / ldrmodules to see unusual module load behavior.
  • Dump suspicious addresses and scan with YARA or a disassembler.

Networking

What to extract:

  • Local IP/port, remote IP/port, PID, protocol, connection state, timestamp .

Tools:

  • netscan (reconstructs TCP/UDP endpoints from memory), connscan (searches for connection objects).

Hunting cues:

  • Non-typical processes (e.g., svchost child running from C:\Users) making outbound connections.
  • Persistent periodic connections (beacons) to the same remote host.
  • Connections to known-malicious or poorly-reputed IPs — flag for enrichment.

Persistence locations to check

Registry & config:

  • HKLM\Software\Microsoft\Windows\CurrentVersion\Run and RunOnce (HKLM/HKCU).
  • CurrentControlSet\Services (Windows services with Start=2 or Start=0).
  • AppInit_DLLs, Winlogon (Userinit, Shell).
  • Scheduled Tasks (TaskCache\Tree, .job remnants).

Filesystem & other:

  • Startup folder, Task Scheduler files, signed/unsigned driver entries.
  • Look inside process command-lines for schtasks, sc create, or reg add patterns.

File and filesystem traces in RAM

What to look for:

  • MFT fragments, resident file data, filename strings, and recently opened file handles.
  • filescan and mftparser can recover fragments and timestamps, useful for deleted or overwritten files.

Why it helps:

Small scripts, dropped loaders, or configuration files occasionally live fully in memory and can be reconstructed even if deleted from disk.

Common in-memory artifacts

  • Credentials: clear-text or hashed credentials may appear in memory (process command-lines, environment variables, or credential caches). LSASS is a primary target: do not dump or export LSASS contents without appropriate authorization and secure handling.
  • Decrypted payloads: packers and droppers often decrypt payloads only in memory — these can be recovered by dumping RWX regions.
  • Command histories and shell buffers: many shells and scripting engines keep recent commands in memory buffers; they often contain attacker commands or URLs.

Timeline reconstruction

Approach:

  • Pull timestamps from process creation, network sessions, registry LastWrite times, and MFT fragments.
  • Correlate these events to build a timeline: execution → network comms → persistence actions → cleanup.

Tip: convert all times to UTC and include the host timezone in the report.

Reporting & evidence preservation

  • Export suspect memory regions and store them with checksums.
  • Document exact versions of tools and plugins used.
  • Produce a narrative (what happened, how you know, confidence, remediation suggestions) and append raw outputs in an annex.

Small cheat sheet: commands & quick examples

bash
# image identification
vol.py -f memory.img imageinfo

# profile-finding
vol.py -f memory.img kdbgscan --output=kdbg.txt

# process listing
vol.py -f memory.img --profile=Win10x64_#### pslist

# network scan
vol.py -f memory.img --profile=Win10x64_#### netscan

# find suspicious executable pages
vol.py -f memory.img --profile=Win10x64_#### malfind

# print registry key from memory
vol.py -f memory.img --profile=Win10x64_#### printkey -K "\Microsoft\Windows\CurrentVersion\Run"

Note

When you must collect volatile data before imaging, prefer non-invasive reads and keep commands minimal. Examples to run on Windows (record exact output and timestamps):

  • whoami /all — current user, groups, and privileges.
  • wmic process list full or tasklist /v — quick process inventory.
  • netstat -ano — active network connections and owning PIDs.
  • route print — routing table.
  • systeminfo — OS build, patch level, uptime.

Avoid tools or commands that will load code into memory or change many handles (installing agents, enabling debug options, or reconfiguring services).
