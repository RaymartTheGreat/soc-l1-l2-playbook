Original Notes

Threat Hunting: Endgame

This room focuses on the "Actions on Objectives" phase of the Cyber Kill Chain—where attackers execute their final goals such as data exfiltration, data destruction, encryption for ransom, or credential theft. The objective is to implement the threat hunting process to detect and investigate these advanced adversary actions using the MITRE ATT&CK framework as a guide.

1- Collection (TA0009)

Definition:

Techniques adversaries use to gather valuable data from target systems. Collected data may be exploited for further compromise, intelligence gathering, monetization, or to facilitate lateral movement and privilege escalation.

Commonly Used Techniques :

  • Network-based Collection:

    • Man-in-the-middle attacks, ARP/LLMNR poisoning, SMB relay, DHCP spoofing, hijacking, or traffic dumps.

  • Keylogging and Input Capture:

    • Recording keystrokes using API calls, hooks, or malicious drivers.

  • Data Harvesting from Repositories:

    • Collecting data from local files, cloud storage, or databases.

  • Other Techniques:

    • Capturing screenshots, clipboard data, or system memory dumps.

Hunting Collection Activities

The hunt for collection activities involves identifying anomalous data gathering methods, which are closely tied to the attacker’s objectives. One specific case example is hunting for keylogger activity.

Case Example: Hunting Keylogging

Keyloggers record keystrokes to capture sensitive information. This example demonstrates a hunting exercise for detecting keylogger activities using API calls and pattern matching.

Keylogging Characteristics:

Hunting Methodology:

  1. Initial Pattern Search:
    • Objective: Identify logs containing suspicious keylogger-related API calls.
    • KQL Query Example:

``jsx GetKeyboardState OR SetWindowsHook OR GetKeyState OR GetAsynKeyState OR VirtualKey OR vKey OR filesCreated OR DrawText ``

  1. Investigate Suspicious Log Files:
    • Data Source:

Focus on logs from Microsoft-Windows-PowerShell/Operational where keylogger patterns are observed.

    • Key Fields to Display:

winlog.channel, winlog.event_data.Path, winlog.event_data.ScriptBlockText

  1. Correlate Suspicious Script and Database Files:
    • Objective:

Identify suspicious file names such as chrome-update_api.ps1 or chrome_local_profile.db.

    • KQL Query Example:

``jsx chrome-update_api.ps1 OR chrome_local_profile.db ``

    • Key Fields to Display:

winlog.channel, winlog.event_data.Path, winlog.event_data.ScriptBlockText, winlog.event_data.CommandLine, winlog.event_data.Data.#text

  1. Detailed Analysis of the Keylogger Database:
    • Objective:

Examine the contents of the keylogger database file to verify the logging of keystrokes.

    • KQL Query Example:

``jsx winlog.event_data.ScriptBlockText: "chrome_local_profile.db" ``

    • Key Fields:

winlog.event_data.Path, winlog.event_data.ScriptBlockText, winlog.event_data.Payload

  1. Event Correlation and Timeline Construction:
    • Approach:

Use the "View surrounding documents" feature to correlate parent-child process relationships and build a timeline from the initial suspicious command execution to subsequent keylogger activity.

2- Exfiltration (TA0010)

Definition:

Techniques adversaries use to steal or leak data from a compromised system or network. This phase often employs compression and encryption to maximize data transfer while avoiding detection.

Examples of Methods:

  • Data Transfer to C2 Servers/Channels:

    • Sending stolen data directly to attacker-controlled servers.

  • Transfer over Alternative Protocols:

    • Using non-standard or less-monitored channels (e.g., Bluetooth, web services, cloud storage).

  • Traffic Duplication:

    • Duplicating legitimate traffic to embed exfiltrated data.

  • Exfiltration via Encrypted/Unencrypted Channels:

    • Utilizing encrypted tunnels or even unencrypted paths with strict size limits.

  • Exfiltration over Portable Devices:

    • Transferring data via removable media.

Case Example: Data Exfiltration over ICMP

This case demonstrates a hunting exercise focused on detecting data exfiltration over the ICMP protocol using native Windows artifacts. Instead of relying solely on network traffic, this exercise examines system-level events and process executions to reveal exfiltration activity.

Hunting Methodology

1. Initial Pattern Search

  • Objective:

Identify system tool calls that may be used for data transfer.

  • Approach:

Search for process execution logs containing common exfiltration tool keywords.

  • KQL Query Example:

``jsx $ping OR $ipconfig OR $arp OR $route OR $telnet OR $tracert OR $nslookup OR $netstat OR $netsh OR $smb OR $smtp OR $scp OR $ssh OR $wget OR $curl OR $certutil OR $nc OR $ncat OR $netcut OR $socat OR $dnscat OR $ngrok OR $psfile OR $psping OR $tcpvcon OR $tftp OR $socks OR $Invoke-WebRequest OR $server OR $post OR $ssl OR $encod OR $chunk OR $ssl ``

  • Insight:

The query yields one hit, indicating a suspicious system tool call potentially related to data exfiltration.

2. Focusing on Suspicious System Calls

  • Objective:

Narrow down the activity by filtering for specific system calls.

  • Approach:

Run a focused query on system calls related to network information:

KQL Query Example:

``jsx System.Net.Networkinformation.ping* ``

  • Key Field:

winlog.event_data.Path

  • Insight:

The result shows a suspicious PowerShell script executing a ping-related call, suggesting the script is transferring files using ICMP packets.

3. Detailed Analysis of the Exfiltration Script

  • Objective:

Identify the specific exfiltration activity by analyzing script details.

  • Approach:

Filter for the exfiltrating script using a query such as:

KQL Query Example:

``jsx icmp4data.ps1* ``

  • Key Fields to Display:
    • winlog.event_data.Path
    • winlog.event_data.ScriptBlockText
  • Insight:

The query returns precise results that identify the exfiltration script, the exfiltrated file, and the upload destination server.

Total number of sent ICMP packets

Name of the exfiltrated document & the server's IP address

3- Impact (TA0040)

Definition:

Techniques used by adversaries to disrupt availability, compromise system integrity, or hinder functionality. These techniques include data destruction, data manipulation, ransomware, defacement, and other forms of disruptive actions.

Examples of Methods:

  • Ransomware: Encrypting data to extort payment.
  • File Destruction and Removal: Deleting or corrupting critical files.
  • Data Manipulation: Altering data to render it unreliable or unusable.
  • Service Disruption: Disabling or altering system services (e.g., shadow copy deletion).
  • Account and Access Manipulation: Changing permissions to lock out legitimate users.
  • System Configuration Changes: Modifying network settings or boot configurations to disrupt operations.

Case Example: Data Disruption/Manipulation

This case example demonstrates a hunting exercise for detecting data disruption and manipulation, focusing on native system processes used for shadow backup removal and corruption of system recovery points—similar to techniques used by advanced persistent threats like the Olympic Destroyer APT group.

Hunting Methodology

1. Initial Pattern Search

  • Objective:

Identify system tool calls that lead to data disruption or manipulation.

  • Approach:

Search for logs with command-line strings that indicate the use of native system utilities which could be used to delete or corrupt data.

  • KQL Query Example:

``jsx del OR rm OR vssadmin OR wbadmin OR bcdedit OR wevutil OR shadow OR recovery OR bootstatuspolicy ``

2. Visualizing Log Sources

  • Objective:

Determine which log sources are providing the most insight into potential Impact activities.

  • Approach:

Visualize events by filtering on the winlog.channel field and selecting a table format.

  • Insight:

The Security log may provide low-hanging fruit as it records system tool executions and other critical events.

3. Narrowing Down to Specific Events

  • Objective:

Isolate the starting point of disruptive activity.

  • Approach:

Filter for a specific suspicious event by its Process ID:

``jsx winlog.event_data.ProcessId: "1972" ``

    • Key Fields to Display:
      • winlog.event_data.CommandLine
      • winlog.event_data.ParentProcessName
      • winlog.event_data.ProcessId
      • winlog.event_data.ParentProcessId
      • winlog.event_data.ParentImage

  • Insight:

This query reveals a clear parent-child process relationship, identifying the primary process that initiated the disruptive actions.
