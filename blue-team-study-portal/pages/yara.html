YARA – Pattern Matching for Malware Detection

“The pattern matching swiss knife for malware researchers (and everyone else)” — VirusTotal

YARA is a rule-based engine used for identifying malware by looking for binary and textual patterns in files. It's a staple tool for threat hunters, malware analysts, and incident responders who need to scan files, memory, or entire systems for known artifacts or behaviors.

Instead of relying solely on hash values or signatures, YARA allows you to write detection logic based on:

  • Specific strings (plaintext, hex, regex)
  • Structural characteristics (via modules)
  • Logical conditions (size, repetition, combinations)

What YARA Rules Actually Do

YARA rules act like behavioral fingerprints. They detect:

  • Known malware families
  • Artifacts linked to threat actors or campaigns
  • Indicators like command-and-control (C2) strings, file paths, encoders, and memory patterns

The goal isn't just to say “this is malware,” but to define how and why a file or sample is suspicious.

Anatomy of a YARA Rule

1. rule Name

  • Starts with the rule keyword followed by a descriptive identifier.
  • Should tell you what the rule is for — threat name, family, behavior, etc.
java
rule M_APT_Dropper_Rootsaw_Obfuscated

2. meta Section (Optional but Useful)

This is where you describe the rule. It doesn’t affect detection but helps analysts understand:

  • Who wrote it
  • What it’s detecting
  • Where it came from
java
meta:
  author = "Mandiant"
  description = "Detects obfuscated ROOTSAW payloads"

3. strings Section

This is where you define the actual indicators. These can be:

  • Text strings
  • Hex patterns
  • Regular expressions

Each string is assigned a variable (e.g., $s1, $http, $codeblock) for use in the condition logic later.

java
strings:
  $1 = "function _"
  $2 = "new XMLHttpRequest();"
  $3 = { 48 8D 0D ?? ?? 00 00 }

4. condition Section

This is the logic block. It tells YARA when to trigger a match based on the defined strings and file properties.

java
condition:
  all of them

This line means: match if all the defined strings appear in the file.

Example of a YARA Rule

  1. Rule name: M_APT_Dropper_Rootsaw_Obfuscated. The rule’s title is well-chosen and gives the user a good idea of what to use it for. In this case, it is to detect a dropper called Rootsaw that is obfuscated.
  2. Meta: It is good practice to include relevant data that provides more information about the rule. This helps the user of the YARA rule know what to use the rule for, who wrote it, and where to apply it.
  3. Strings: The strings included in this example help the user find a file containing those strings. How do malware analysts choose those strings? They analyze the malware and determine what uniquely identifies it. The strings used are text strings. The first two lines are straightforward.
  4. Condition: This rule requires that all defined strings be present to have a match. This means all the strings defined in part 3 must have a match in the same file being matched against.

Example: Obfuscated Malware Rule (ROOTSAW)

Below is a complete example of a YARA rule designed to detect obfuscated samples of a malware dropper known as ROOTSAW. It contains all four components we just reviewed:

java
rule M_APT_Dropper_Rootsaw_Obfuscated
{
  meta:
    author = "Mandiant"
    description = "Detects obfuscated ROOTSAW payloads"

  strings:
    $a1 = "function _"
    $a2 = "new XMLHttpRequest();"
    $a3 = "'\\x2e\\x7a\\x69\\x70'"
    $a4 = "'\\x4f\\x70\\x65\\x6e'"
    $a5 = "\\x43\\x3a\\x5c\\x57"

  condition:
    all of them
}

Why This Rule Works

  • $a1 and $a2 match JavaScript function names often used in droppers.
  • $a3, $a4, and $a5 are hex-encoded string values seen in obfuscated code.
  • all of them enforces that every indicator must be present, which helps reduce false positives.

Why Malware Uses Strings

Malware, like any software, uses strings to define behavior API calls, file paths, network URLs, error messages, and encryption keys. These strings make for high-confidence indicators when writing YARA rules.

TypeExampleDescription
Ransomware12t9YDPgwueZ9NyMgw519p7AA8isjr6SMwBitcoin wallet for ransom payments
Botnet12.34.56.7C2 server IP address
Webshell<?php system($_GET['cmd']); ?>Common PHP backdoor pattern

These strings help you detect malicious files even if their hash changes which is exactly what YARA is built for.

Matching by File Size and Content

YARA lets you combine string matches with file attributes like size, type, or number of string occurrences. Here's an example that only matches small files:

java
rule hello_small_files {
  strings:
    $s = "Hello World!"

  condition:
    $s and filesize < 10KB
}

Why this matters:

You might only want to flag small scripts or droppers that match your string — this avoids scanning huge binaries unnecessarily.

YARA Condition Logic / Operators That Matter

You can use Boolean, relational, and even bitwise logic inside your condition block.

Boolean Logic

OperatorDescription
andAll specified conditions must be true
orAt least one condition must be true
notInverts the logic

Example:

text
condition:
  $s1 and ($s2 or $s3)

This matches if $s1 is present and either $s2 or $s3.

Group Matching Keywords

Instead of listing each string individually, you can reference all of them or a subset:

KeywordMeaning
all of themAll strings must match
any of themAt least one must match
none of themRule matches only if no strings match
1 of themExactly one string must match
2 of ($group*)2 strings from the group must match

Using String Occurrence Counts

You can also check how many times a string appears using #:

text
condition:
  #s1 >= 5

This matches if $s1 appears 5 or more times in the scanned file.

File Attributes

You can reference metadata like file size, PE header data, and entry points.

Examples:

java
condition:
  filesize < 500KB

condition:
  $a1 and pe.entry_point == 0x1000

This is useful when narrowing down detection to specific formats like PE files or memory-resident payloads.

YARA String Types and Modifiers

Different malware requires different string types. YARA supports several ways to define them — each with modifiers that improve match reliability.

1- Text Strings (ASCII)

Basic form, case-sensitive by default.

java
$s1 = "malicious.exe"

Make it case-insensitive:

java
$s1 = "malicious.exe" nocase

2- Wide Strings (UTF-16)

Used in Windows binaries — two-byte encoding.

java
$s1 = "cmd.exe" wide

3- Hexadecimal Strings

For matching binary data, shellcode, or compiled byte patterns.

java
$s1 = { E2 34 B6 C8 A3 FB }

Enhancements:

java
$s1 = { E2 34 ?? C8 A3 FB }       // wildcard byte
$s2 = { E2 34 [2-4] A3 FB }       // jump: 2–4 bytes gap
$s3 = { E2 34 (C5|B5) A3 FB }     // alternatives

4- XOR-Encoded Strings

Catches 1-byte XOR obfuscation — common in malware.

java
$s1 = "http://malicious.site" xor

5- Base64 Strings

For encoded payloads and configs:

java
$s1 = "api_key=1234" base64

6- Regular Expressions

For flexible matching:

java
$s1 = /user=\w+&pass=\w+/ nocase

Mixed Usage Example:

java
strings:
  $a1 = "powershell" xor
  $a2 = { 48 8D ?? ?? ?? ?? }  // hex
  $a3 = /-enc\s+[A-Za-z0-9+/=]+/ base64
  $a4 = "cmd.exe" wide

condition:
  any of ($a*)

This rule handles multiple evasion techniques in a single signature.

Valhalla bridges the gap between open-source tooling and real-world CTI. You don’t need to write everything from scratch — learn by using their rules and adapting them.

Real-World Rule Example: Emotet

Here’s a simplified version of a YARA rule that detects Emotet, a prolific banking trojan and loader used in spam campaigns:

java
rule MAL_Emotet_Loader
{
  meta:
    author = "Nextron"
    description = "Detects Emotet loader binary"

  strings:
    $a1 = "client_hello"
    $a2 = "get_proc_address"
    $a3 = { 6A 40 68 00 30 00 00 }

  condition:
    2 of them and filesize < 300KB
}
  • $a1, $a2: strings seen in unpacked Emotet binaries
  • $a3: shellcode pattern
  • 2 of them: balances precision and flexibility
  • filesize < 300KB: filters out irrelevant files
