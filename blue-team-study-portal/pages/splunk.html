Original Notes

Splunk

Splunk is one of the leading SIEM solutions in the market that provides the ability to collect, analyze and correlate the network and machine logs

Splunk ingests machine data (logs, events, metrics), parses and indexes it, and makes it searchable so analysts can monitor, detect, investigate, and report incidents.

Simple flow:

Forwarder (agent) → Indexer (parse & store) → Search Head (run SPL, build dashboards/alerts) → Outputs/Actions (email, webhook, ticket).

Core components

Splunk Forwarder

A lightweight agent installed on endpoints to collect and forward data to Splunk. It’s low overhead and recommended for production ingestion.

Common sources:

  • Web servers (HTTP logs)
  • Windows machines (Event Logs, PowerShell, Sysmon)
  • Linux hosts (syslog, application logs)
  • Databases (connection logs, errors)

Indexer

Parses incoming data (timestamps, line breaking), normalizes into field-value pairs, and stores events in indexes (hot → warm → cold → frozen). Indexers also answer search requests.

Search Head

The UI and search orchestration layer where analysts run SPL, save searches/reports, create dashboards and alerts. In distributed setups the SH distributes searches to indexers (or uses an SHC).

Creating dashboards

Dashboards = collection of panels (each panel = a saved report or an inline search).

workflow

  1. Search & Reporting → Dashboards → Create Dashboard.
  2. Fill Title / ID / Description; set Permissions (Private until tested).
  3. Open dashboard → Edit → Add Panel → New from Search (paste SPL) or New from Report.
  4. Choose visualization (line, column, pie, table, single value, map).
  5. Add inputs (tokens) for interactivity (dropdowns, timepicker).
  6. Save; optionally set as Home Dashboard.

External image reference: A screenshot of the Create New Dashboard option in Splunk

Panel best practices

  • One question per panel.
  • Precompute expensive queries (joins/transactions) with scheduled searches; store results in summary index or lookup.
  • Limit default time range (e.g., 24h) to keep panels snappy.
  • Use meaningful titles and descriptions.

Suggested SOC dashboard panels

  • Total open alerts / incidents (table or KPI)
  • Alerts opened vs closed (24h) — efficiency gauge
  • Failed login trends (timechart / heatmap)
  • Top external destinations per host (table)
  • New / rare processes (table)
  • Event volume per collector (single value / timechart)
  • Attack map (geo visualization for source IPs)

Creating alerts

Alerts = saved search + schedule (or real-time) + trigger conditions + actions.

High-level process

  1. Search Query — define what to detect (signatures or behavior).
  2. Search Timing — scheduled (recommended) or real-time (resource-heavy).
  3. Alert Trigger — threshold, number of results, custom condition. Use time windows.
  4. Alert Action — index the alert, add to Triggered Alerts, send email, webhook, run script, or create ticket.

External image reference: A screenshot of the Save As Alert menu with the details filled in

Steps to create an alert

  1. Build and test the search in Search & Reporting.
  2. Click Save As → Alert.
  3. Fill metadata: Title, Description, Permissions (Private / Shared in App).
  4. Choose Alert Type: Scheduled (every N minutes/hours) or Real-time (use sparingly).
  5. Set Trigger Conditions: number of results or custom condition (where count > X).
  6. Configure Trigger Window (e.g., earliest=-10m latest=now()).
  7. Configure Throttle: e.g., suppress alerts for the same user for 60 minutes.
  8. Choose Trigger Actions: index + webhook/ticketing (recommended SOC flow); email for urgent contacts.
  9. Save and test with historical or synthetic data.

Example alerts

Failed-login threshold user >5 failed logins within 5 minutes:

python
index=Omar sourcetype=WinEventLog:Security EventCode=4625 earliest=-5m
| stats count by Account_Name, src_ip
| where count > 5

Schedule: every 5 minutes. Throttle: 60 min per Account_Name.

Port scan detection many destination ports probed by same IP:

python
index=firewall OR index=Omar
| bin _time span=1m
| stats dc(dest_port) as ports by src_ip, _time
| where ports > 20

Schedule: every 1–5 minutes.

Data onboarding

When adding data (Upload / Forward / API / Syslog / DB), Splunk processes it in stages:

  1. Select source — file, network, API, or host.
  2. Choose sourcetype — critical; drives parsing & field extraction.
  3. Input settings — select index, define host, set source metadata.
  4. Review parsing — confirm timestamp extraction, line breaking, event boundaries.
  5. Done & monitor — verify ingestion (check _internal, index volume, index=_internal sourcetype=splunkd).
pick an accurate sourcetype early / it prevents parsing issues later.

Injecting data

Universal Forwarder (production / recommended)

Ask Ops to install UF and provide:

  • inputs.conf monitor stanzas or Windows event channels.
  • Target index name (e.g., vpn_logs).
  • Desired sourcetype (e.g., vpn:openvpn, WinEventLog:Security).

Example inputs.conf:

python
[monitor:///var/log/openvpn.log]
disabled = false
index = vpn_logs
sourcetype = vpn:openvpn

Example outputs.conf (forwarder → indexers):

python
[tcpout]
defaultGroup = indexers

[tcpout:indexers]
server = indexer1.example.com:9997

Heavy Forwarder (preprocess / filter / route)

Use when you must parse, drop, or route events before indexing.

Example transforms.conf + props.conf to drop heartbeat events:

python
# transforms.conf
[drop_heartbeat]
REGEX = heartbeat
DEST_KEY = queue
FORMAT = nullQueue

# props.conf
[your_sourcetype]
TRANSFORMS-null = drop_heartbeat

Upload via Splunk Web

Splunk Home → Add Data → Upload → choose file → set sourcetype & index → Review → Submit. Good for one-off files only.

Key ingestion concepts (analyst focus):

  • Index — logical store for events (retention per index).
  • Sourcetype — defines parsing rules; set correctly.
  • _time — event timestamp; always validate during investigations.
