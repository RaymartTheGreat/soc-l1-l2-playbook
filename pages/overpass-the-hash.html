Original Notes

Overpass-the-Hash

Overpass-the-Hash is a credential misuse technique that bridges Pass-the-Hash and Pass-the-Ticket attacks. It allows attackers to leverage an NTLM hash to request a Kerberos Ticket-Granting Ticket (TGT) from the Key Distribution Center (KDC) in an Active Directory environment, effectively converting NTLM credentials into Kerberos credentials without needing the plaintext password.

MITRE ATT&amp;CK ID: T1550.002

1- Understanding the Attack

  • Credential Source:

NTLM hashes obtained from compromised systems (e.g., via memory dumping or tools like Mimikatz).

  • Authentication Mechanism:

Uses the NTLM hash to request a Kerberos TGT from the KDC, leveraging the compatibility between NTLM and Kerberos protocols in Windows environments.

  • Result:

Once a TGT is obtained, the attacker can request Service Tickets (STs) to access network resources without the need for the actual password.

2- Steps of an Overpass-the-Hash Attack

a. Initial Access and Hash Extraction

  • Objective:

Gain access to a target system and extract NTLM hashes.

  • Example Tool:

Mimikatz

Example Command:

``jsx mimikatz # sekurlsa::logonpasswords ``

Output Example:

``jsx Username : admin NTLM : 5d41402abc4b2a76b9719d911017c592 ``

b. Converting NTLM Hash to a Kerberos Ticket

  • Process:

Inject the extracted NTLM hash into the Kerberos authentication process to request a TGT.

  • Tool and Example Command:

Mimikatz

``jsx mimikatz # sekurlsa::pth /user:admin /domain:example.com /ntlm:5d41402abc4b2a76b9719d911017c592 ``

    • This command uses the NTLM hash to request and inject a TGT into the current session.

c. Verifying and Using the Kerberos Ticket

  • Verification:

Check for the injected ticket using:

``jsx mimikatz # kerberos::list ``

  • Utilization:

With the valid TGT, the attacker can now obtain Service Tickets and access domain resources (e.g., network shares, databases) to move laterally or escalate privileges.

3- Key Tools

  1. Mimikatz
    • Usage:

Dumping NTLM hashes and performing the Overpass-the-Hash injection.

    • Example Command:

``` sekurlsa::pth /user:<username> /domain:<domain> /ntlm:<hash>

```

  1. Impacket Toolkit
    • Usage:

Tools like psexec.py and wmiexec.py can utilize NTLM hashes for lateral movement.

  1. PowerShell Empire
    • Usage:

Provides modules to automate Overpass-the-Hash attacks.

4- Example Scenario

Target Environment

  • Setup:

An Active Directory domain using both NTLM and Kerberos for authentication.

Attack Flow

  1. Compromise a Machine:
    • Use phishing or exploit vulnerabilities to gain access and dump NTLM hashes from LSASS.
  2. Perform Overpass-the-Hash:
    • Inject the NTLM hash to request a Kerberos TGT using Mimikatz:

``jsx mimikatz # sekurlsa::pth /user:admin /domain:example.com /ntlm:5d41402abc4b2a76b9719d911017c592 ``

  1. Access Network Resources:
    • Verify the TGT and then use it to request Service Tickets, allowing lateral movement and privilege escalation.

5- Detection and Defensive Strategies

Detection

  • Event Sequences:

Overpass-the-Hash often leaves a trail where NTLM authentication (Event ID 4776) is followed immediately by Kerberos TGT requests (4768). This unusual sequence can signal an attacker pivoting from NTLM hashes to Kerberos tickets.

  • Ticket Anomalies:

Pay close attention to Kerberos TGT (4768) and TGS (4769) events that involve privileged accounts. Certain encryption types or ticket flags — especially when inconsistent with normal traffic can indicate forged requests.

Likewise, Kerberos logons (4624) tied to the same account, logon GUID, and IP address appearing repeatedly may point to replayed or forged tickets.

  • Monitoring Tools:

SIEM and EDR platforms can help correlate these events and highlight anomalies. Alerting on unusual combinations of ticket options, encryption types, and account names is especially useful.

  • Example Queries:

These queries illustrate how detection might be implemented in a SIEM:

    • Detect a TGT request for Administrator with unusual encryption and ticket flags:

``jsx event.code: 4768 AND winlog.event_data.TargetUserName: "Administrator" AND winlog.event_data.TicketEncryptionType: "0x17" AND winlog.event_data.TicketOptions: "0x40800010" ``

    • Spot a suspicious TGS request tied to a forged TGT from a specific IP:

``jsx event.code: 4769 AND winlog.event_data.TargetUserName: "Administrator@CORP.LOCAL" AND winlog.event_data.IpAddress: "::ffff:192.168.0.28" AND winlog.event_data.TicketEncryptionType: "0x12" ``

    • Identify Kerberos logons using replayed/forged tickets:

``jsx event.code: 4624 AND winlog.event_data.LogonGuid: "{e11813f7-f0e8-1275-8ee8-725ffd1c69cc}" AND winlog.event_data.TargetUserName: "Administrator" AND winlog.event_data.IpAddress: "192.168.0.28" ``

Defensive Strategies

  • Secure LSASS:

Enable Credential Guard or LSASS Protection to prevent attackers from extracting hashes.

``powershell reg add HKLM\SYSTEM\CurrentControlSet\Control\Lsa /v RunAsPPL /t REG_DWORD /d 1 /f ``

  • Restrict NTLM Usage:

Use Group Policy to disable or restrict NTLM (e.g., Network Security: Restrict NTLM).

  • Strengthen Authentication:

Enforce multi-factor authentication (MFA) and rotate privileged account credentials frequently.

  • Monitor Authentication Events:

Configure SIEM rules to raise alerts when NTLM and Kerberos activity appears in suspicious combinations.

6- Summary

  • Protocol Targeted: Kerberos (leveraging NTLM credentials)
  • Authentication Material: NTLM hash converted into a Kerberos Ticket-Granting Ticket (TGT)
  • Scope of Attack: Credential misuse enabling lateral movement and privilege escalation within an Active Directory environment
  • Key Tools: Mimikatz, Impacket Toolkit, PowerShell Empire
