Original Notes

Insecure Direct Object Reference (IDOR)

IDOR is a type of Broken Access Control vulnerability. It happens when an application exposes a reference (like id=123) to an internal object without properly checking if the requester is authorized to access it.

This allows attackers to view, modify, or delete data that belongs to other users.

OWASP 2021 lists Broken Access Control (including IDOR) as the #1 web vulnerability.

How IDOR Works

Unlike SQLi or Command Injection, IDOR is not about input sanitization. Instead, it’s about missing or weak authorization checks.

Example:

text
https://example.com/get_user_information?id=1
  • If the logged-in user’s ID = 1, the request works normally.
  • If the attacker changes it to id=2, and the app does not verify ownership, they now access another user’s data.

This can apply to:

  • User profiles (user_id)
  • Documents (doc_id)
  • Payments (transaction_id)
  • API endpoints with predictable IDs

Attacker Goals

Depending on what object is exposed, attackers may:

  • Steal personal information (names, emails, payment data)
  • Access restricted documents (contracts, invoices, medical records)
  • Perform unauthorized actions (update/delete another user’s content)

Automated tools (e.g., Wfuzz, Burp Intruder) are often used to brute-force sequential IDs.

Detection

Where to look:

  • Web server / app logs → suspicious parameter manipulation.
  • API logs → unusual sequences of object IDs.
  • WAF logs → blocked enumeration attempts.

What to look for:

  • Repeated requests to the same endpoint with incrementing/decrementing IDs (id=1, id=2, id=3, …).
  • High-volume requests from the same source in a short time.
  • Tool fingerprints in headers or user-agents (e.g., wfuzz, intruder).
  • Abnormal response size differences → larger/smaller content depending on object retrieved.
  • HTTP 302 redirects vs 200 OK → may indicate failed vs successful attempts.

Example

Access logs show repeated requests to:

text
/wp-admin/user-edit.php?user_id=1
/wp-admin/user-edit.php?user_id=2
/wp-admin/user-edit.php?user_id=3
...

Indicators:

  • Tool: User-Agent = wfuzz/3.1.0 (automated scanning).
  • Volume: 15+ requests in seconds.
  • Cloudflare IP: attacker traffic routed via CDN.
  • Response codes: mostly 302, not 200. Suggests unsuccessful attempts.
  • Response size analysis: same size across multiple requests → likely no sensitive data returned.

Prevention

  • Enforce authorization checks on every object request.
  • Use session data (e.g., user ID from session) instead of trusting client-supplied parameters.
  • Avoid predictable identifiers (use UUIDs instead of sequential numbers).
  • Log and alert on suspicious enumeration patterns.
