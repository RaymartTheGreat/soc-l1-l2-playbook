Notes/Tips

  • DLL side-loading: an attacker places a malicious DLL in the same directory as a trusted executable so when that executable runs it loads the attacker’s DLL instead of the real one. Windows searches for required DLLs starting in the executable’s current directory before consulting system folders; a signed or legitimate EXE copied into Temp (or a similarly writable folder) will happily load any DLL found there. In this case the attacker dropped a genuineMsMpEng.exeinto Temp and paired it withMpsVc.dllso the trusted EXE would load the malicious DLL and execute the payload without needing to tamper with system binaries.
  • lsass.exe is Local Security Authority Subsystem Service in Windows which Handles authentication, password changes, access tokens, and Active Directory logins its location should be C:\Windows\System32\lsass.exe attackers often use a similarly named file in Downloads to blend in

Ransomware Attack (LetsDefend)

https://app.letsdefend.io/challenge/ransomware-attack

Before start answering questions we can check system information we see its a Windows 7 Home Desktop, not domain joined, 32-bit OS, and the username is “charles”

1- Please you find the dropped dll, include the whole path including the dll file

C:\users\charles\appdata\local\temp\mpsvc.dll

I noticed MsMpEng.exe running from the Temp folder as a child of cmd.exe — that’s immediately suspicious because the real Defender service runs from C:\Program Files\Windows Defender\MsMpEng.exe. I checked the MsMpEng.exe file’s hash and it matched the legitimate binary

which pointed to DLL side-loading rather than a fake EXE. Knowing side-loading looks for DLLs in the executable’s working directory first, I inspected the Temp folder and found

MpsVc.dllits hash matched malware on VirusTotal, confirming it was the malicious DLL

DLL side-loading: an attacker places a malicious DLL in the same directory as a trusted executable so when that executable runs it loads the attacker’s DLL instead of the real one. Windows searches for required DLLs starting in the executable’s current directory before consulting system folders; a signed or legitimate EXE copied into Temp (or a similarly writable folder) will happily load any DLL found there. In this case the attacker dropped a genuineMsMpEng.exeinto Temp and paired it withMpsVc.dllso the trusted EXE would load the malicious DLL and execute the payload without needing to tamper with system binaries.

2- What is the MD5 hash for the dll?

040818b1b3c9b1bf8245f5bcb4eebbbc

3- What i s the name of ransomware note that got dropped?

2s6lc-readme

Around the DLL drop I inspected nearby files and found a ransom note text file; opening it and reading its strings confirmed it was the ransom note

4- What is the URL that the initial payload was downloaded from? (Include the whole URL with the payload)

http://192.168.75.129:8111/documents/lsass

I looked at events near the DLL drop and saw a lsass.exe entry in the Downloads folder referenced around the same time. I opened the File Download History, searched for lsass, and traced the download to that URL

lsass.exe is Local Security Authority Subsystem Service in Windows which Handles authentication, password changes, access tokens, and Active Directory logins its location should be C:\Windows\System32\lsass.exe attackers often use a similarly named file in Downloads to blend in

5- The ransomware drops the copy of the legitimate application into the Temp folder. Please provide the filename including the extension

msmpeng.exe

As seen in Q1, the attacker placed a copy of the Defender executable in Temp and relied on it to load the malicious MpsVc.dll

6- What is name of the ransomware?

sodinokibi
