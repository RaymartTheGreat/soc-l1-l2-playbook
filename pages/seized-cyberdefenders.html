Seized (CyberDefenders)

Scenario

Using Volatility, utilize your memory analysis skills as a security blue team analyst to investigate the provided Linux memory snapshots and figure out attack details.

Instructions:

  • Use the latest version of Volatility, place the attached Volatility profile "Centos7.3.10.1062.zip" in the following path: volatility/volatility/plugins/overlays/linux.

https://cyberdefenders.org/blueteam-ctf-challenges/seized/

note: we have to put the zip file in this path

1- What is the CentOS version installed on the machine?

7.7.1908

the memory profile is labeled Centos7.3.10.1062 we do a google search and found this

https://en.wikipedia.org/wiki/CentOS#CentOS_version_7

2- There is a command containing a strange message in the bash history. Will you be able to read it?

shkCTF{l3ts_st4rt_th3_1nv3st_75cc55476f3dfe1629ac60}

Dumped the bash history from memory (linux_bash plugin), spotted the base64-encoded flag string, decoded it to reveal the flag

3- What is the PID of the suspicious process?

2854

Inspected running processes and open network sockets; identified the listening ncat process, which indicates a backdoor shell—the PID shown for that was 2854

4- The attacker downloaded a backdoor to gain persistence. What is the hidden message in this backdoor?

shkCTF{th4t_w4s_4_dumb_b4ckd00r_86033c19e3f39315c00dca}

Found the GitHub download command in bash history, navigated to the repository (snapshot.py) to retrieve its payload, and decoded the embedded base64 string to uncover the flag

5- What are the attacker's IP address and the local port on the targeted machine?

192.168.49.1:12345

Checked network connections in memory using netstat

6- What is the first command that the attacker executed?

python -c import pty; pty.spawn("/bin/bash")

When looking at the suspicious ncat reverse shell process, I followed its activity to see what it launched. The first recorded command showed Python being used to spawn a proper bash shell, which attackers commonly do to upgrade a simple shell into an interactive one.

7- After changing the user password, we found that the attacker still has access. Can you find out how?

shkCTF{rc.l0c4l_1s_funny_be2472cfaeed467ec9cab5b5a38e5fa0}

Even after the user’s password was changed, the attacker could still log in. By inspecting system modifications, I noticed they had edited /etc/rc.local. Inside it, a command was placed to add their SSH public key into authorized_keys. This gives persistent access since SSH key authentication bypasses the need for a password.

8- What is the name of the rootkit that the attacker used?

sysemptyrect

To check for rootkits, I analyzed system calls and kernel modules present in memory. The results showed several hooked system calls, and among them, the rootkit “sysemptyrect” was identified as responsible for hiding malicious activity.

9- The rootkit uses crc65 encryption. What is the key?

1337tibbartibbar

Digging into the loaded kernel module parameters exposed by the rootkit, I found a variable related to its encryption mechanism. The field storing the CRC65 key clearly revealed the attacker’s chosen value: 1337tibbartibbar.
