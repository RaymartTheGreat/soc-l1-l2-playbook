Original Notes

TCP (The Three-Way Handshake)

TCP is a connection‑oriented, reliable transport‑layer protocol. Before any data is exchanged, TCP establishes a session using a three‑way handshake, then guarantees in‑order, error‑free delivery via sequencing, acknowledgments, flow control, and retransmissions. Finally, it cleanly tears down the connection with a four‑step FIN sequence.

Key Characteristics

  • Connection‑oriented: Requires setup (three‑way handshake) before sending data.
  • Reliable delivery: Uses sequence numbers, ACKs, checksums, and retransmissions.
  • Ordered data stream: Segments arrive in order—or are reordered by the receiver.
  • Flow control: Receiver advertises a window size to throttle sender speed.
  • Congestion control: Algorithms (e.g., Slow Start, Congestion Avoidance) adapt to network conditions.

TCP Segment Structure

Each TCP segment consists of a header (20–60 bytes) and payload.

FieldSizeDescription
Source Port16 bitsPort on the sender’s host (chosen from ephemeral range).
Destination Port16 bitsPort where the receiving service is listening (e.g., 80 for HTTP).
Sequence Number32 bitsByte index of the first data byte in this segment.
Acknowledgment Number32 bitsNext byte the sender expects to receive (cumulative ACK).
Data Offset (HLEN)4 bitsHeader length in 4‑byte words (min 5 → 20 bytes; max 15 → 60 bytes).
Reserved3 bitsReserved for future use; set to zero.
Control Flags9 bitsURG, ACK, PSH, RST, SYN, FIN, plus ECE, CWR, NS for congestion control.
Window Size16 bitsBytes the receiver can accept (flow‑control window).
Checksum16 bitsCovers header + data for error detection.
Urgent Pointer16 bitsIf URG flag set, points to last urgent data byte.
Options & Padding0–40 bytesExtra features (MSS, window scaling, timestamps).
Payload (Data)VariableThe actual application data (up to MSS).

Three‑Way Handshake

Establishes a TCP connection and synchronizes initial sequence numbers (ISNs):

StepPacketDescription
1SYNClient → Server: “SYNchronize. My ISN = A.”
2SYN + ACKServer → Client: “ACK your A. My ISN = B, and I SYNc with you.”
3ACKClient → Server: “ACK your B. Let’s begin data transfer at Seq = A+1, Ack = B+1.”

Once Step 3 completes, the TCP connection is established.

Data Transfer & Flow Control

  • Sequencing: Each byte is numbered; segments carry a starting Seq number.
  • Acknowledgments: Receiver sends ACK for the next expected byte.
  • Sliding Window: Sender may transmit up to Window Size bytes without waiting for ACK.
  • Retransmission: If ACK isn’t received before a timeout, the sender retransmits.
  • Congestion Control:
    • Slow Start: Begin with small congestion window (cwnd), double every RTT until threshold.
    • Congestion Avoidance: Increase cwnd linearly after threshold, back off on loss.

Connection Teardown

A four‑segment exchange ensures all data is delivered before closing:

  1. FIN →: Host A says “I’m done sending.”
  2. ACK ←: Host B acknowledges the FIN.
  3. FIN ←: Host B finishes its own data and sends FIN.
  4. ACK →: Host A acknowledges B’s FIN.

After the final ACK, both sides enter TIME_WAIT to catch delayed segments.

Advantages & Disadvantages

AdvantagesDisadvantages
Guarantees data integrity, delivery, and orderingHigher overhead: connection setup/teardown, ACKs, retransmissions, and state tracking
Flow and congestion control adapt to network conditionsIncreased latency and reduced throughput compared to UDP
Widely supported and essential for applications requiring reliable transfers (web, email)Resource‑intensive on servers (each connection consumes memory and socket state)

Understanding the sequence and acknowledgment numbers

Example handshake and number progression

1 - Client -> Server: SYN

python
Client:12345 -> Server:80  SYN  seq=1000  ack=0
  • First packet in the three‑way handshake.
  • Client chooses an initial sequence number (ISN) = 1000.
  • No ACK yet because nothing received from the other side.

2 - Server -> Client: SYN,ACK

python
Server:80 -> Client:12345  SYN,ACK  seq=5000  ack=1001
  • Server replies with its own ISN = 5000.
  • ack=1001 means "I received the client's SYN (seq 1000) and expect next byte at 1001."

3 - Client -> Server: ACK

python
Client:12345 -> Server:80  ACK  seq=1001  ack=5001
  • Client acknowledges server ISN. ack=5001 means client expects server bytes starting at 5001.
  • Handshake complete. Connection established.

4 - Client -> Server: PSH,ACK (GET /login)

python
Client:12345 -> Server:80  PSH,ACK  seq=1001  ack=5001  payload=GET /login
  • Client sends application data. If payload length = 200 bytes, the next sequence the client would use is 1001 + 200 = 1201.
  • Subsequent packets from the client will show seq=1201 to indicate the next byte position.
  • Sequence number (seq): the byte index of the first byte of data carried by this segment. It is how the sender numbers the stream.
  • Acknowledgment number (ack): the next byte the receiver expects to get from the other side. It is cumulative acknowledgement of bytes received.
  • Both numbers advance forward as data is sent and acknowledged.

Why these numbers matter for detection and hijacking

  • TCP endpoints trust that sequence and acknowledgment numbers represent the current place in the data stream.
  • If an attacker can observe (sniff) or predict the current sequence numbers, they can craft a packet with the correct seq and ack values and inject data into the ongoing connection. The receiver will accept that packet as if it came from the legitimate peer.
  • If injected packets advance the server's view of the stream, the legitimate client and server become desynchronized. That often produces RST packets, retransmissions, or connection failures which are visible in packet captures and logs.

Quick example of an injected packet that would be suspicious

  • Legitimate progression: client last sent up to seq=20301.
  • Malicious injected packet: Attacker(spoofed Client) -> Server PSH,ACK seq=900000 payload=...

This seq value is far outside the expected window. Monitoring tools or analysts tracking stream progression will flag packets that fall outside a reasonable window based on previously observed seq/ack values.
