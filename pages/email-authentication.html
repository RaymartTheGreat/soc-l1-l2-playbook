Original Notes

Email Authentication

Attackers often spoof trusted domains to trick users into opening malicious emails. To help detect and prevent this, domain owners can configure SPF, DKIM, and DMARC records in DNS. These work together to verify that messages are actually coming from the domains they claim to represent.

SPF (Sender Policy Framework)

SPF is a DNS TXT record that lists which IPs or mail servers are allowed to send email on behalf of a domain. When an email is received, the recipient's mail server checks the domain’s SPF record to see if the sender’s IP is authorized.

  • Email claims to be from user@microsoft.com → Receiver gets it from 203.0.100.15 → DNS SPF for Microsoft lists only Microsoft IP ranges → If 203.0.100.15 is not in the list, SPF fails.

Example:

bash
nslookup -type=txt google.com | grep -i spf
dig TXT google.com | grep -i spf

SPF Record Breakdown:

text
v=spf1 include:mailgun.org include:protection.outlook.com -all
  • v=spf1 – Specifies this is an SPF record
  • include: – Allows mail from other domains
  • all – Emails from other sources should fail the check

What all Means:

  • all: Hard fail (email is not allowed)
  • ~all: Soft fail (email may be accepted but marked suspicious)
  • +all: Allow all (not recommended)
  • ?all: Neutral (no enforcement)

Important Note:

SPF only checks the envelope sender, not what you see in the “From” field. So an attacker could still register a similar-looking domain (e.g., amaz0n-support.com), configure SPF, and pass the check. That’s why you should always check the domain itself, not just whether SPF passed.

DKIM (DomainKeys Identified Mail)

DKIM helps verify that an email really came from the claimed domain and that it hasn’t been modified in transit.

How It Works:

  1. The sending mail server creates a hash of selected headers and the body.
  2. It signs that hash with the domain’s private key and adds the result as the DKIM-Signature: ... b=... header.
  3. The receiver fetches the public key from the sender’s DNS record (selector._domainkey.domain.com).
  4. The receiver verifies the signature:
    • If the locally computed hash matches the decrypted hash from the public key → DKIM Pass.
    • If it does not match or the signature is invalid → DKIM Fail.

Record Example:

bash
v=DKIM1; k=rsa; p=<public_key>
Even if DKIM is set up correctly, an attacker could still use a lookalike domain or compromise a real mailbox to send signed messages.

  • Doesn’t prove who actually sent the message only that it wasn’t changed after signing.
  • Email appears to be sent from microsoft.com, but if an attacker creates their own domain (for example, malicious.com) and signs emails with their private key, DomainKeys Identified Mail (DKIM) will only confirm that the message was signed by malicious.com not microsoft.com. So although the email shows “From: microsoft.com” DKIM validation proves it actually came from malicious.com.

DMARC (Domain-based Message Authentication, Reporting and Conformance)

DMARC ties SPF and DKIM together. It lets domain owners tell receiving servers what to do when both SPF and DKIM fail and it provides a way to get reports.

Example:

If an email says From: ceo@microsoft.com

  • SPF fails (not from a Microsoft IP)
  • DKIM fails (signature invalid)
  • DMARC policy says reject

→ Receiver rejects the message.

Policy Options:

  • p=none – Do nothing (monitoring only)
  • p=quarantine – Mark the email as spam
  • p=reject – Reject the email completely

DMARC Record Example:

bash
v=DMARC1; p=quarantine; rua=mailto:contact@yourdomain.com
  • v=DMARC1 – Declares a DMARC record
  • p= – Specifies the policy
  • rua= – Email for aggregate reports

Authentication-Results Header

When a receiving mail server processes an email, it may include an Authentication-Results header showing the results of SPF, DKIM, and DMARC checks.

But here’s the catch: passing these checks doesn’t mean the email is safe. It only means the message came from a domain authorized by SPF/DKIM/DMARC. Attackers can:

  • Register a lookalike domain and configure valid SPF, DKIM, and DMARC records
  • Send phishing from services like Gmail or Yahoo (which pass by default)
  • Compromise a legitimate mailbox and use it to send attacks

Conclusion

SPF, DKIM, and DMARC reduce spoofing and phishing risk, but they don’t guarantee an email is safe. These checks verify authenticity, not intent.

Attackers can still:

  • Register lookalike domains with valid records
  • Compromise real mailboxes
  • Use trusted services like Gmail to send phishing

Always check if the domain should be sending the message — not just whether the checks passed.
