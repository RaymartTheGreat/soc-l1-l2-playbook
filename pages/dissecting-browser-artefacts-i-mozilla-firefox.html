Original Notes

Dissecting Browser Artefacts I: Mozilla Firefox

Aside from the applications that are prone to threat actors' activities, browser applications are also a treasure trove of information when it comes to digital forensics. This is because most user-targeted attacks nowadays are being delivered through the web.

This task will delve into browser forensics and how browser artefacts can be correlated to other malicious activities.

Browser Forensics 101

As mentioned, browser forensics is crucial in digital forensics and incident response. It provides valuable insights into a user's online activity, including websites visited, login credentials, download history, bookmarks and extensions or plugins.

First, let's run down a few common artefacts and elaborate on some use cases that would give us quick wins in determining notable actions worthy of investigation.

ArtefactScenarios
Browsing HistorySince this artefact provides the exact site's URL and the time and date of the visit, the user's online activities can be easily plotted, whether related to access to a malicious link or unauthorised internal documents.
DownloadsMalicious files downloaded through the browser can be easily determined through this artefact, as well as where they are stored in the workstation and what link was used to download the file. The file name can also be used and correlated with evidence of execution to determine the extent of the potential compromise.
Login CredentialsIf the user has saved their login data, this artefact allows you to correlate the pair of credentials and where they were submitted. Leveraging the contents of this artefact enables you to determine whether the user has a bad practice of saving credentials, has submitted credentials to a phishing website, or the pair of credentials is being reused.
CookiesCookies might be a minimal piece of data. Yet, they can still be used to determine the user's activity on a specific website, like confirming if the user has successfully authenticated and accessed a resource.
SessionsLeveraging this artefact allows you to combine all the user's activity in a specific timeframe or a single browsing session. By knowing this, it is easy to correlate the sites accessed by the user and determine the pattern of the activity.
ExtensionsCertain threat actors now leverage fake browser extensions and await for users to deploy and compromise users. Examples such as extensions harvesting login data or recording screen activity are prominent nowadays.

In the subsequent tasks, these artefacts will be discussed through forensic analysis on three different browsers: Firefox, Google Chrome and Microsoft Edge.

The artefacts mentioned above can be checked by navigating through the GUI of these browsers. However, we will focus on analysing the metadata files generated by these applications.

Mozilla Firefox

Firefox's browsing data are stored in the user's AppData\Roaming\Mozilla\Firefox\Profiles directory. This contains the Firefox profiles' directories in the following regex format - [a-z0-9]{8}\.default.*. Given this, use the command below to enumerate each user directory and see if any Firefox artefacts are available.

powershell
PS C:\Users\Administrator> ls C:\Users\ | foreach {ls "C:\Users\$_\AppData\Roaming\Mozilla\Firefox\Profiles" 2>$null}

From here, we can start analysing the contents of the discovered Firefox metadata directory. Below is the summary of artefacts worthy of checking for Firefox.

File / Directory NameArtefact ContentsFile Type
places.sqliteContains the browsing history and bookmark metadata.SQLite
logins.json / key4.dbContains credentials saved through the browser.JSON / SQLite
cookies.sqliteContains cookies from sites accessed.SQLITE
extensions.json / extensions directoryContains all artefacts related to Firefox extensions.JSON / Folder
favicons.sqliteFavicon metadata indicating sites accessed.SQLite
sessionstore-backupsHandles sessions and tabs metadata.Folder (jsonlz4 files)
formhistory.sqliteContains input data submitted by the user in web forms.SQLite

Scenario: Phishing Link Access

Since we are talking about browser artefacts, let's use the scenario of a user accessing a potential phishing website. To start with, let's run the SQLite browser and open the places.sqlite file.

Once opened, the initial view rendered is the list of database tables where the metadata is stored. You may click on the Browse Data section and select the moz_places table.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/5309dc33d55de12ebb61071ad860648e.png

From this view, you can start gathering the following information:

ArtefactDescription

| URL and Title | A bit of self-explanatory information. However, the combination of URL and title can be used to determine some unusual mismatches that may lead to the discovery of a phishing website. e.g. evil[.]com uses a title named Login | Microsoft 365 | | Visit Count | Defines how many times the user has accessed the website. | | Last Visit Date | Contains the last time the user accessed the site in epoch format. This means the data in this column needs to be converted to recover the readable timestamp. | | Typed | Defines whether the user accessed the website by typing via the browser's search bar or the site is accessed via a click or redirection. |

To gather more information about the sites visited by the user, we can use the IDs of each moz_places data and correlate it to the other tables. One possible use case is determining all the timestamps (not just the last visit date) when the user accessed the site. This can be done by correlating the places ID in the moz_historyvisits table.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/55c9cc06a43db03829c98ad5e52702a9.png

You may have observed that the table also has a visit_type column. This column defines how the user accessed the site, which is essential to obtain the complete context of the access attempt. As a reference, you may use the table below to determine the visit type:

Visit TypeDescription
1 - TRANSITION_LINKThe user followed a link and got a new top-level window.
2 - TRANSITION_TYPEDThe user typed the page's URL in the URL bar, selected it from the URL bar autocomplete results, and clicked on it from a history query (from the History sidebar, History menu, or history query in the personal toolbar or Places organiser).
3 - TRANSITION_BOOKMARKThe user followed a bookmark to get to the page.
4 - TRANSITION_EMBEDSet when some inner content is loaded. This is true of all images on a page and the iframe's contents. It is also true of any content in a frame, regardless of whether or not the user clicked something to get there.
5 - TRANSITION_REDIRECT_PERMANENTThe transition was a permanent redirect.
6 - TRANSITION_REDIRECT_TEMPORARYThe transition was a temporary redirect.
7 - TRANSITION_DOWNLOADThe transition is a download.
8 - TRANSITION_FRAMED_LINKThe user followed a link and got a visit in a frame.
9 - TRANSITION_RELOADThe page has been reloaded.

The next interesting table in this file is the moz_annos (Annotations) table, which contains the download metadata. This again requires the correlation of the place_id (from the moz_places table) to determine the source of the download attempt. Moreover, this table also contains the location where the downloaded file is being stored.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/58874497c31a15e256b6e6ff16321d44.png

Moving on to the following Firefox metadata, let's open the cookies.sqlite file. This SQLite file contains all the cookies and the associated site stored in the moz_cookies table.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/1bb1f94508daa3944c4673a14ce2b048.png

The good thing about checking the cookies is that it can determine if the user has logged in to a specific website. This can be done by checking the known cookies on commonly used websites and finding a cookie issued after successful authentication. Moreover, the cookies can also be used to determine any mismatches between the cookie name/value and the accessed website. This approach applies to checking potential cases of man-in-the-middle attacks like websites hosted using evilginx2.

For a quick example, let's refer to Microsoft's documentation of web browser cookies used in Microsoft Entra authentication (which also applies to O365). According to the documentation, these cookies handle the user's session information:

  • ESTSAUTH - Contains user's session information to facilitate SSO. Transient.
  • ESTSAUTHPERSISTENT - Contains user's session information to facilitate SSO. Persistent.

Given these, we can use them as a reference to see if there are any O365 cookies seen in the entries of another website, which may indicate a potential phishing website leveraging a MITM attack.
