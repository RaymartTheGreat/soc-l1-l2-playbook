Original Notes

Brute Force / Credential Stuffing / Password Spraying

  • Brute force: many password attempts against a single account (or small set).
  • Password spraying: try a small set of common passwords (e.g., Password1, Welcome123) across many accounts to avoid lockouts.
  • Credential stuffing: use leaked username:password lists from past breaches to try those combos at scale.

auth primer

  • Auth result codes: 200 = OK, 401/403 = auth failed/forbidden, 302 often used after successful login redirect.
  • Lockout mechanisms: account lockouts or throttling break naive brute force but can be bypassed by distributed attacks.
  • MFA: multi-factor stops simple credential reuse even if password is correct.
  • Session cookies / tokens are issued after successful auth these are the attacker's goal.

How it works

Classic brute force:

Attacker targets alice@example.com and tries hundreds/thousands of passwords quickly:

text
POST /login
username=alice@example.com&password=Password123

Repeated failures followed by a success (or many failures only).

Password spraying (distributed):

Attacker tries Welcome1 for user1, user2, ... user1000 — one attempt per account to avoid lockout.

Credential stuffing:

Attacker feeds a list of leaked combos:

python
alice:Summer2020
bob:Qwerty!23
...

What attackers can gain

  • Account takeover (user data, payment methods)
  • Privilege escalation (if admin creds found)
  • Fraud (payments, subscriptions)
  • Lateral access (use account to reach internal APIs or escalate)
  • Persistence (create API keys, webhooks, backdoors)

Detection

Where to look

  • Authentication logs (success/failure events, response codes)
  • Web access logs (POST /login, response codes, user-agent, IP)
  • WAF / bot-management logs (rate-limits, CAPTCHAs triggered)
  • IAM / application audit logs (failed logins, lockouts, password reset events)
  • Geo/IP logs (impossible travel, many countries in short time)

High-signal indicators

  • High count of 401/403 or login failures for a single account in short time.
  • Many auth attempts from many distinct IPs for the same username (distributed attack).
  • Many different usernames tried from one IP (spraying/scanning).
  • Same User-Agent or identical request fingerprint used across attempts.
  • Successful logins immediately following many failures (credential stuffing hit).
  • Short inter-arrival times between failed auths (automated tool behavior).
  • Requests accompanied by known automation UAs (python-requests, curl, scanner UAs) or missing browser headers.
  • Multiple accounts showing auth failures hitting the same small password set (spraying).
  • Many failed resets or email-change attempts (account takeover attempts).

SOC checks & triage

  1. Identify affected accounts and check for successful logins/redirections or suspicious post-login actions.
  2. Block or throttle offending IPs temporarily (WAF/firewall) and monitor for distributed sources.
  3. Force password reset / notify affected users if credentials found valid or if large-scale attempts detected.
  4. Check for MFA / prompt for re-enrollment or force MFA for high-risk accounts.
  5. Search logs for similar patterns across other apps/services (credential stuffing often tries many targets).
  6. Check for lateral movement from any account that showed a successful login during the window.
  7. Preserve logs (auth, access, proxy) for IR and for reporting to downstream teams.
  8. Apply account-level mitigations: progressive delays, incremental backoff, CAPTCHAs, blocking common passwords.

Prevention & hardening

  • MFA for all privileged & risky accounts (ideally for all users).
  • Rate-limiting & progressive delays: increase response time after N failures.
  • Block common passwords (denylists, weak-password checks at registration).
  • Use CAPTCHA / bot mitigation on login and password reset endpoints (invisible when possible).
  • Device/browser fingerprinting + risk scoring to detect distributed attacks.
  • Password reuse detection / breached-password checks at login or signup (e.g., check hash against breached list).
  • Lockout policy tuned for usability: temporary lockouts + notify user + out-of-band unlock.
  • Monitoring & alerting: anomalous auth velocity, many failures across many accounts, sudden spike in successful logins.
  • Log everything: IP, UA, origin header, response code, response size, timestamp — correlate quickly.
