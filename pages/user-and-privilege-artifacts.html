User and Privilege Artifacts

/etc/passwd — User Account Information

The /etc/passwd file stores essential information about user accounts on a Linux system. It is a text file where each line represents a user account, with fields separated by colons (:). Despite the name, this file does not contain actual passwords—it contains placeholders and metadata.

Structure Example:

bash
username:x:uid:gid:description:home_directory:login_shell

Key Fields:

  • username: The account name.
  • x: Indicates the password is stored in /etc/shadow.
  • uid: User ID. System users are usually <1000, while normal user accounts start from 1000 upwards.
  • gid: Group ID.
  • description: Usually optional; describes the user.
  • home_directory: Default path for the user's files.
  • login_shell: Default shell executed on login.

Forensic Relevance:

  • Check for hidden or suspicious users (e.g., accounts mimicking system users).
  • Analyze uid values — attacker-created accounts often have uid=0 for root access.
  • Use cat /etc/passwd | column -t -s : for a cleaner view.

/etc/shadow — Encrypted Password Storage

The actual password hashes are stored in the /etc/shadow file, which is only readable by root. Each line corresponds to a user in /etc/passwd.

Why it matters:

  • Contains password hashes used during authentication.
  • Includes fields for password aging, expiration, and account lockout.
  • Attackers may try to extract this file to crack passwords offline using tools like Hashcat or John the Ripper.

Command to view (as root):

bash
sudo cat /etc/shadow

Forensic Use:

  • Dump and analyze password hashes.
  • Check for accounts with no password or locked accounts.
  • Look for recent modifications to this file — might indicate account tampering.

/etc/group — Group Memberships

The /etc/group file contains group definitions and lists of users who are members of each group. Format is colon-separated, similar to /etc/passwd.

Structure:

bash
group_name:password_placeholder:gid:user_list

Why this matters:

  • Group membership affects user privileges.
  • Review for unauthorized users in groups like sudo, adm, wheel, docker, etc.

Command:

bash
cat /etc/group

/etc/sudoers — Privilege Escalation Rules

This file defines which users and groups are permitted to use sudo for privilege escalation. It can only be modified and read by root. Improper changes here can grant unintended root-level access.

Command:

bash
sudo cat /etc/sudoers

Look for:

  • Entries that allow full or passwordless sudo access (ALL=(ALL) NOPASSWD:ALL).
  • Suspicious users or groups listed with elevated permissions.
  • Use of include or includedir directives that pull in other sudo rule files (e.g., /etc/sudoers.d/).
