Deep-diving on Office Applications III: Microsoft OneDrive

The last application to be analysed for this task is OneDrive, a known cloud storage service provided by Microsoft. Given the nature of the application, we will focus on analysing files synced locally and determine any unusual indicators. First, the OneDrive artefacts are stored in the user's Appdata\Local\Microsoft\OneDrive\logs directory. This directory contains additional directories, such as Business* and Personal, which differ only by the subscription of the account used. Microsoft products are licensed in a typical enterprise environment, so the artefacts are commonly seen in the Business folder.

From the Business folder, two important files contain significant information:

  • SyncEngine.odl - Contains all operations performed, including the information of each file processed. The odlgz files are the compressed files of SyncEngine.odl.
  • SyncDiagnostics.log - A log file that contains the diagnostic information related to the synchronisation process of OneDrive. It stores details about sync activities, such as download and upload progress, bytes uploaded or downloaded, and the number of files processed.

You may have observed that the SyncDiagnostics.log is only a text file that can be easily analysed. However, the SyncEngine.odl files require additional parsing to be analysed.

To parse the OneDrive artefacts inside the ODL files, we will use OneDriveExplorer, another open-source tool hosted in GitHub. Note that the tool must be run as administrator since it will try to scan the OneDrive directories of each existing user in the machine.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/cac6414a6b37291dc942f828dff23bcb.png

Next, navigate to File > Live system to start the OneDrive metadata harvesting. As mentioned, the tool will traverse every user's OneDrive AppData directory and load the files that contain significant information.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/bd9954a78eeb9d8e5ffcaa31493f480a.png

Once done, you will see a nested list of files/directories on the left sidebar, the child contents in the middle, and properties of the chosen item at the rightmost view.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/99174234f86540bb3ed4a4cd2f724455.png

You can start traversing the files and directories from here and observe their properties.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/164f26b62b435f3fa5026c9b1e0fffa2.png

Based on the example above, we can take note of the following information:

  • name - Name of the file.
  • lastChange - Timestamp of file's last modification.
  • status - Determines if the files are stored in the cloud or only stored locally.
  • size - Size of the file.

In addition to this, it is also good to check the properties of the main OneDrive folder. Given the example below, it is essential to take note of the associated OneDrive link and the folder's location on the local machine. By checking the associated OneDrive link, you can determine if any unusual OneDrive folders are synced in the local machine. One example use case is for exfiltrating files by syncing and uploading a stolen set of files in a personal OneDrive folder.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/232ee26571b89a479553062e6e6961f0.png

Lastly, the files deleted can also be a good source of information. The tool view contains the list of deleted files, the date of deletion, and the file's location in the machine. Based on this information, any unusual files deleted to hide tracks can be discovered.
