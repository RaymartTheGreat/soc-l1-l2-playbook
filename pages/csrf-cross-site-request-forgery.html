Original Notes

CSRF (Cross-Site Request Forgery)

CSRF is when an attacker makes a victim’s browser send an authenticated request (with the victim’s cookies/session) to a target site — causing the victim to perform actions they didn’t intend (change password, transfer funds, etc.). The attacker does not need the victim’s credentials; the browser automatically attaches session cookies.

How it works

Victim is logged in to bank.example. Attacker hosts a page with an auto-submitting form:

html
<form action="https://bank.example/transfer" method="POST" id="f">
  <input name="to" value="attacker-acc">
  <input name="amount" value="1000">
</form>
<script>document.getElementById('f').submit()</script>

Victim visits attacker page → browser sends POST /transfer to bank.example with bank session cookie → transfer happens without victim interaction on the bank site.

What attackers can gain

  • Transfer funds / make payments
  • Change user email, password, phone
  • Add/remove admin privileges or change settings
  • Trigger destructive actions (delete data)
  • Plant persistent changes (replace webhook, change payout account)

Detection

Where to look

  • Web server / app logs (POST to state-changing endpoints)
  • App audit logs (account changes, password resets)
  • WAF logs (blocked cross-site forms)
  • Browser console / front-end monitoring if available

Log-level indicators

  • POST/PUT/DELETE requests with a Referer or Origin from an external domain (not your domain).
  • State-changing requests missing CSRF token or with token validation failures (if your app logs token checks).
  • Multiple identical state-change requests from many different referers (attacker testing).
  • Unexpected successful actions from a user session without a recent login (session used by CSRF).

Example suspicious HTTP request

python
POST /account/change-email HTTP/1.1
Host: bank.example
Cookie: session=abcd1234
Referer: https://attacker.example/
Content-Type: application/x-www-form-urlencoded

email=attacker%40evil.com

Why suspicious:

  • Referer is external and the request is state-changing.
  • Browser sent valid session cookie — attacker didn’t need it.

Prevention / hardening

  • Anti-CSRF tokens (synchronizer pattern): include a per-session random token in forms and validate server-side on submit.

  • SameSite cookies: set SameSite=Lax or Strict for session cookies where possible (prevents many cross-site form submissions).

SameSite cookies : are a browser security feature that control whether cookies are sent along with cross-site requests. They help protect against CSRF (Cross-Site Request Forgery) and some tracking issues.

  • Origin / Referer checks: validate Origin (preferred) or Referer header on state-changing requests — reject if not your origin.
  • Double-submit cookie: if synchronizer not possible, use a token cookie + token parameter and compare them.
  • Require re-auth for sensitive actions: ask user to re-enter password or confirm MFA for transfers or account changes.
  • Use CORS correctly: do not rely on CORS to prevent CSRF for browser form submissions — CORS protects cross-origin XHR, but forms still submit.

CORS: It’s a security mechanism implemented by web browsers that controls whether a web page can request resources (like APIs, images, fonts, etc.) from a domain different from the one that served the page.

  • Limit cookie lifetime & use secure flags: HttpOnly, Secure, and reasonable expiry.
