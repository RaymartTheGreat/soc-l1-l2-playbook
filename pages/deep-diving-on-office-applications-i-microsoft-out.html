Original Notes

Deep-diving on Office Applications I: Microsoft Outlook

Aside from web browsers, more typical applications essential to delivering employees' day-to-day tasks are installed in enterprise workstations. Some examples would be messaging apps, email clients and file storage solutions.

While discussing these examples, Microsoft applications immediately pop into our minds. Most organisations nowadays rely on their products, such as Teams, Outlook, and OneDrive.

In this task, we will focus on potential forensic artefacts seen in these Microsoft products and analyse their contents to determine any unusual or malicious activity.

Similar to the browser forensics task, forensic artefacts can be checked by navigating through the GUI of these Microsoft products. However, we will focus on discussing and analysing the metadata files generated by these applications. Moreover, we will not discuss every metadata file; we will only focus on the important ones, depending on the applicable use case.

Microsoft Outlook Analysis

The first application to be analysed for this task is Microsoft Outlook, a commonly used email client across organisations that use Microsoft 365. In this application, we will investigate based on the email artefacts generated by the application.

To get started, let's recursively search for each user's directory and look for the AppData\Local\Microsoft\Outlook\ directory.

Administrator: WindowsPowerShell

powershell
PS C:\Users\Administrator> ls C:\Users\ | foreach {ls "C:\Users\$_\AppData\Local\Microsoft\Outlook\" 2>$null | findstr Directory}

In the discovered directory, you will see an OST file (.ost extension) in the following naming convention - user_email@domain.com.ost. This file serves as the local copy of the emails stored, allowing users to access the emails offline. Once connected to the internet, the Outlook client will synchronise to the Microsoft server to update this record. Given this, we can assume that the contents of the OST file are the latest set of emails of the account after its last synchronisation attempt.

To open the OST file, we will use a tool named XstReader. Run the tool using the shortcut from the Desktop directory, navigate to the location mentioned above and open the OST file to be investigated.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/d415c1276aab1beab9a43c52962b2e18.png

Once the OST file is loaded, you will see that the left sidebar contains the email data in tree format. Using the sidebar, navigate to Root - Mailbox > IPM_SUBTREE and start checking the noteworthy email folders, such as Inbox, Sent Items, Deleted Items, and Junk Email.

Scenario: Phishing Emails

From the previous tasks, we have determined some unusual activity from the machine and links accessed through the browser. Given this, let's expand the investigation further by building our methodology for investigating mailbox contents:

Email with file attachments

One of the most common attack vectors threat actors use to compromise their targets is phishing with attachments. Given this, we can start checking the emails with attachments by checking the paperclip symbol in the first column.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/fcccfec0235f84c08e654ca52f04c610.png

Next, you can click one of the emails with an attachment to view its contents on the right side of the application.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/14e7d5b1e05b3f5aa9919a70f21f5291.png

From here, you can start investigating the attachments by acquiring a copy. We won't be touching the file analysis part for brevity purposes. However, here are some pointers to determine if the attachment is malicious.

Nowadays, threat actors deliver varying attachments, such as encrypted archives (.zip, .7z), documents (.doc, .xls), and HTML files (.html, .htm), to compromise their targets. Focus on checking for potential code execution (PowerShell scripts/executable binaries) and credential harvesting indicators (phishing website attachment) since it is typically the goal of these malicious actors.

Lastly, it is also possible to determine if the user has opened the file directly using the Outlook client. To see this, navigate to the same user's AppData\Local\Microsoft\Windows\INetCache\Content.Outlook directory.

powershell
PS C:\Users\Administrator> ls -rec C:\Users\jane.adams\AppData\Local\Microsoft\Windows\INetCache\Content.Outlook

    Directory: C:\Users\jane.adams\AppData\Local\Microsoft\Windows\INetCache\Content.Outlook

Mode                LastWriteTime         Length Name
----                -------------         ------ ----
d-----        2/29/2024   6:59 PM                3PMB1RPU

    Directory: C:\Users\jane.adams\AppData\Local\Microsoft\Windows\INetCache\Content.Outlook\3PMB1RPU

Mode                LastWriteTime         Length Name
----                -------------         ------ ----
-a----        2/29/2024   6:59 PM           6271 survey_results_updated (002).zip
--r---        2/29/2024   6:59 PM           6271 survey_results_updated.zip

This folder serves as the temporary cache of attachments opened using the client. Hence, the file under investigation should appear here if the user has opened it directly.

NOTE: The contents of this directory will disappear once the Outlook client is terminated since it only stores the files as a cache for the current process.

Email with links

Aside from attachments, it is also good to review emails that contain URLs. This is because most email gateways nowadays are now very restrictive of file attachment extensions, and because of this, threat actors tend to deliver malicious content through phishing links.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/e0f2fff8d54e7a216cb8b0a8d0646625.png

Aside from seeing the URL directly from the view, it is important also to determine if the text matches the hyperlink. In some cases, threat actors masquerade the link by varying the text to which the hyperlink is attached.

Once you determine a malicious link, the next thing to do is to correlate the information from the browser artefacts. By doing this, you can confirm if the user has clicked the potential phishing link and determine whether the attacker successfully compromised the victim user.

Reviewing email headers

Lastly, it is also essential to review the headers of the email. This can be done by clicking the Properties radio button on the lower right corner of the application.

External image reference: https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/52fc572a38acc36d5605b4a9e4efea55.png

With the current view, you may review the email headers and note the underlying details to complete the analysis. This includes checking the notable headers, such as Date, From, Received (path from sender to recipient), and X-Mailer (potentially indicates the email client used).
