Original Notes

Threat Hunting: Pivoting

This section focuses on hunting malicious activity that indicates potential internal network pivoting after an initial foothold has been achieved. It covers the attacker's mindset when moving laterally within a network, correlating subsequent actions, and distinguishing benign from suspicious internal reconnaissance activities.

1- Discovery (TA0007)

Definition:

Techniques adversaries use to understand the compromised internal network, identify sensitive systems, and gather information to facilitate further lateral movement.

Examples of Methods:

  • User Reconnaissance:

    • Using built-in commands like whoami, net user, net localgroup, or qwinsta for account and privilege enumeration; and dir or ls for file and folder listing.

  • Host Reconnaissance:

    • Commands such as hostname, wmic, ipconfig, or systeminfo to gather host information; using net start or sc.exe query for service enumeration; or GUI-based tools (e.g., Windows Security) to check running security controls.

  • Internal Scanning:

    • Executing commands like arp to display the ARP table, performing ping sweeps for asset discovery, or scanning open ports with tools such as Nmap or PowerShell’s built-in capabilities.

  • Internal Domain Reconnaissance:

    • Utilizing commands such as net * /domain, nltest /dclist, or PowerShell scripts (e.g., PowerView, BloodHound) to enumerate domain users, groups, and access control lists.

Hunting Discovery

The hunt for the Discovery tactic involves detecting unusual information-gathering activities that blend in with legitimate administrative commands. Below are key scenarios for hunting discovery activities:

Scenario 1: Host Reconnaissance Activity

  • Data Source:

winlogbeat- index in Kibana.

  • Methodology:
    1. **Identify Use of Enumeration Tools:

      2. Monitor process creation events (Sysmon Event ID 1) for tools used in host reconnaissance:**

      • Command-line Tools: whoami.exe, hostname.exe, net.exe, systeminfo.exe, ipconfig.exe, netstat.exe, tasklist.exe.
      • KQL Query:

``jsx winlog.event_id: 1 AND process.name: (whoami.exe OR hostname.exe OR net.exe OR systeminfo.exe OR ipconfig.exe OR netstat.exe OR tasklist.exe) ``

    1. Key Fields to Display:
      • host.name
      • user.name
      • process.parent.command_line
      • process.command_line
    2. Sequence Analysis:

      3. Sort the results in ascending order by timestamp to view the sequence of enumeration commands.

  • Findings:

For instance, user bill.hawkins executed multiple enumeration tools on WKSTN-2, all spawned by cmd.exe. Further investigation into the parent process ID of the first whoami /priv command revealed that the cmd.exe instance was initiated by PowerShell.exe with suspicious indicators (e.g., usage of IEX, DownloadString, -WindowStyle hidden, and -ep bypass).

  • Next Steps:

Use the parent process ID (e.g., 6324) to query for additional events associated with the suspicious cmd.exe process:

``jsx host.name: WKSTN-2* AND winlog.event_id: 1 AND process.pid: 6324 ``

Scenario 2: Internal Network Scanning

  • Data Source:

packetbeat- index for network connections and winlogbeat- index for process correlation.

  • Methodology:
    1. Visualize Internal Network Connections:

      2. Create a Lens visualization table with:

      • Rows: host.name, source.ip, destination.ip
      • Metric: Unique count of destination.port
      • KQL Query:

``jsx source.ip: 10.0.0.0/8 AND destination.ip: 10.0.0.0/8 AND destination.port < 1024 ``

    1. Correlate Network Activity with Processes:

      2. In the Discover tab on winlogbeat-, identify the process that initiated the network connections:

``jsx winlog.event_id: 3 AND source.ip: 10.10.184.105 AND destination.ip: 10.10.122.219 AND destination.port < 1024 ``

      • Key Fields:

host.name, user.name, process.name, destination.port

  • Findings:

The investigation reveals that the process n.exe initiated port scanning from WKSTN-2, targeting INTSRV01. This is suspicious since the connections were established by a user account rather than a system or service account.

  • Next Steps:

Further analyze events associated with n.exe and its parent process to backtrack the origin of the scanning activity and assess potential pivot points.

Scenario 3: Active Directory Enumeration

  • Data Source:

winlogbeat- index in Kibana.

  • Methodology:
    1. Monitor LDAP Connections:

      2. Hunt for LDAP queries using Sysmon Event ID 3 (Network Connection) for connections on port 389 (LDAP) and 636 (LDAP over SSL). Exclude benign processes like mmc.exe:

``jsx winlog.event_id: 3 AND source.ip: 10.0.0.0/8 AND destination.ip: 10.0.0.0/8 AND destination.port: (389 OR 636) AND NOT process.name: mmc.exe ``

      • Key Fields:

host.name, user.name, process.name, destination.port

    1. Identify Suspicious Processes:

      2. This query may reveal processes such as SharpHound.exe, a known tool for Active Directory reconnaissance.

  1. Findings:

The presence of SharpHound.exe strongly indicates domain enumeration activity. Its usage suggests that the attacker is gathering detailed AD information for lateral movement.

  • Next Steps:

Investigate subsequent actions by SharpHound.exe and its parent process to understand the scope of enumeration and potential pivoting into other systems.

2- Privilege Escalation (TA0004)

Definition:

Techniques that allow an attacker to elevate their privileges on a system or network—from a low-privileged user to an account with extensive permissions (e.g., SYSTEM). This escalation paves the way for executing further attack phases such as lateral movement and data exfiltration.

Examples of Methods:

Hunting Privilege Escalation

Hunting for privilege escalation involves detecting unusual events where low-privileged processes lead to actions executed under highly privileged contexts. Below are key scenarios to uncover such activities.

Scenario 1: Abusing SeImpersonatePrivilege

  • Data Source:

winlogbeat- index in Kibana.

  • Methodology:
    1. Hunt for SYSTEM Account Processes with Unusual Parent Users:

Identify process creation events (Sysmon Event ID 1) where a process is running as the SYSTEM account but is spawned by a non-privileged parent.

      • KQL Query:

``jsx winlog.event_id: 1 AND user.name: SYSTEM AND NOT winlog.event_data.ParentUser: "NT AUTHORITY\SYSTEM" ``

    1. Key Fields to Display:
      • host.name
      • user.name
      • process.parent.command_line
      • process.command_line
      • winlog.event_data.ParentUser
    2. Findings:

One notable entry shows a process spawned by the IIS APPPOOL\DefaultAppPool user executing a binary named spoofer.exe, which subsequently leads to a process running as SYSTEM.

    1. Deep Dive into the Suspicious Binary:
      • Query for events related to spoofer.exe:

``jsx winlog.event_id: 1 AND process.name: spoofer.exe ``

      • Investigation:

Using the binary’s hash in VirusTotal reveals that spoofer.exe is attributed to PrintSpoofer, a known tool that abuses the SeImpersonatePrivilege to escalate privileges.

      • Conclusion:

The DefaultAppPool account, which by default has SeImpersonatePrivilege, was exploited to launch spoofer.exe, leading to a successful privilege escalation.

  • Next Steps:
    • Correlate subsequent SYSTEM-level events to determine further malicious actions.
    • Trace preceding events on the compromised host to understand the sequence leading up to the escalation.

Scenario 2: Excessive Service Permission Abuse

  • Data Source:

winlogbeat- index in Kibana.

  • Methodology:
    1. Hunt for Service Registry Modifications:

Identify changes to service configurations—specifically, modifications to the ImagePath registry key that determines which binary a service executes.

      • KQL Query:

``jsx winlog.event_id: 13 AND registry.path: HKLM\\System\\CurrentControlSet\\Services\\\\ImagePath* ``

    1. Key Fields to Display:
      • host.name
      • process.name
      • registry.path
      • winlog.event_data.Details
    2. Findings:

The query reveals two events where the ImagePath was modified to point to C:\Users\bill.hawkins\Documents\update.exe. These events are notable after excluding standard modifications (e.g., those involving svchost.exe).

    1. Deep Dive into the Modification Event:
      • Use the "View surrounding documents" feature to inspect events before and after the modification of the SNMPTRAP service’s ImagePath.

      • Observation:

Prior to the modification, the sc.exe process executed a command:

``jsx sc.exe config SNMPTRAP binPath= C:\Users\bill.hawkins\Documents\update.exe ``

initiated by the user bill.hawkins (a non-privileged account).

      • Further Analysis:

A subsequent cmd.exe process started the modified SNMPTRAP service. Confirm this with:

``jsx winlog.event_id: 1 AND process.parent.name: services.exe AND process.name: update.exe ``

      • Conclusion:

The SYSTEM account spawned update.exe after the registry modification, confirming that excessive service permission abuse was leveraged for privilege escalation.

  • Next Steps:
    • Trace subsequent actions by the SYSTEM account for further malicious activity.
    • Investigate the chain of events before the service modification to fully understand the attacker's methodology.

3- Credential Access (TA0006)

Definition:

Techniques used by attackers to steal or discover valid usernames, passwords, or password hashes. This tactic enables adversaries to gain further access, escalate privileges, and move laterally by leveraging the compromised credentials.

Examples of Methods:

Hunting Credential Access

Hunting for credential access involves actively searching for indicators of adversaries attempting to dump, harvest, or misuse credentials within a system or network. Below are key scenarios and methodologies used to detect these activities.

Scenario 1: LSASS Credential Dumping

Hunting Mimikatz Execution

  • Data Source:

winlogbeat- index in Kibana.

  • Methodology:
    1. Identify Mimikatz-Related Commands:

Use the Discover tab with a query for command-line strings associated with Mimikatz:

``jsx winlog.event_id: 1 AND process.command_line: (mimikatz OR DumpCreds OR privilege\:\:debug OR sekurlsa\:\:) ``

    1. Key Fields to Display:
      • host.name
      • user.name
      • process.parent.command_line
      • process.name
      • process.command_line
  1. Findings:

The results indicate an attempt to dump LSASS credentials on a host (e.g., WKSTN-1) using the PowerShell version of Mimikatz (Invoke-Mimikatz.ps1) downloaded from GitHub.

  • Next Steps:

Investigate further to see if subsequent authentication events originate from credentials obtained by this method.

Scenario 2: LSASS Process Dumping

  • Data Source:

winlogbeat- index in Kibana.

  • Methodology:
    1. Hunt for LSASS Dump Files:

Look for file creation events related to LSASS dump files:

``jsx winlog.event_id: 11 AND file.path: *lsass.DMP ``

    1. Key Fields to Display:
      • host.name
      • winlog.event_data.User
      • process.name
      • file.path
  1. Findings:

The query shows that the account (e.g., bill.hawkins) dumped the LSASS process by creating an lsass.DMP file in the user's AppData\Local\Temp directory.

  • Next Steps:

Track subsequent events for authentication or lateral movement using the dumped credentials.

Scenario 3: Credential Harvesting via DCSync

  • Data Source:

winlogbeat- index in Kibana.

  • Methodology:

DCSync abuses how domain controllers in an Active Directory network communicate and replicate data. Normally, domain controllers synchronise directory information, including password hashes, via the Directory Replication Service Remote protocol (MS-DRSR). The replication request to a domain controller requires the following privileges:

  • DS-Replication-Get-Changes (1131f6aa-9c07-11d1-f79f-00c04fc2dcd2)
  • DS-Replication-Get-Changes-All (1131f6ad-9c07-11d1-f79f-00c04fc2dcd2)
  • Replicating Directory Changes All (9923a32a-3607-11d2-b9be-0000f87a36b2)
  • Replicating Directory Changes In Filtered Set (89e95b76-444d-4c62-991a-0facbeda640c)
  1. Detect DC Sync Activity:

Use the following KQL query to identify events where domain credentials may have been replicated:

``jsx winlog.event_id: 4662 AND winlog.event_data.AccessMask: 0x100 AND winlog.event_data.Properties: (1131f6aa-9c07-11d1-f79f-00c04fc2dcd2 OR 1131f6ad-9c07-11d1-f79f-00c04fc2dcd2 OR 9923a32a-3607-11d2-b9be-0000f87a36b2 OR 89e95b76-444d-4c62-991a-0facbeda640c) ``

  1. Key Fields to Display:
    • host.name
    • winlog.event_data.SubjectUserName
    • winlog.event_data.AccessMask
    • winlog.event_data.Properties
  2. Findings:

The results indicate that the account backupadm attempted a DCSync attack against the domain controller (e.g., DC01), suggesting that this account might be a Domain or Enterprise Admin.

  • Follow-Up Investigation:
    • Query for process creation events by the suspicious account:

``jsx winlog.event_id: 1 AND user.name: backupadm ``

    • Analyze unusual commands issued by backupadm via cmd.exe, confirming malicious behavior.
  • Next Steps:

Trace the compromise chain to determine how backupadm was obtained—possibly through LSASS dumping—and assess further impact.

Scenario 4: Brute-Forcing Accounts

  • Data Source:

winlogbeat- index in Kibana.

  • Methodology:
    1. Identify Failed Logon Attempts:

Create a visualization using Lens to view failed logon attempts with Event ID 4625:

``jsx winlog.event_id: 4625 ``

    1. Key Fields to Display:
      • host.name
      • user.name
      • source.ip
      • winlog.event_data.LogonType
    2. Detailed Investigation for a Specific Account:

Focus on an account (e.g., jade.burke) with numerous failed attempts:

``jsx winlog.event_id: 4625 AND user.name: jade.burke ``

    1. Verify Successful Authentication:

Check if the account later successfully logged in using Event ID 4624:

``jsx winlog.event_id: 4624 AND user.name: jade.burke AND source.ip: 10.10.184.105 ``

    1. Correlate Process Activity Post-Authentication:

Investigate processes spawned by jade.burke on the target host:

``jsx host.name: WKSTN-1* AND winlog.event_id: 1 AND user.name: jade.burke ``

  • Findings:

The results reveal that jade.burke generated numerous failed logon attempts from a specific source IP and then successfully authenticated to WKSTN-1, where unusual commands were executed.

  • Next Steps:

Determine the impact of the executed commands and trace how the brute-forcing activity was initiated—potentially linking it to prior credential dumping activities.

4- Lateral Movement (TA0008)

Definition:

Techniques that adversaries use to gain remote access to additional systems within a network. This tactic involves exploiting internal remote services, using legitimate administrative tools, and authenticating with valid credentials to pivot from a compromised host to new targets.

Examples of Methods:

Hunting Lateral Movement

Hunting for lateral movement involves correlating remote authentication events and abnormal process activities across internal hosts. Below are key scenarios and methodologies for detecting lateral movement activities.

Scenario 1: Lateral Movement via WMI

  • Data Source:

winlogbeat- index in Kibana.

  • Methodology:
    1. Monitor WMI-Related Process Activity:

Detect remote command executions by searching for events where the parent process is WmiPrvSE.exe:

``jsx winlog.event_id: 1 AND process.parent.name: WmiPrvSE.exe ``

    1. Key Fields to Display:
      • host.name
      • user.name
      • process.parent.command_line
      • process.command_line
    2. Analysis:

The results show that WmiPrvSE.exe has spawned multiple instances of cmd.exe on a host (e.g., WKSTN-1) using the account clifford.miller. The executed command pattern—such as:

``jsx cmd.exe /Q /c 1> \\127.0.0.1\ADMIN$\ 2>&1 ``

—is indicative of Impacket's wmiexec.py, a known lateral movement tool.

  • Investigation:

Use the "View surrounding documents" feature on the first WmiPrvSE.exe event to identify the closest authentication event for clifford.miller:

    • Filters to apply:
      • host.name: WKSTN-1.threathunting.thm
      • user.name: clifford.miller
      • winlog.event_id: 4624

Findings: The authentication event reveals that clifford.miller performed a Network Logon (Logon Type 3) from IP 10.10.184.105. This suggests that the source host (10.10.184.105) might be compromised, providing the pivot point for the lateral movement.

  • Next Steps:

Investigate the child processes spawned by WmiPrvSE.exe and trace back events on the source IP to determine how the initial compromise occurred.

Scenario 2: Authentication via Pass-the-Hash

  • Data Source:

winlogbeat- index in Kibana.

  • Methodology:
    1. Identify Potential Pass-the-Hash (PtH) Events:

Look for network logon events that indicate the usage of password hashes. Use the following indicators:

      • Event ID: 4624
      • Logon Type: 3 (Network)
      • LogonProcessName: containing NtLmSsp
      • KeyLength: 0 (indicating hash usage)

KQL Query Example:

``jsx winlog.event_id: 4624 AND winlog.event_data.LogonType: 3 AND winlog.event_data.LogonProcessName: NtLmSsp AND winlog.event_data.KeyLength: 0 ``

    1. Key Fields to Display:
      • host.name
      • user.name
    2. Analysis:

The query results show that two users (excluding known false positives such as ANONYMOUS LOGON) have authenticated via this method. Notably, clifford.miller appears again, with authentication events originating from the compromised host 10.10.184.105.

  • Investigation:

Use the "View surrounding documents" feature for clifford.miller's first Pass-the-Hash event to examine subsequent process creation events:

    • Filters to apply:
      • host.name: WKSTN-1.threathunting.thm
      • user.name: clifford.miller
      • winlog.event_id: 1

Findings: Multiple cmd.exe executions following the PtH event correlate with previous WMIExec activities. This reinforces the suspicion that clifford.miller leveraged Pass-the-Hash to remotely execute commands, furthering lateral movement.

Next Steps:

Determine the impact of the executed commands and investigate how the source host (10.10.184.105) was initially compromised to enable this activity.
