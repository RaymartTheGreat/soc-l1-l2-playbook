Original Notes

EDR (Endpoint Detection and Response)

Endpoint Detection & Response (EDR) instruments endpoints (workstations, servers, VMs, cloud workloads, IoT) with a sensor/agent that collects granular telemetry and forwards it to a central console where correlation, ML and rules generate alerts. The EDR also provides response controls (isolate, kill, quarantine, remote shell, artifact collection). EDR evolved to close gaps left by prevention-only tools (AV/EPP) — especially for fileless attacks, living-off-the-land techniques, and remote work scenarios.

Why EDR matters

  • Devices work off-network more often; perimeter controls alone don’t see everything.
  • EDR gives persistent, device-level visibility and allows containment where network controls can’t reach.
  • It detects stealthy techniques (in-memory payloads, living-off-the-land) that signature-only tools commonly miss.

Telemetry (what to collect and why)

EDR telemetry is the evidence chain for detection and investigation. Keep these fields available and searchable:

  • Time metadata: _time, hostname, host_id, user
  • Process context: process_name, parent_process_name, pid, ppid, cmdline, process_hash
  • File activity: file_path, file_hash (sha256 preferred), file_action (create/write/delete)
  • Registry & persistence: registry_key, registry_value, service_name
  • Network: src_ip, dst_ip, dst_port, dns_query, upstream_domain
  • Memory/behavior: mem_event (injection, suspicious allocation), module_load
  • Event type labels: process_start, file_drop, reg_write, net_conn, mem_inject

Rationale: process + cmdline + network + file/registry form the minimal chain to attribute an activity and build detections.

Core components

  • Sensor (agent): captures process events, command lines, file/registry changes, network connections, and memory indicators; also executes remote commands from the console.
  • Control plane (console): centralizes telemetry, runs detection logic, applies threat intel and ML, and pushes actions to sensors.
  • Storage & search: historical telemetry archive for hunting and reconstruction.
  • Integrations: threat feeds and other tooling for enrichment (optional but common).

Detection approaches

EDR systems typically combine multiple detection methods. Use them together for higher confidence.

  1. IOC matching (known bad)
    • Match file hashes, IPs, and domains against threat feeds. Fast, high-confidence for known threats.
  2. Behavioral logic (pattern-based)
    • Rules that capture suspicious sequences or unusual parent→child relationships. Example pattern: a user-facing app spawning a system installer and then a shell back-connection.
  3. Anomaly detection (baseline deviations)
    • Model typical endpoint behavior (typical processes, hours of activity, common cmdlines). Flag deviations (e.g., file writes to atypical system paths at midnight).
  4. Memory / fileless indicators
    • Detect in-memory payloads, suspicious remote thread creation, or reflective DLL loads that leave little/no disk trace.
  5. Correlation-driven alerts
    • Combine low-signal events (e.g., odd cmdline + rare outbound domain + new autorun) into a single higher-severity detection.
  6. Mapping to frameworks
    • Tag detections with tactic/technique labels (e.g., persistence, execution) to speed understanding — not a detection method itself but useful metadata.

Response capabilities

From the console the operator can instruct sensors to:

  • Network isolate the endpoint (restrict outbound/inbound connectivity).
  • Stop/terminate processes selectively.
  • Quarantine files so they cannot be executed.
  • Open a remote shell to run commands or scripts and gather live artifacts.
  • Collect artifacts centrally: memory snapshots, event logs, copies of suspicious files, registry hives.

These actions are executed by the sensor and recorded as part of telemetry for auditability.

SOC Analyst Workflow with EDR

A SOC analyst typically works through EDR alerts in a structured way. The workflow looks like this:

  1. Alert intake & triage
    • Review new alerts in the EDR console.
    • Prioritize based on severity, MITRE technique, and affected asset criticality.
    • Suppress/close false positives (known admin tools, testing, patching activity).
  2. Initial context gathering
    • Pull endpoint details: hostname, user, IP, logged-in sessions.
    • Review related process tree (parent/child relationships).
    • Check command line arguments, hashes, and network destinations.
  3. Threat validation
    • Compare process, file, or domain against threat intel feeds.
    • Correlate across multiple alerts/events (was this a single event or part of a chain?).
    • Look for persistence mechanisms (registry keys, scheduled tasks, services).
  4. Scoping the impact
    • Search across all endpoints for the same hash, domain, or command line.
    • Identify lateral movement attempts or multiple compromised systems.
    • Pivot on user account activity / is the same user active elsewhere?
  5. Response actions
    • Isolate the endpoint if confirmed malicious activity is ongoing.
    • Kill the malicious process or quarantine the file.
    • Collect memory dump, logs, or suspicious executables for deeper analysis.
  6. Recovery & remediation
    • Work with IT to reimage or patch affected hosts.
    • Reset compromised accounts, rotate credentials if needed.
    • Remove persistence artifacts (scheduled tasks, services, autoruns).
  7. Reporting & lessons learned
    • Document the incident timeline, IOCs, and attacker behavior.
    • Feed new detection rules back into the EDR (custom rules, watchlists).
    • Share findings with the wider SOC to prevent recurrence.

EDR vs XDR

FeatureEDR (Endpoint Detection and Response)XDR (Extended Detection and Response)
ScopePrimarily focuses on protecting endpoint devices like desktops, laptops, and servers.Expands coverage to include email, identities, network traffic, cloud workloads, and other parts of the environment.
Data SourceGathers telemetry and logs only from endpoints.Collects and correlates data from multiple layers and platforms across the organization.
VisibilityOffers visibility limited to endpoint-level activities.Delivers unified visibility across the entire IT ecosystem, from endpoints to the cloud.
DetectionDetects threats based mainly on endpoint indicators and behavioral analysis.Identifies threats through advanced analytics and correlation across multiple domains.
ResponseResponds only to endpoint incidents (for example, isolating a device or stopping a process).Enables automated and coordinated responses across several layers, such as blocking malicious emails or disabling user access.
Use CasesUseful for detecting malware, ransomware, and insider activity on endpoints.Designed for more advanced scenarios like phishing detection, cloud threat correlation, and identity-based or supply chain attacks.
ManagementUsually managed through endpoint-specific security consoles.Offers a single, integrated console for threat detection, investigation, and response across multiple environments.
