Original Notes

Osquery

osquery (open-source, originally developed at Facebook, 2014) exposes an endpoint’s runtime state as queryable tables (a relational view of OS artifacts). You use SQL-like SELECT queries to retrieve structured endpoint data — processes, users, network sockets, installed software, registry/startup entries, files and hashes, etc. It runs on Windows, Linux, macOS and FreeBSD and is commonly deployed at scale via Fleet/Kolide or similar managers.

Quick interaction

  • Start: osqueryi (in-memory shell)
  • Useful meta-commands:
    • .tables [TERM] — list tables (filterable)
    • .schema <table> — see columns/types
    • .mode <pretty|csv|line|column> — change output format
    • .exit / .quit — exit shell

Query pattern:

sql
SELECT <columns> FROM <table> WHERE <filter> LIMIT <n>;

Always inspect .schema for columns you plan to use.

Core tables

  • processes — running processes, cmdline, parent, uid/gid
  • users — user accounts, profile dirs, type
  • programs — installed applications (inventory)
  • process_open_sockets — active network sockets by PID
  • startup_items / registry — startup/persistence artifacts (Windows)
  • file / hash — file metadata and computed hashes

Why SOCs & IR teams use osquery

  • Structured, auditable visibility. System state is returned as rows/columns, making it straightforward to consume, compare and store in a SIEM or investigation timeline.
  • Detection engineering friendly. Reusable queries (packs) map naturally to detection rules and scheduled collection.
  • Fast triage & evidence collection. Low-latency queries let analysts capture process details, command lines, sockets and file hashes during an investigation.
  • Cross-platform parity. Same query concepts across OS families reduces tooling fragmentation.
  • Extensible & automatable. Integrates with orchestration, threat intel enrichment and central management (Fleet, Kolide, etc.).
  • Low footprint, read-oriented by default. Designed for observation rather than modification.

Example queries

PowerShell / suspicious script execution

sql
SELECT pid, name, path, cmdline, parent
FROM processes
WHERE name LIKE '%powershell%' OR cmdline LIKE '%-EncodedCommand%' OR cmdline LIKE '%IEX%';

Flags encoded/inline script use and common PowerShell abuse patterns.

Unusual parent→child relationships

sql
SELECT p.pid, p.name, p.cmdline, p.parent, pp.name AS parent_name
FROM processes p
JOIN processes pp ON pp.pid = p.parent
WHERE pp.name LIKE '%explorer.exe%' AND p.path NOT LIKE 'C:\\Windows\\System32\\%';

Detects processes spawned from common hosts but executing from abnormal locations.

Outbound connections to non-private IPs

sql
SELECT pid, name, remote_address, remote_port
FROM process_open_sockets
WHERE remote_address NOT LIKE '10.%' AND remote_address NOT LIKE '192.168.%' AND remote_address NOT LIKE '172.16.%'
LIMIT 200;

Quick list of processes communicating off-net for IOC enrichment.

Registry Run keys (Windows persistence)

sql
SELECT key, path, name, data
FROM registry
WHERE path LIKE 'HKEY_LOCAL_MACHINE\\Software\\Microsoft\\Windows\\CurrentVersion\\Run%';

Locate programs configured to run at boot (common persistence vector).

File hash check for suspicious locations

sql
SELECT path, md5, sha256 FROM hash WHERE path LIKE 'C:\\Users\\%\\AppData\\Roaming\\%';

Capture file hashes for IOC lookups in threat feeds.

Process → user mapping (investigative)

sql
SELECT p.pid, p.name, p.cmdline, u.uid, u.username
FROM processes p
JOIN users u ON u.uid = p.uid
ORDER BY p.start_time DESC
LIMIT 100;

Determine which user launched a process during triage.
