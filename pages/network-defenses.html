Network Defenses

Network Intrusion Detection

Network Intrusion Detection System (NIDS) is either a piece of software or a dedicated appliance that inspects network traffic and raises alerts when suspicious or malicious behavior is detected. These alerts are passed on to security analysts for investigation.

Depending on how it is deployed, a NIDS can be positioned in different ways:

  • Inline – When the detection system is placed directly in the traffic path, all packets must pass through it. In this configuration, it acts as an Intrusion Prevention System (IPS), allowing it to actively block or reset malicious connections. The downside is that if the device fails, it can disrupt or block all traffic.
  • Network Tap – The system is connected through a dedicated hardware tap that makes a copy of network traffic. This ensures monitoring without interfering with the live traffic flow.
  • Passive/SPAN Port – The NIDS is linked to a switch’s SPAN (mirror) port, which sends a copy of traffic to the detection system. This allows full visibility but does not affect normal traffic flow.

In most cases, NIDS solutions are focused on generating alerts for later investigation. Only when deployed inline do they also prevent threats (at which point they are considered IPS).

Popular Network Intrusion Detection Tools

  • Snort – One of the most widely used open-source intrusion detection systems. It supports a large library of community-driven rules and can be deployed quickly for immediate threat detection. Snort Website
  • Suricata – Another open-source solution that operates at the application layer, providing deep packet inspection and enhanced visibility compared to traditional packet-based systems. Suricata Website
  • Zeek (formerly Bro) – A powerful monitoring framework that goes beyond traditional IDS by also providing network analysis and traffic logging capabilities. Zeek Website

Network Intrusion Prevention

A Network Intrusion Prevention System (NIPS) builds on the capabilities of NIDS but goes a step further by taking automatic defensive actions when threats are detected. Instead of just alerting analysts, it can block or contain malicious activity in real time.

For example, if a compromised machine within the network starts scanning other hosts, a NIPS can be configured to instantly drop its connections, block future traffic from that source, and notify analysts.

NIPS Solutions

Most modern tools like Snort, Suricata, and Zeek can operate in both IDS and IPS modes, depending on how they are configured and deployed.

Firewalls

firewall acts as a security boundary, regulating the flow of network traffic between trusted and untrusted zones. For instance, a firewall placed between a company’s internal network and the internet will only allow approved traffic such as HTTP, HTTPS, or DNS, while blocking everything else. This prevents attackers from freely probing or exploiting internal systems.

Different types of firewalls exist, and they differ mainly in the layer of the OSI model they operate on and the level of traffic inspection they provide.

Packet-Filtering Firewalls (Layer 3 – Network Layer)

  • One of the earliest firewall types.
  • Works by filtering traffic based on IP addresses, ports, and protocols.
  • Rules are straightforward but limited: they cannot inspect the actual content of packets.
  • Pros: Fast and lightweight.
  • Cons: Limited security, as attackers can craft malicious packets that bypass simple filters.

Stateful Inspection Firewalls (Layer 3 & 4 – Network & Transport Layers)

  • Builds on packet filtering by tracking the state of network connections (e.g., TCP handshake).
  • Only allows packets that are part of a valid, established session.
  • Example: It can block unsolicited inbound packets even if they appear valid individually.
  • More secure than simple packet filters, but still not aware of application-level threats.

Application-Level Gateways / Proxy Firewalls (Layer 7 – Application Layer)

  • Operates as an intermediary between clients and servers (proxy).
  • Inspects application-specific traffic such as HTTP, FTP, or SMTP.
  • Can block malicious requests, enforce content filtering, and hide internal systems from direct exposure.
  • Example: A proxy firewall can block file transfers over FTP but still allow FTP login requests.
  • Downside: Higher latency and not always compatible with all applications.

Next-Generation Firewalls (NGFWs) (Multiple Layers, incl. Layer 7)

  • An evolution of stateful firewalls with deep packet inspection capabilities.
  • Can identify and control applications regardless of port or protocol.
  • Provide integrated threat detection features such as intrusion prevention (IPS), SSL inspection, and malware detection.
  • Example: Allow Skype voice calls but block Skype file transfers.
  • More resource-intensive and costly compared to traditional firewalls.

Web Application Firewalls (WAFs) (Layer 7 – Application Layer)

  • Designed specifically to protect web applications.
  • Intercepts HTTP/HTTPS requests and filters malicious payloads (SQL injection, XSS, CSRF, etc.).
  • Shields web servers from direct exposure by sitting between clients and the application.
  • Limitation: May degrade application performance and doesn’t apply to non-web services.

Cloud Firewalls (Virtual Firewalls)

  • Delivered as a cloud service or software-defined firewall within virtualized environments.
  • Protects cloud infrastructure and workloads (e.g., AWS Security Groups, Azure NSGs).
  • Provides scalable security in dynamic environments where physical firewalls are not practical.
  • Often includes advanced logging, automation, and integration with cloud-native monitoring tools.

Event Monitoring

Modern networks produce a huge amount of log data. Devices such as routers, firewalls, proxies, and intrusion detection systems continuously generate logs, which can be sent to a Security Information and Event Management (SIEM) platform for centralized collection and analysis.

A SIEM provides security teams with dashboards, correlation rules, and alerts that help detect suspicious or malicious activity across the entire environment. By consolidating data from many sources, analysts gain better visibility and can investigate incidents more efficiently.

Some examples of valuable event logs from network devices include:

Web Proxy Logs

  • Track web requests made by employees.
  • Provide visibility into visited domains, URLs, and categories.
  • Can be compared against blacklists or threat intelligence feeds to block access to known malicious sites.
  • Example: If a user tries to access a phishing website or adult content, the SIEM can generate an alert.

Perimeter Firewall Logs

  • Firewalls are the first line of defense against external threats.
  • They log blocked and allowed traffic, which can reveal scanning activity, brute-force attempts, or potential denial-of-service (DoS/DDoS) attacks.
  • Example: A sudden flood of connection attempts from a single IP may indicate a port scan.

Intrusion Detection/Prevention Logs (IDS/IPS)

  • Provide detailed alerts about suspicious or malicious traffic patterns.
  • Can show whether an attack was merely detected or actively blocked (IPS).
  • Example: A signature-based alert detecting an exploit attempt against a vulnerable web server.

VPN and Remote Access Logs

  • Record connections from external users into the corporate network.
  • Useful for detecting unusual access patterns, such as logins from unexpected geolocations or outside normal working hours.
  • Example: A successful VPN login from a country where the company has no employees may indicate credential theft.

Switches, Routers, and Network Infrastructure Logs

  • Track routing updates, interface changes, and configuration modifications.
  • Help identify misconfigurations or attempts at network tampering.
  • Example: Unexpected route changes could indicate malicious redirection attempts.
By collecting and correlating all these logs, analysts can detect early warning signs of attacks, such as scanning, data exfiltration, or lateral movement, and respond before major damage occurs.

Network Access Control (NAC)

Network Access Control (NAC) is a security approach that regulates which devices can connect to a network and what they are allowed to access once connected. NAC enforces security policies to ensure that only trusted, compliant, and authorized devices are granted access.

It generally operates in two stages: pre-admission (before access) and post-admission (after access).

Pre-Admission Control

  • Verifies a device’s security posture before allowing it onto the network.
  • Common requirements include:
    • Latest operating system patches.
    • Active and updated antivirus software.
    • Proper configuration of security tools (e.g., host-based firewalls, endpoint protection).
  • If a device fails these checks, it can be blocked or placed into a restricted “quarantine” network until it complies.
  • Especially useful for BYOD (Bring Your Own Device) and guest networks, where personal devices might not meet corporate security standards.

Post-Admission Control

  • Continues monitoring and enforcing policies even after the device has been allowed onto the network.
  • Helps prevent insider threats or lateral movement by restricting what resources users can access.
  • Typically implemented using Role-Based Access Control (RBAC). For example:
    • An employee laptop may access email servers and shared drives but not domain controllers.
    • A guest device may only be allowed internet access, with no internal resources.

Why NAC Matters

  • Prevents rogue or compromised devices from gaining full network access.
  • Reduces the attack surface by limiting what devices can interact with.
  • Provides an additional layer of control beyond authentication and firewall rules.
  • Complements other defenses like IDS/IPS and endpoint protection.

Web Proxy

A web proxy acts as an intermediary between users inside the network and resources on the internet. Instead of connecting directly to external websites, requests are routed through the proxy, which then forwards the request on behalf of the user. The response is sent back to the proxy and then delivered to the client.

This setup improves security, privacy, and control over web traffic.

How Web Proxies Work

  1. A user’s request (e.g., visiting a website) is sent to the proxy.
  2. The proxy evaluates the request against its rules and security policies.
  3. If approved, the proxy forwards the request to the external server.
  4. The server responds, and the proxy passes the response back to the user.

Security Benefits of Web Proxies

  • Blocking Malicious Content – Known malicious domains or URLs can be preemptively blocked before users ever connect to them.
  • Phishing Protection – If phishing emails contain malicious links, those URLs can be blacklisted at the proxy, preventing accidental clicks from causing harm.
  • Content Filtering – Organizations can restrict access to categories such as gambling, adult content, or peer-to-peer file-sharing.
  • Anonymization – Since external websites only see the proxy’s IP address, the internal network structure is hidden.
  • Logging & Monitoring – Every request is logged, allowing security teams to analyze browsing activity for compliance and investigations.

Example Scenario

An employee receives a phishing email with a link to BensGardeningSupplies.xyz, which hosts malware. When the employee clicks the link:

  • The request goes through the corporate proxy.
  • The proxy blocks the domain because it matches a blacklist entry.
  • The user is prevented from reaching the malicious site, reducing risk of compromise.

Types of Web Proxies

  • Forward Proxy – Used by internal clients to access external resources.
  • Reverse Proxy – Protects internal servers by handling requests from external users, often used for load balancing and security.
  • Transparent Proxy – Operates without requiring configuration on client machines; traffic is automatically redirected through the proxy.
