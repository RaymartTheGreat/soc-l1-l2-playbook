Original Notes

Logging Fundamentals

Logging records timestamped events across applications, operating systems, network devices, and cloud services.

Why It Matters

  • Forensics: Trace actions by time, user, and system.
  • Incident Response: Reconstruct attacker activity and prioritize alerts.
  • Threat Hunting: Identify anomalies and patterns in large datasets.
  • Compliance: Maintain audit trails for regulations like PCI‑DSS and GDPR.

Anatomy of a Log Entry

Every structured log entry includes these core fields:

  1. Timestamp
    • Purpose: Correlate events across systems—ensure NTP sync.
    • (e.g., 2025‑08‑05T10:23:47.128Z)
  2. Source
    • Identifier: Hostname, IP, or service tag (e.g., host=Omar.local)
    • Purpose: Locate origin—include region/AZ in cloud setups.
  3. Level
    • Values: DEBUG, INFO, WARN, ERROR, CRITICAL
    • Purpose: Prioritize issues.
  4. Component
    • Identifier: Application or module name (e.g., component=nginx)
    • Purpose: Pinpoints the exact software module.
  5. Message
    • Payload: Free text or structured JSON/KV pairs (e.g., {"user":"alice","action":"login","status":"failed"})
    • Contains event details.

Example

json
2025-08-05T10:23:47.128Z host=Omar.local level=ERROR component=payment-processor message={"txn_id":"12345","error":"timeout"}

Windows Event Logs

Channels

  • Application: Events from applications or programs (e.g., app errors or warnings).
  • Security: Audit events like logon attempts, resource access, and policy changes.
  • Setup: Events related to application setup and Windows installation.
  • System: Events from Windows system components like drivers and services.
  • Forwarded Events: Events collected from remote computers via event subscriptions.

Access Methods

  1. Event Viewer (GUI)
  2. Wevtutil.exe (CLI)
  3. Get-WinEvent (PowerShell)

Key Fields

  • Log Name (e.g., Security)
  • Source (e.g., Microsoft-Windows-Security-Auditing)
  • Event ID (e.g., 4625 = failed logon)
  • Level (Information, Warning, Error, Critical)
  • Category/Task
  • Record & Correlation IDs
  • Date/Time (UTC & Local)
  • User & Computer
  • Event Data/Description

Sysmon

A Microsoft Sysinternals tool that extends Windows logging with detailed process, network, and file operation events—crucial for deep-dive investigations and threat hunting.

Key IDs:

  • Event ID 1: Process creation (with command line)
  • Event ID 3: Network connection (source/dest IP, port)
  • Event ID 7: Image loaded (DLL loads)
  • Event ID 8: CreateRemoteThread (process injection)
  • Event ID 11: FileCreate (file writes)
  • Event ID 12–14: Registry modifications (key create, delete, value change)
  • Event ID 15: FileCreateStreamHash (alternate data streams)
  • Event ID 22: DNS query

Why Use Sysmon

  • Gives visibility into attacker techniques (e.g., process hops, lateral movement).
  • Generates structured, rich events that SIEM tools can parse easily.

Syslog (UNIX)

Protocol for sending log messages to a centralized server, standardized by RFC 5424.

External image reference: https://d2y9h8w1ydnujs.cloudfront.net/uploads/content/images/f0be9badb2b7bb3ab87f5ef469c14a1bc5c6a5097513b16ce0decf740840dec30c1d72cce29320b64589084e1646.png

A Syslog message is made of three components; a Priority Value (PRI), a Header, and a Message.

Priority Value (PRI)

The Priority Value is calculated from the combination of the Facility Code and the Severity Level. To determine PRI, the following formula is used:

(Facility code * 8) + Severity value = PRI

Header

The header provides key identifying details, such as the Timestamp, Hostname, Application name, and Message ID. This information helps determine the source of the system message.

Message

The message itself may appear as human-readable text or as machine-readable content. While the protocol defines the format, it does not control the actual content. Each message sent to the Syslog server includes two labels that make it easier to process. The first label indicates the function (facility) of the application that produced the log. For instance, mail servers usually log with the mail facility. The second label indicates the severity level. Following these labels, the action is specified—commonly a filename within the /var/log directory, where the logs will be stored.

Why Syslog Matters

Syslog is critical in modern security and operations because it allows centralized log collection from servers, network devices, and security tools. By forwarding all events to a Syslog server or SIEM (e.g., Splunk, QRadar, Graylog, ELK), analysts can:

  • Aggregate logs from different sources into one place.
  • Correlate and enrich data (e.g., add threat intel, geo-IP, user context).
  • Detect security incidents such as brute force attempts, lateral movement, or malware activity.
  • Meet compliance requirements by keeping long-term log archives.

In short, Syslog doesn’t just move log data / it enables tools like Splunk and other SIEMs to make sense of events across the environment and turn raw logs into actionable intelligence.

Log Aggregation

Log aggregation is the practice of collecting log data from multiple systems and consolidating it into a central location for analysis, monitoring, and alerting. This enables security teams to search, correlate, and visualize log events across an entire environment, forming the foundation for SIEM (Security Information and Event Management) platforms.

Common Log Aggregation Methods

  1. Syslog Forwarding

A widely adopted protocol that allows systems and devices to forward logs to a central syslog server. The format is structured and easy to parse, making it ideal for network appliances and UNIX-based systems.

  1. Event Streaming

Technologies like SNMP, NetFlow, and IPFIX stream telemetry and operational data from network hardware. Aggregators ingest this data for traffic analysis, anomaly detection, and infrastructure monitoring.

  1. Log Forwarders / Agents

Lightweight agents (e.g., Fluentd, Logstash, Beats) run on endpoints or servers, collecting logs, enriching them with metadata, and forwarding them to the aggregator in near real-time.

  1. Direct API Access

Some aggregators integrate directly with platforms (e.g., AWS, Azure, GCP) using APIs to pull logs or events. This method requires authentication and platform-specific configurations.

Log Data Types

When logs are ingested into a SIEM, they typically fall into two broad categories:

  • Structured Logs

These logs follow a consistent format with clearly defined fields (e.g., source IP, timestamp, user). Examples include Windows Event Logs, Apache access logs, and firewall logs. Structured logs are easier to parse and normalize.

  • Unstructured Logs

These come from applications that don’t enforce a standard forma log entries might vary in structure or span multiple lines. Parsing and correlating unstructured data requires custom rules, regex patterns, or machine learning techniques.
