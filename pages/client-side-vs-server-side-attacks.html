Original Notes

Client-side vs Server-side attacks

Client-side attacks exploit the user’s browser/device or their actions. The attacker runs malicious code in the user’s environment to steal sessions, credentials, or data that lives on the endpoint.

Server-side attacks target the server, application logic, backend components, or database. These attacks exploit flaws in code, configuration, or input handling and usually leave traces on the server and network.

Client-side attacks

Attacks executed inside the user’s browser or on the endpoint (e.g., injected JavaScript, malicious plugins, or social-engineering that causes unsafe client behavior).

Why they matter: Modern web apps load lots of third-party scripts and browser features; that increases the client attack surface. Sensitive data (cookies, localStorage, session state) often lives in the browser, so successful client compromise can lead to account takeover or data leakage.

Short example: A page contains an invisible iframe or script that runs when a user visits. That script reads session cookies or performs actions as the user, allowing the attacker to impersonate the victim without touching the server directly.

SOC visibility limitations

  • Traditional server logs and perimeter captures rarely show what runs inside a user’s browser.
  • Client attacks can operate without generating suspicious HTTP requests or abnormal network patterns visible to the SOC.
  • Detection usually requires browser-side controls (Content Security Policy, browser instrumentation), endpoint telemetry, EDR, or user-behavior analytics.

Common client-side attacks

  • Cross-Site Scripting (XSS): Malicious script runs in the victim’s browser (stored or reflected). Real impact: session theft, DOM manipulation, credential harvesting.
  • Cross-Site Request Forgery (CSRF): Browser is tricked into sending authenticated requests for the user (state-changing actions).
  • Clickjacking: Invisible overlays make users click things they didn’t intend (e.g., enable webcam, change settings).

Server-side attacks

Attacks that exploit server code, backend services, misconfigurations, or improper input handling. They target the application stack (web server, app code, database, OS).

Why they matter: Server compromises often expose entire datasets, allow persistence, or let attackers run commands with application privileges.

Short example: An input field is concatenated directly into a SQL query. An attacker supplies a payload that changes the SQL logic and dumps the database.

Better Detection

  • Server-side activity is processed by the server and usually logged, and it traverses network infrastructure so there are traces in access logs, error logs, and packet captures. That makes detection and forensic analysis more feasible than client-only compromises.

Common server-side attacks

  • Brute force: Automated credential guessing against login endpoints. Look for high-frequency POSTs or many failed logins from the same IP.
  • SQL Injection (SQLi): Malicious input alters database queries; can lead to data exfiltration or modification.
  • Command Injection: User input passed to system commands without sanitization attacker executes arbitrary OS commands.

Visibility & Detection

Logs & network captures

Access logs

  • Typical fields: client IP, timestamp, request line, status code, response size, referrer, user-agent.
  • Strength: show server-side events and patterns (scan, brute force, exploitation attempts).
  • Limitation: often do not include POST bodies or full payloads (depends on config); sensitive data submitted in the request body may be absent.

Network captures (pcap)

  • Show full packets (headers, payloads) if traffic is unencrypted or decrypted. Can reveal full POST bodies, cookies, and file transfers.
  • Use Wireshark filters (e.g., http) and “Follow HTTP Stream” to reassemble requests/responses.
  • Limitation: HTTPS prevents payload inspection without keys or TLS interception.

WAFs

WAFs inspect HTTP(S) traffic and can block or challenge malicious requests before they reach the app. They help where application hardening is incomplete.

Typical rule types

  • Block known bad patterns (SQLi payloads, malicious User-Agents).
  • IP reputation / geo restrictions.
  • Custom rules for app-specific paths (e.g., enforce methods allowed on endpoints).
  • Rate limiting (throttle login attempts, API abuse).
  • Challenge-response (CAPTCHA) to separate human users from bots.
WAFs are strong first-line mitigations but can be evaded by crafted payloads, custom tools, or encrypted traffic if TLS is not inspected. Use WAFs with secure coding, input validation, and proper logging.
