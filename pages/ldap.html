Original Notes

LDAP

Lightweight Directory Access Protocol (LDAP) is an open, cross-platform protocol designed for accessing and managing directory information. In Active Directory (AD), LDAP is the backbone for querying, updating, and managing data about users, groups, and resources. Its standardized communication mechanism makes LDAP essential for centralized identity management and authentication in Windows environments.

Key Points:

  • Centralized Management: LDAP enables systems and applications to access and modify AD data in a structured, hierarchical format.
  • Critical Role in AD: LDAP is used for directory lookups, user authentication, and maintaining organizational data integrity.

Directory Structure

LDAP organizes data in a tree-like hierarchy that resembles a file system:

  • Top-Level Domain (TLD):

dc=ldap,dc=thm

  • Subdomains/Organizational Units (OUs):

ou=people ou=groups ou=resources

  • Entries & Naming:
    • Distinguished Names (DNs): Full, unique paths to entries (e.g., cn=John Doe,ou=people,dc=example,dc=com).
    • Relative Distinguished Names (RDNs): The individual name component (e.g., cn=John Doe).
    • Attributes: Properties assigned to entries (e.g., mail=john@example.com).

Communication Ports:

  • Standard LDAP traffic typically uses port 389.
  • For secure connections (LDAP over SSL, known as LDAPS), communication occurs over port 636.

How LDAP Works

LDAP operates on a client-server model:

  • Directory System Agent (DSA): The LDAP server (typically the Domain Controller in AD) that processes requests.
  • Directory User Agent (DUA): The client (e.g., a user’s workstation or application) that initiates LDAP sessions.

An LDAP session begins when a client connects to a DSA.

The process involves the following steps:

  1. Connection Establishment: The client establishes a connection to the LDAP server.
  2. Binding: The client “binds” to the server using authentication credentials. This step sets the state for the session.
  3. Directory Query: The client issues queries (or updates) using a Distinguished Name (DN).
  4. Response and Unbinding: The server processes and returns the result, then the client disconnects.

LDAP Authentication Methods

LDAP supports multiple methods to authenticate users, primarily categorized into two types:

  1. Simple Authentication:
    • This method relies on a straightforward username and password combination, which is sent as part of the bind request.
    • It also includes cases of anonymous or unauthenticated access, although these are less common in secure environments.
  2. SASL (Simple Authentication and Security Layer) Authentication:
    • SASL allows LDAP to integrate with other authentication services like Kerberos.
    • Employs challenge-response mechanisms for enhanced security.
Security Note:
By default, LDAP transmits data in plaintext. It is critical to use TLS/SSL (LDAPS) to encrypt communications and safeguard credentials and other sensitive information.

Security Concerns with LDAP:

  1. Plaintext Communication: Without LDAPS, LDAP transmits sensitive data, including credentials, in plaintext.
  2. Weak Bind Authentication: Simple binds can use weak or no authentication, exposing systems to brute force and impersonation.
  3. Misconfigured Permissions: Overly permissive access controls can expose critical AD data.
  4. Injection Vulnerabilities: Improper input validation can lead to LDAP injection attacks.
  5. Exposed LDAP Endpoints: Unsecured LDAP servers become rich sources for enumeration and reconnaissance.

LDAP Injection

LDAP Injection is a critical security vulnerability that occurs when user input is not properly sanitized before being incorporated into LDAP queries. This oversight allows attackers to manipulate these queries, leading to unauthorized access or manipulation of the LDAP directory data.

LDAP Injection exploits the way web applications construct LDAP queries. When user input is directly included in these queries without proper validation or encoding, attackers can inject malicious LDAP statements. This can result in unauthorized access to sensitive information, modification of directory data, or bypassing authentication mechanisms.

The process is analogous to SQL Injection, where malicious SQL statements are injected into queries to manipulate database operations. In LDAP Injection, the malicious code targets LDAP queries instead.

Common Attack Vectors

  1. Authentication Bypass: Modifying LDAP authentication queries to log in as another user without knowing their password.
  2. Unauthorized Data Access: Altering LDAP search queries to retrieve sensitive information not intended for the attacker's access.
  3. Data Manipulation: Injecting queries that modify the LDAP directory, such as adding or modifying user attributes.

Brief

What It Is & Its Purpose:

LDAP is the protocol used to query and modify directory services like Active Directory. It underpins how AD organizes and provides access to user, computer, and group information.

Attackers target LDAP to:

  • Enumerate the AD Environment: Gather details about users, groups, organizational units, and computer accounts.
  • Map Out Attack Paths: Identify potential high-value targets and understand group memberships that may allow privilege escalation.
  • Exploit Misconfigurations: Abuse weak or anonymous binds to extract sensitive directory data.
