Campfire-2 / AS-REP roasting

Scenario

Forela's Network is constantly under attack. The security system raised an alert about an old admin account requesting a ticket from KDC on a domain controller. Inventory shows that this user account is not used as of now so you are tasked to take a look at this. This may be an AsREP roasting attack as anyone can request any user's ticket which has preauthentication disabled.

https://app.hackthebox.com/sherlocks/736

Before diving into the logs, it’s important to understand what AS-REP Roasting is

AS-REP Roasting is a Kerberos attack technique where an attacker targets user accounts that have pre-authentication disabled. In a normal Kerberos authentication flow, pre-authentication requires users to encrypt a timestamp with their password before receiving a ticket. When this is disabled, any unauthenticated user can request a Kerberos ticket for the target account, and the KDC will return an AS-REP message (Authentication Service Response) encrypted with the user’s password hash. The attacker can then brute-force or crack this hash offline to retrieve the cleartext password.

This attack doesn’t require prior access to the network and can be performed anonymously, making it a favorite method for initial access and privilege escalation in Active Directory environments.

Detecting AS-REP Roasting in Logs

To detect AS-REP Roasting, we focus on specific Windows Security Event Logs:

  • Event ID: 4768 – A Kerberos authentication ticket (TGT) was requested.

with

Pre-Authentication Type = 0

  • This indicates that pre-authentication is disabled, a requirement for AS-REP Roasting to work.

Service Name = krbtgt

  • All legitimate TGT requests go through the krbtgt service, which handles Kerberos ticket issuance.

Ticket Encryption Type = 0x17 (RC4-HMAC)

  • RC4 encryption is legacy and vulnerable. Attackers prefer it because the returned hash is easier to crack.
Tip: Filtering for Pre-Authentication Type = 0 will eliminate most noise and help you focus on accounts that are truly vulnerable to AS-REP Roasting.

Once you identify a vulnerable account that received a ticket (likely the victim), the next step is to determine who requested it.

  • Review the IP address, hostname, or Logon Account in the same log entry or logs around the same timestamp.
  • Look for unusual accounts, source machines, or activities that correlate with the timestamp of the AS-REP request.

2024-05-29 06:36:40

  • Encryption Type = 0x17 (RC4-HMAC)
  • Pre-authentication Type = 0 (Disabled)
  • Service Name = krbtgt

These attributes confirm that the request matches the pattern of an AS-REP Roasting attack.

5- We do not have any artifacts from the source machine yet. Using the same DC Security logs, can you confirm the user account used to perform the ASREP Roasting attack so we can contain the compromised account/s?

happy.grunwald

To identify the account used in the attack, that pivoted using the source IP address (172.17.79.129) the same machine that sent the AS-REP request.

By reviewing log entries around the same timestamp, we found a 4769 event (service ticket request) just before the AS-REP event. This log shows that the user happy.grunwald was active on the same machine.

While this event is not inherently malicious, it confirms that happy.grunwald was logged into the attacking machine and was likely the account used to perform the AS-REP Roasting attack. This suggests that happy.grunwald is the compromised account being used to move laterally and escalate privileges, while arthur.kyle was the targeted victim account.
