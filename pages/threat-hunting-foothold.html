Original Notes

Threat Hunting: Foothold

This section focuses on how to detect early intrusion attempts (initial foothold) and the subsequent execution phase once adversaries breach defenses. It outlines the attacker’s mindset, various tactics to gain initial access, and methods to execute malicious commands using built-in tools or scripting.

1- Initial Access (TA0001)

Definition:

Techniques that adversaries use to gain their first entry. This phase is all about delivering a payload to secure a foothold.

Examples of Methods:

Key Point:
The foothold isn’t limited to workstations—it extends to any entry point (account or machine) that an attacker can leverage.

Hunting for Initial Access Attempts

Scenario 1: Brute-Forcing via SSH on Jumphost

  • Data Source:filebeat- index in Kibana.
  • Methodology:
    1. Identify Failed SSH Attempts:
      • Use a Lens visualization table filtering by index (filebeat), with rows for source.ip and user.name and metric count.
      • KQL Query:host.name: jumphost AND event.category: authentication AND system.auth.ssh.event: Failed

    1. Identify Successful SSH Logins:
      • Update the query to filter for accepted events from the top suspicious IPs:host.name: jumphost AND event.category: authentication AND system.auth.ssh.event: Accepted AND source.ip: (167.71.198.43 OR 218.92.0.115)

    1. Follow-Up Investigation:
      • Once a successful login is detected (e.g., from IP 167.71.198.43 using the dev account), the next step is to analyze the subsequent commands executed via SSH.

Scenario 2: Remote Code Execution on Web01

  • Data Source:packetbeat- index in Kibana.
  • Methodology:
    1. Detect Directory Enumeration:
      • Create a Lens table visualization for HTTP traffic targeting web01 on port 80.
      • KQL Query:host.name: web01 AND network.protocol: http AND destination.port: 80
      • Look at http.response.status_code—a high count of 404 responses suggests enumeration (e.g., using Gobuster).

    1. Drill Down on Enumeration:
      • Use the Discover tab with a query filtering on status code 404, the attacker’s IP, and additional fields (query, user_agent, url.query) to confirm directory enumeration.

    1. Identify Successful Exploitation:
      • Change the query to filter for successful status codes (200, 301, 302) to determine valid endpoints.
      • Sort events by timestamp to observe the attack flow, noting the shift from enumeration to exploitation (e.g., focusing on the /gila directory and the use of suspicious PHP code).

Scenario 3: Phishing Links and Attachments

  • Data Source:winlogbeat- index in Kibana.
  • Methodology:
    • For Web Browser Downloads (Chrome):
      • Query for file creation events (Sysmon Event ID 11) by chrome.exe:host.name: WKSTN-* AND process.name: chrome.exe AND winlog.event_id: 11
      • Add columns for computer name, user, and file path.
      • Identify unusual downloads (e.g., files like chrome.exe or microsoft.hta in the Downloads folder).

    • For Outlook Attachments:
      • Query for file creation events from OUTLOOK.EXE:host.name: WKSTN-* AND process.name: OUTLOOK.EXE AND winlog.event_id: 11

      • Focus on attachments that are temporarily stored (e.g., Update.zip in the Outlook cache).
      • Confirm the attachment's contents by querying for related events with the file name.

      • Follow-up by tracking subsequent child processes spawned by the suspicious LNK file.

2- Execution (TA0002)

Goal:

Understand how adversaries execute their malicious payloads after gaining initial access.

Examples of Methods:

Hunting for Malicious Execution

Scenario 1: Usage of Command-Line Tools

  • Data Source:winlogbeat- index.
  • Methodology:
    1. Detect Process Creations:
      • Query for Sysmon Event ID 1 where processes include cmd.exe or powershell.exe:host.name: WKSTN-* AND winlog.event_id: 1 AND process.name: (cmd.exe OR powershell.exe)
      • Add fields such as computer name, user, parent command line, and process command line.

    1. Analyze Anomalies:
      • Note unusual parent-child relationships, e.g., cmd.exe launched from C:\Windows\Temp\installer.exe, which is suspicious.
    2. Deep Dive with PowerShell Script Block Logging:
      • Use Sysmon Event ID 4104 to capture PowerShell script block execution:host.name: WKSTN-* AND winlog.event_id: 4104
      • Filter out benign noise (e.g., repeated “Set-StrictMode” events) to focus on anomalies like the usage of Invoke-Empire (an indicator of the Empire C2 agent).

Scenario 2: Abuse of Built-In System Tools (LOLBAS)

  • Data Source:winlogbeat- index.
  • Methodology:
    • Query for process creation (Event ID 1) and network connection events (Event ID 3) involving tools such as mshta.exe, certutil.exe, and regsvr32.exe:host.name: WKSTN-* AND winlog.event_id: (1 OR 3) AND (process.name: (mshta.exe OR certutil.exe OR regsvr32.exe) OR process.parent.name: (mshta.exe OR certutil.exe OR regsvr32.exe))

    • Analyze fields including computer name, user, parent command line, process command line, and destination IP.
    • Findings:
      • Certutil:

        • Used to download a binary (e.g., installer.exe) to C:\Windows\Temp.(Remember: This binary was also noted in the command-line tools investigation.)

      • Regsvr32:

        • Accessed a remote file (teams.sct) and spawned an encoded PowerShell command.

      • Mshta:

        • Spawned an encoded PowerShell command.

Scenario 3: Scripting and Programming Tools

  • Data Source:winlogbeat- index.
  • Methodology:
    • Query for process creation and network connection events involving scripting tools (python, php, nodejs):host.name: WKSTN- AND winlog.event_id: (1 OR 3) AND (process.name: (python OR php OR nodejs) OR process.parent.name: (python OR php OR nodejs*))
    • Include fields for computer name, user, parent command line, process command line, destination IP, and destination port.

(Python Spawn a child cmd.exe process)

Using this process PID, we can search all processes spawned by this cmd.exe instance by using it as our process.parent.pid:

host.name: WKSTN-* AND winlog.event_id: (1 OR 3) AND process.parent.pid: 1832

    • Findings:
      • Python was seen spawning a child cmd.exe process and initiating a network connection to an external IP (e.g., 167.71.198.43:8080).

      • Further investigation involves obtaining the process ID of the spawned cmd.exe and correlating events using that PID.
      • This pattern suggests the possibility of a reverse shell (e.g., via a malicious script like dev.py).

3- Defense Evasion

Overview:

Defense Evasion encompasses techniques adversaries use to bypass detection by security systems. They disguise malicious actions as legitimate operations by disabling security software, deleting logs, spoofing processes, or exploiting known bypasses to avoid triggering alerts. Often, these techniques are coupled with execution tactics to allow continued stealthy activity.

Objective:

Bypass detection mechanisms, whether these are software solutions (like antivirus) or human analysts monitoring logs.

Common Techniques Include:

Note: The key idea is that while adversaries try to hide, their actions still leave traces in logs and process metadata that can be hunted.

Hunting for Defense Evasion

Scenario 1: Disabling Security Software

  • Goal:

Detect attempts to disable critical security controls (e.g., Windows Defender).

  • Methodology:
    1. Data Source:

      2. Use the winlogbeat- index on Kibana.

    2. KQL Query Example:

``jsx host.name: WKSTN- AND (DisableRealtimeMonitoring OR RemoveDefinitions*) ``

      • Explanation:
        • *DisableRealtimeMonitoring* is used with PowerShell’s Set-MpPreference to disable real-time scanning.
        • *RemoveDefinitions*is used with MpCmdRun.exe to remove Windows Defender signatures.
    • Key Fields to Display:
      • winlog.computer_name
      • user.name
      • process.parent.command_line
      • process.name
      • process.command_line

    1. Findings:
      • An example finding from WKSTN-1 shows Set-MpPreference executed by a malicious installer binary.
      • Similarly, MpCmdRun.exe -RemoveDefinitions is seen from a cmd.exe instance (correlated with previous investigations).

Scenario 2: Log Deletion Attempts

  • Goal:

Identify when a threat actor tries to remove evidence by clearing Windows Event Logs.

  • Methodology:
    1. Data Source:

      2. Use the winlogbeat- index.

    2. KQL Query Example:

`` host.name: WKSTN-* AND winlog.event_id: 1102 ``

      • Explanation:

        • Event ID 1102 is generated when Windows Event Logs are cleared.

    1. Investigation Steps:
      • Review events to confirm that logs were cleared from a workstation (e.g., WKSTN-1).
      • Use the “View surrounding documents” feature to gather context—including command lines used for deletion.
      • Add columns such as process.name and process.command_line to help correlate the deletion event.

Scenario 3: Execution through Process Injection

  • Goal:

Detect process injection—a technique where an attacker injects code (shellcode) into another process to execute commands stealthily.

  • Methodology:
    1. Data Source:

      2. Use the winlogbeat- index.

    2. KQL Query Example:

``jsx host.name: WKSTN-* AND winlog.event_id: 8 ``

      • Explanation:

        • Sysmon’s Event ID 8 (CreateRemoteThread) indicates when one process creates a thread in another, a common sign of process injection.

    • Key Fields to Display:
      • winlog.computer_name
      • process.executable
      • winlog.event_data.SourceUser
      • winlog.event_data.TargetImage

    1. Findings:
      • Example: A process (chrome.exe) from Clifford Miller’s workstation is observed injecting a thread into explorer.exe.

      • Note the unusual relationship where a user-level application (chrome.exe) is targeting a system process like explorer.exe.

4- Persistence (TA003)

Overview:

Persistence refers to the techniques adversaries use to maintain long-term access to a compromised system—even if the system restarts or a user logs out. By implanting persistence mechanisms immediately after gaining execution, attackers ensure continued control over the environment while making their presence harder to detect and remove.

Key Methods Include:

Scenario 1:Scheduled Task Creation

Objective:

Detect unusual or unauthorized creation of scheduled tasks that may be used to automate malicious activities.

Methodology:

  • Data Source:

    • Use the winlogbeat- index within Kibana.

  • Query Approach:

    • Look for Windows Advanced Audit Policy events (e.g., Event ID 4698 for scheduled task creation) or use keywords like schtasks and Register-ScheduledTask.

  • Example KQL Query:

``python host.name: WKSTN- AND (winlog.event_id: 4698 OR (schtasks OR Register-ScheduledTask*)) ``

  • Key Fields to Display:
    • winlog.computer_name
    • user.name
    • process.command_line
    • winlog.event_id
    • winlog.event_data.TaskName

Findings & Analysis:

  • Common scheduled tasks (such as OneDrive Reporting) may be benign.
  • An unusual task might be identified—for example, a task named "Windows Update" that runs a PowerShell command every minute and is associated with suspicious domains (e.g., www.oneedirve.xyz).

  • Next Steps:

    • Investigate the parent process (e.g., a cmd.exe instance) that executed the task creation to trace earlier activity.

Scenario 2: Registry Key Modification

Objective:

Detect modifications to critical registry keys that may indicate attempts to ensure persistence by auto-launching malicious binaries.

Methodology:

  • Data Source:

    • Use the winlogbeat- index and focus on registry modification events (Sysmon Event ID 13) within the Microsoft-Windows-Sysmon/Operational channel.

  • Initial Query Example:

``jsx host.name: WKSTN-* AND winlog.event_id: 13 AND winlog.channel: Microsoft-Windows-Sysmon/Operational ``

    • Note: This query may return a large number of events since many benign registry changes occur.
  • Narrowing Down the Results:

    • Focus on commonly abused registry paths such as:

    • CurrentVersion\Run or RunOnce
    • CurrentVersion\Explorer\User or CurrentVersion\Explorer\Shell
  • Refined KQL Query Example:

``jsx host.name: WKSTN- AND winlog.event_id: 13 AND winlog.channel: Microsoft-Windows-Sysmon/Operational AND registry.path: (CurrentVersion\\Run OR CurrentVersion\\Explorer\\User OR CurrentVersion\\Explorer\\Shell*) ``

  • Key Fields to Display:
    • winlog.computer_name
    • user.name
    • process.name
    • registry.path
    • winlog.event_data.Details

Findings & Analysis:

  • An example finding could include a registry entry such as: Registry Path: HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\RunOnceEx\0001\Depend\1Registry Data: C:\Windows\Temp\installer.exe

  • This entry indicates that the suspicious binary (installer.exe) is configured to execute on startup.
  • Alternative Approach:

    • Use process filtering (e.g., for reg.exe or powershell.exe) to pinpoint registry modifications driven by these utilities.

  • Next Steps:

    • Correlate the event with the parent process (e.g., cmd.exe) to backtrack and understand the chain of events leading to this persistence mechanism.

5- Command and Control (TA0011)

Overview:

Command and Control (C2) refers to the methods adversaries use to communicate with compromised systems within a target network. At this stage, attackers issue remote commands to fulfill their objectives, such as further internal compromise or data exfiltration. This communication is typically designed to blend in with regular network traffic, making detection challenging.

Purpose:

Maintain two-way communication between the attacker and the compromised system to control activities remotely.

Common Communication Channels:

Key Characteristics to Monitor:

  • Egress Traffic:

    • High outbound traffic or unusual file uploads may indicate C2 communication.

  • Ingress Traffic:

    • Unexpected connections from external sources may suggest command instructions.

  • Cleartext Commands:

    • Host commands in unencrypted traffic can indicate an established C2 channel.

  • Volume & Bandwidth:

    • An unusually high count of connections or elevated data volumes can be a red flag.

Hunting Command and Control

The hunt for C2 activities involves uncovering covert communication channels hidden within normal network traffic. The following scenarios illustrate three common C2 methods:

Scenario 1: Command and Control over DNS

Objective:

Detect adversaries using DNS protocols to establish a C2 channel.

Methodology:

  • Data Source:

    • Use the packetbeat- index for DNS query data and correlate with winlogbeat- for process details.

  • Visual Analysis:
    • Create a visualization table (using Lens) configured to display:
      • Rows: dns.question.registered_domain and host.name
      • Metric: Unique Count of dns.question.subdomain
    • KQL Query Example:

``jsx network.protocol: dns AND NOT dns.question.name: *arpa ``

      • Explanation: This query excludes reverse DNS lookups to focus on standard queries.

  • Investigation Steps:
    • Identify domains with an unusually high count of unique subdomains (e.g., a domain like golge.xyz showing many unique subdomain queries).
    • Drill down using the Discover tab with a refined query:

``jsx network.protocol: dns AND NOT dns.question.name: *arpa AND dns.question.registered_domain: golge.xyz AND host.name: WKSTN-1 ``

      • Add additional fields (such as the query field) to understand the nature of the requests.

    • Correlate with winlogbeat data to identify the process generating these DNS requests:

``python host.name: WKSTN-1* AND destination.ip: 167.71.198.43 AND destination.port: 53 ``

      • Key fields to display include host.name, user.name, process.parent.command_line, process.name, and process.command_line.

Based on the results, it can be observed that all connections to 167.71.198.43:53 are generated by nslookup.exe

Scenario 2: Command and Control over Cloud Applications

Objective:

Identify C2 channels that leverage legitimate cloud services to disguise malicious traffic.

Methodology:

  • Data Source:

    • *Use packetbeat- to monitor network connections and winlogbeat- to correlate process activity.*

  • Visual Analysis:
    • Modify the visualization table (from the DNS analysis) by removing the unique subdomain metric and sorting the count in ascending order.
    • Look for cloud service domains that are not typically accessed by your workstations (e.g., domains like discord.gg).

  • Investigation Steps:
    • Use a winlogbeat query to correlate the cloud connection:

``jsx host.name: WKSTN-1 AND discord.gg* ``

      • This reveals the process associated with the connection.

    • Identify that the connection is initiated by a suspicious binary (e.g., C:\Windows\Temp\installer.exe).
    • Query for child processes spawned by this binary:

``python host.name: WKSTN-1* AND winlog.event_id: 1 AND process.parent.executable: "C:\\Windows\\Temp\\installer.exe" ``

      • ( multiple cmd.exe commands) confirming C2 activity over Discord.

Scenario 3: Command and Control over Encrypted HTTP Traffic

Objective:

Uncover C2 communications that use custom, encrypted HTTP channels.

Methodology:

  • Data Source:

    • *Use the packetbeat- index to monitor HTTP traffic and winlogbeat- for process correlations.*

  • Visual Analysis:
    • Create a visualization table (using Lens) configured with:
      • Rows: host.name, destination.domain, and http.request.method
      • Metric: Count of HTTP requests
    • KQL Query Example:

``jsx network.protocol: http AND network.direction: egress ``

      • This displays outbound HTTP requests.

  • Investigation Steps:
    • Identify unusual outbound connections to distinct domains (e.g., a domain like cdn.golge.xyz showing a high volume of GET requests).
    • Refine the query to focus on the suspicious domain:

``jsx host.name: WKSTN-* AND network.protocol: http AND network.direction: egress AND destination.domain: cdn.golge.xyz ``

      • Adjust the table rows to include only pertinent fields.

    • Correlate these findings with winlogbeat data using a query such as:

`` host.name: WKSTN- AND cdn.golge.xyz* ``

      • This correlation can reveal that the C2 connection was established using a malicious PowerShell command.

    • Next Steps:

the next step of this investigation is to identify the extent of these malicious activities by correlating the subsequent events generated after the C2 connection to cdn[.]golge[.]xyz was established. Moreover, it is also good to trace back how the attacker gained initial access in the first place before attempting to develop continuous C2 access.
