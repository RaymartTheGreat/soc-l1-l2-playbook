Windows Internals

Understanding Windows Processes

A process represents the execution of a program in Windows. It consists of multiple components that define how the process interacts with system resources. Each process contains:

  • A virtual address space – The memory allocated for execution.
  • Executable code – The program’s logic.
  • Open handles – Access to system resources.
  • Security context – Determines permissions and privileges.
  • Process ID (PID) – A unique identifier.
  • Environment variables – Configuration settings.
  • Priority class – Defines execution priority.
  • Threads – The actual execution units within a process.

Processes in Action

Windows applications run as processes. Examples of system processes:

  • MsMpEng – Microsoft Defender.
  • wininit – Handles keyboard and mouse interactions.
  • lsass – Stores user credentials.

Process Manipulation and Attacks

Attackers often manipulate processes to evade detection. Common attack techniques include:

  • Process Injection (T1055) – Injecting code into another process.
  • Process Hollowing (T1055.012) – Replacing the memory of a legitimate process.
  • Process Masquerading (T1055.013) – Renaming a malicious process to appear legitimate.

Threads in Windows

A thread is the fundamental execution unit within a process. While a process provides structure, threads handle execution.

Thread Characteristics

  • Threads share resources with their parent process.
  • Each thread has unique data, such as stack, local storage, and execution context.
  • Threads can be targeted by attackers for code execution exploits.
ComponentPurpose
StackStores function calls, local variables, and execution context.

Virtual Memory in Windows

Windows virtual memory allows applications to use memory as if it were physical RAM. This ensures:

  1. Isolation – Prevents processes from interfering with each other.
  2. Address Translation – Converts virtual addresses to physical ones.
  3. Paging – Moves inactive data to disk when memory is full.

Address Space Breakdown

  • 32-bit systems: 4GB total (2GB for processes, 2GB for OS).
  • 64-bit systems: 256TB total, with a similar split.

Virtual Memory Mechanisms

  • Paging – Moves excess data to disk storage.
  • Address Windowing Extensions (AWE) – Allows applications to allocate larger memory spaces.

Dynamic-Link Libraries (DLLs)

A DLL is a shared library containing code and data used by multiple applications simultaneously.

List of the top ranked 30 DLL names by calling frequency

Benefits of DLLs

  • Modularization – Efficient organization of code.
  • Code Reusability – Used across multiple programs.
  • Memory Optimization – Loads only necessary components.

DLL Exploitation Techniques

  • DLL Hijacking (T1574.001) – Replacing or modifying legitimate DLLs.
  • DLL Side-Loading (T1574.002) – Loading malicious DLLs from trusted applications.
  • DLL Injection (T1055.001) – Injecting malicious code into running processes.

Creating a DLL

DLLs in C++ use a specific structure:

cpp
#include "stdafx.h"
#define EXPORTING_DLL
#include "sampleDLL.h"
BOOL APIENTRY DllMain(HANDLE hModule, DWORD ul_reason_for_call, LPVOID lpReserved)
{
    return TRUE;
}
void HelloWorld()
{
    MessageBox(NULL, TEXT("Hello World"), TEXT("In a DLL"), MB_OK);
}

DLL Loading Techniques

  1. Load-Time Dynamic Linking

``cpp #include "sampleDLL.h" int main() { HelloWorld(); return 0; } ``

  1. Run-Time Dynamic Linking

``cpp HINSTANCE hinstDLL; typedef void (*DLLPROC)(); hinstDLL = LoadLibrary("sampleDLL.dll"); DLLPROC HelloWorld = (DLLPROC)GetProcAddress(hinstDLL, "HelloWorld"); HelloWorld(); FreeLibrary(hinstDLL); ``

Kernel vs. User Mode Execution

Windows operates in two distinct modes:

  • User Mode: Where most applications run with limited access to system resources. If an application crashes, it does not affect the entire OS.
  • Kernel Mode: Where core Windows components and drivers execute. A failure here can crash the entire system (Blue Screen of Death).

Understanding these modes helps explain why some malware targets kernel-mode execution for persistence and privilege escalation.

Process States and Scheduling

Processes in Windows transition between different states:

  • Created: The process is initialized but not yet running.
  • Ready: The process is loaded into memory and waiting for CPU execution.
  • Running: The process is actively being executed.
  • Waiting/Blocked: The process is waiting for an external event (e.g., file read, network response).
  • Terminated: The process has finished execution or was forcibly ended.

Windows uses a preemptive scheduler to allocate CPU time based on priority and fairness algorithms.

Page Tables and Memory Protection

The Memory Manager maps virtual addresses to physical memory using page tables. Key memory protections include:

  • DEP (Data Execution Prevention): Prevents code execution from non-executable memory regions.
  • ASLR (Address Space Layout Randomization): Randomizes memory addresses to prevent predictable exploits.
  • Control Flow Guard (CFG): Prevents indirect call/jump hijacking by verifying control flow integrity.

These protections are critical in modern exploit mitigation techniques.

Handles and Object Manager

Windows uses handles to reference system objects (files, registry keys, threads, etc.). The Object Manager manages these handles and enforces security policies. Attackers often exploit:

  • Handle Inheritance: Malware can inherit handles from a privileged parent process.
  • Handle Duplication: Attackers can duplicate existing handles to escalate privileges.

Tools like Sysinternals Handle.exe help investigate handle-based attacks.

Windows API and System Calls

Most applications interact with Windows through the Windows API, which abstracts low-level system calls.

  • Win32 API: Provides core system functions (e.g., file handling, memory allocation, threading).
  • Native API (NtDLL.dll): Lower-level functions used internally by the OS. Malware often directly calls these APIs (e.g., NtCreateThreadEx) to bypass security hooks.

Understanding the API hierarchy is crucial for both development and reverse engineering.
