First what is the MFT?

The Master File Table (MFT) is the central metadata database for NTFS. Every file and directory on an NTFS volume has an MFT entry (also called an MFT record) and a unique MFT record number . Each entry stores attributes such as timestamps, filename(s), security descriptor, and pointers to the file’s data ($DATA). The MFT itself is a special file on the volume ($MFT) and describes all other files—including its own metadata.

Why the MFT is useful for forensics

  1. Comprehensive metadata — MFT entries store creation/modify/access/change timestamps, file size, parent path, flags, and security info, giving a rich snapshot of file state.
  2. Deleted-file recovery — entries often remain after deletion until reused, allowing recovery of filenames, timestamps, and data pointers even when the file was removed.
  3. Timeline reconstruction — multiple MFT timestamps (and filename vs standard-information timestamps) help build a precise activity timeline.
  4. Track file operations — renames, moves, attribute changes, and evidence of overwrites are reflected in MFT attributes.
  5. Malware / suspicious activity detection — unusual MFT activity (e.g., NTDS copies in temporary paths, rapid create/modify patterns) is a strong indicator of malicious behavior.
  6. Correlation point — MFT data can be correlated with the USN Journal, $LogFile, and Windows event logs to validate and enrich findings.

5- Finding the hex offset of an MFT record is beneficial in many investigative scenarios. Find the hex offset of the stager file from Question 3.

16E3000

To find the offset of invoice.bat, we use its MFT entry number (23436). Each entry is 1024 bytes, so 23436 × 1024 = 23998464 (decimal). Converting to hexadecimal gives 16E3000, which is the offset used in forensic tools to locate the file’s raw data.

6- Each MFT record is 1024 bytes in size. If a file on disk has smaller size than 1024 bytes, they can be stored directly on MFT File itself. These are called MFT Resident files. During Windows File system Investigation, its crucial to look for any malicious/suspicious files that may be resident in MFT. This way we can find contents of malicious files/scripts. Find the contents of The malicious stager identified in Question3 and answer with the C2 IP and port.

43.204.110.203:6666

When a file is smaller than 1024 bytes (the size of a single MFT record), it can be stored directly inside the record itself. These are called resident files, and their content is kept in the $DATA attribute of the MFT entry.

To extract this information, we can use MFTECmd with the --de option, which dumps the details of a specific entry number. Since we already know the entry number of the malicious stager file (23436) from earlier, the command looks like this:

bash
MFTECmd.exe -f <PATH_TO_$MFT> --de 23436

Running this displays the full details of the entry, including the DATA attribute. In the ASCII section of that output, we can clearly see the contents of the stager script. This reveals the PowerShell command inside, which points to the attacker’s C2 IP address and port used for the second-stage payload.
