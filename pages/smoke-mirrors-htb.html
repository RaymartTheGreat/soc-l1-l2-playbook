Notes/Tips

  • LSA (Local Security Authority) Protection, also known as LSA Protection Mode or LSA RunAsPPL (Run as Protected Process Light), is a security feature in Windows operating systems designed to enhance the protection of the Local Security Authority Subsystem Service (LSASS). LSASS is a critical component responsible for enforcing the security policy on the system, handling password changes, creating access tokens, and managing user logins.
  • The Antimalware Scan Interface (AMSI) is a Windows security feature that allows applications such as PowerShell, scripting engines, and Office macros to send content to antivirus/EDR solutions for scanning before execution. Its main purpose is to detect and block malicious code at runtime, even if the payload is obfuscated.
  • AMSI relies on a DLL named amsi.dll, which provides functions used by Windows processes to perform these scans. The most important function in this DLL is AmsiScanBuffer. This function is responsible for scanning script content in memory and returning whether it is malicious or safe.

Smoke & Mirrors (HTB)

Scenario

Byte Doctor Reyes is investigating a stealthy post-breach attack where several expected security logs and Windows Defender alerts appear to be missing. He suspects the attacker employed defense evasion techniques to disable or manipulate security controls, significantly complicating detection efforts. Using the exported event logs, your objective is to uncover how the attacker compromised the system's defenses to remain undetected.

https://app.hackthebox.com/sherlocks/Operation Blackout 2025: Smoke &amp; Mirrors

I will use splunk for this challenge

1- The attacker disabled LSA protection on the compromised host by modifying a registry key. What is the full path of that registry key?

HKLM\SYSTEM\CurrentControlSet\Control\LSA

I filtered for the LSA registry path and found it was modified. The logs show a reg add command was used to change the RunAsPPL value to 0, which disables LSA protection and allows LSASS to be dumped more easily.

LSA (Local Security Authority) Protection, also known as LSA Protection Mode or LSA RunAsPPL (Run as Protected Process Light), is a security feature in Windows operating systems designed to enhance the protection of the Local Security Authority Subsystem Service (LSASS). LSASS is a critical component responsible for enforcing the security policy on the system, handling password changes, creating access tokens, and managing user logins.

https://medium.com/h7w/lsa-protection-bypass-detection-16e8db3ab66c

2- Which PowerShell command did the attacker first execute to disable Windows Defender?

Set-MpPreference -DisableIOAVProtection $true -DisableEmailScanning $true -DisableBlockAtFirstSeen $true

After searching PowerShell logs for Set-MpPreference, I found the attacker ran this cmdlet with the listed parameters. Set-MpPreference changes Windows Defender settings / these flags disable real-time IOAV, email scanning, and block-at-first-seen, weakening Defender protections.

3- The attacker loaded an AMSI patch written in PowerShell. Which function in the DLL is being patched by the script to effectively disable AMSI?

AmsiScanBuffer

To investigate this activity, I filtered the PowerShell logs using:

python
("*amsi.dll*" OR "*AmsiScanBuffer*" OR "*Add-Type*" OR "*kernel32.dll*" OR "*VirtualProtect*" OR "*GetProcAddress*" OR "*GetModuleHandle*")

These keywords were chosen because they are strongly associated with AMSI bypass techniques in PowerShell.

  • amsi.dll / AmsiScanBuffer → direct references to the AMSI library and its main scanning function.
  • Add-Type → commonly used to inject custom C# code into PowerShell.
  • kernel32.dll, GetProcAddress, GetModuleHandle, VirtualProtect → Windows API calls frequently abused to locate and modify the AMSI function in memory.

By searching for these, I was able to detect the script that patches AmsiScanBuffer, effectively disabling AMSI.

The Antimalware Scan Interface (AMSI) is a Windows security feature that allows applications such as PowerShell, scripting engines, and Office macros to send content to antivirus/EDR solutions for scanning before execution. Its main purpose is to detect and block malicious code at runtime, even if the payload is obfuscated.
AMSI relies on a DLL named amsi.dll, which provides functions used by Windows processes to perform these scans. The most important function in this DLL is AmsiScanBuffer. This function is responsible for scanning script content in memory and returning whether it is malicious or safe.

https://www.offsec.com/blog/amsi-write-raid-0day-vulnerability/

4- Which command did the attacker use to restart the machine in Safe Mode?

bcdedit.exe /set safeboot network

I searched for the string safeboot in the command logs, since any attempt to configure Safe Mode requires this parameter. Filtering for it revealed the attacker used bcdedit.exe /set safeboot network, which forces the system to reboot into Safe Mode with networking.

5- Which PowerShell command did the attacker use to disable PowerShell command history logging?

Set-PSReadlineOption -HistorySaveStyle SaveNothing

I searched for the keyword Set-PSReadlineOption in the PowerShell logs because this cmdlet controls command history behavior. Filtering on this showed the attacker running it with -HistorySaveStyle SaveNothing, which disables saving command history to disk and hides executed commands from later review.
