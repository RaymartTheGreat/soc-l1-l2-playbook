Original Notes

Cross-Site Scripting (XSS)

XSS is when an attacker injects malicious client-side code (usually JavaScript) into a web page that other users load. The browser executes that code in the context of the site, so the attacker can hijack sessions, steal cookies, redirect users, deface pages, or do other client-side abuse.

Tiny HTML/JS primer

  • HTML tags: <script>...</script>, <img src=...>, <a href=...>
  • Inline event handlers: HTML attributes that directly trigger JavaScript when an event occurs. (Runs JS code directly in the HTML.)

onclick, onload, onerror, onmouseover, onsubmit

Example : <button onclick="alert('Hi')">Click me</button>

  • JS functions often abused: alert(), document.cookie, window.location, document.write(), eval()
  • JS schemes in URLs: javascript:... (dangerous when reflected into links)

Example : <a href="javascript:alert('Hi')">Click me</a>

These are the building blocks attackers use to run code in users’ browsers.

How it works

If an app takes ?user= and prints it into the page without sanitizing, a normal request:

text
https://site.example/welcome?user=Omar
→ page shows: Hello Omar

But an attacker can send:

text
https://site.example/welcome?user=<script>alert(1)</script>

If the site returns that string raw, the browser runs the script and shows a popup. Real attacks use payloads that steal cookies, redirect users, or inject UI elements.

[PHOTO: xss-vulnerable-example.png]

Types

  • Reflected (non-persistent) — payload sent in request and immediately reflected in response (search, error pages).
  • Stored (persistent) — payload saved on server (comments, profiles) and served to many users — most dangerous.
  • DOM-based — injection happens entirely in the browser when client-side code writes unsafe input into the DOM (e.g., innerHTML = location.hash).

What attackers can gain

  • Steal session tokens / impersonate users
  • Capture credentials or MFA codes (phishing UI)
  • Redirect users to malicious sites
  • Perform actions on behalf of victim (CSRF-like)
  • Deface content or inject malware / trackers

Detection

Where to look

  • Web access logs (query strings, POST bodies)
  • App logs (stored content creation endpoints)
  • WAF logs (XSS signatures)
  • Frontend source code (search for innerHTML, document.write, eval, location.hash)
  • Browser console logs if you can reproduce in a lab

What to search for (decoded values)

  • Raw/encoded script markers: <script, %3Cscript%3E, </script>
  • Inline event handlers: onload=, onclick=, onerror=, onmouseover=
  • javascript: in href or src parameters
  • Common payload words: alert, prompt, document.cookie, window.location
  • img src=x onerror=... patterns or <svg onload=...> vectors
  • Lots of percent-encoding or double-encoding (%3C, %3E, %25) — attacker hiding payloads
  • Requests creating stored content (POST to comments, profile updates) containing tags or on* attributes

Example access log

Raw:

text
GET /search?s=%3Cscript%3Edocument.cookie%3C%2Fscript%3E HTTP/1.1" 200 2048 "-" "python-requests/2.28"

Decoded → <script>document.cookie</script>

Why suspicious:

  • %3Cscript%3E decodes to <script> — direct JS injection
  • document.cookie is a cookie theft attempt
  • Tool UA python-requests or urllib often indicates automated scanner
  • Multiple similar requests in short time → automated fuzzing

DOM-based hints (search in frontend code)

Look for unsafe patterns in client JS:

  • element.innerHTML = ... with user input
  • document.write(...) or eval(...) on dynamic data
  • location.hash or location.search used directly in HTML insertion

These are high-risk spots for DOM XSS.

Detecting automated scanners

  • High request rate with many different payload permutations every few seconds
  • User-Agent strings like sqlmap, python-requests, urllib, or known scanner UAs
  • Repeated alert(1) / prompt(1) style payloads across many endpoints

Quick checks to run after spotting suspicious payloads

  1. URL-decode parameter(s) and inspect payload.
  2. Check if the payload is reflected in the HTML response (render safely in sandbox).
  3. Search same IP / UA for other payloads or POSTs creating content.
  4. If stored payload found, check which users saw it and snapshot content.
  5. Check for outbound callbacks or redirects initiated by payloads.
  6. Review frontend code for innerHTML, document.write, eval, location.* usage.
  7. If confirmed, remove stored payload, rotate affected session tokens, notify devs to fix encoding/escaping and add CSP.

Prevention / developer tips

  • Output encode user data for the correct context (HTML encode, attribute encode, JS encode).
  • Use framework template escaping (don’t disable it).
  • Validate/whitelist input where possible.
  • Implement Content Security Policy (CSP) to reduce impact.
  • Avoid innerHTML/document.write with unsanitized input; prefer safer DOM APIs.
  • Sanitize HTML if you must allow limited tags (use well-tested libraries).
