Original Notes

ELK (Elastic Stack)

Elastic Stack (ELK) is a lightweight SIEM foundation Beats collect, Logstash parses, Elasticsearch indexes, Kibana hunts & dashboards.

Architecture

[Beats (Filebeat / Winlogbeat / Packetbeat)] → [Logstash (optional parsing/enrichment)] → [Elasticsearch (index/store/analytics)] → [Kibana (Discover / Visualize / Dashboards / Alerts)]

  • Beats — lightweight shippers on endpoints (low overhead).
  • Logstash — use when you need heavy parsing, grok, enrichment, or multi-output routing.
  • Elasticsearch — fast, distributed full-text and analytics store for JSON documents.
  • Kibana — interactive UI for searching, visualizing, alerting, and saving investigations.

Components

Elasticsearch

  • Distributed index and search engine built on Lucene.
  • Stores JSON documents, supports full-text search, aggregations, and analytics.
  • SOC use-case: pivot on IOCs, run high-speed searches across time windows.

Logstash

  • Pipeline pattern: input { } → filter { } → output { }.

  • Powerful plugins: grok, json, date, geoip, mutate, useragent.
  • SOC use-case: normalize heterogeneous vendor logs into consistent fields.

Beats (Filebeat / Winlogbeat / Packetbeat / Metricbeat)

  • Single-purpose, lightweight shippers installed on hosts.
  • Winlogbeat → Windows Event Logs, Filebeat → app logs, Packetbeat → network flows.

  • SOC use-case: reliable data collection with small footprint.

Kibana

  • Web UI: Discover, Visualize, Dashboard, Alerts/Rules, Lens, Maps.
  • Use it for interactive hunts, timelines, saved searches, and incident reports.

How they work together :

Kibana // Discover Tab

Some important elements you’ll typically find in a dashboard interface include:

  1. Logs (Documents): Each log represents a document that stores details about an event, showing different fields and their corresponding values.
  2. Fields Panel: Located on the left side, this panel displays all the fields extracted from the logs. Users can click on a field to add it as a filter or remove it from the current search.
  3. Index Pattern: Allows users to pick an index pattern from the available list to define which data is being queried.
  4. Search Bar: The area where users can type search queries or apply filters to refine results.
  5. Time Filter: Provides options to limit results to a specific time range, offering multiple presets and custom ranges.
  6. Time Interval: A chart visualization that shows how event counts change over time.
  7. Top Bar: Contains various controls, such as saving a search, opening previously saved searches, sharing, and other related actions.

Key fields

Core ECS fields

  • @timestamp — event time
  • agent.name / host.name — host that generated the event
  • user.name / related.user — account involved
  • process.name, process.executable, process.command_line, process.parent.name — process activity
  • event.action, event.category, event.module — event metadata
  • source.ip, destination.ip, user_agent — network context
  • file.path, file.hash.* — file artifacts

text
process.name : "powershell.exe" and (process.command_line : "*Invoke-WebRequest*" OR process.command_line : "*Invoke-RestMethod*" OR process.command_line : "*WebClient*")

text

text

text

Dashboards & Visualizations

Dashboard goals

  1. Detection overview (alerts, top sources, recent suspicious commands).
  2. Endpoint activity (process creates, parent→child mapping).
  3. Authentication & VPN (failed/successful auths, top IPs).
  4. Network anomalies (top dest IPs, unusual ports).
  5. Persistence & changes (services, autoruns, account changes).

Example visualizations

  • Top 10 source countries (Pie / Bar) — source.geo.country_name.
  • Failed logins table — saved search with @timestamp, user.name, source.ip, destination.ip, message.
  • PowerShell execution timeline — line chart of process.name:"powershell.exe".
  • Parent→child mapping — Lens: process.parent.name vs process.name.
  • Recent suspicious commands — data table showing process.command_line for last 24h with filter for suspicious keywords.

Kibana Alerts (Rules) // examples & playbook snippets

Alert: Encoded PowerShell (High)

  • KQL: process.name : "powershell.exe" and process.command_line : "-EncodedCommand"
  • Trigger: any hit in last 5 min.
  • Actions: create Kibana Case, send webhook to SOAR, notify Slack/Email.
  • Webhook payload (example):
json
{
  "title": "Encoded PowerShell detected",
  "severity": "high",
  "host": "{{context.host.name}}",
  "user": "{{context.user.name}}",
  "timestamp": "{{context.alert_time}}",
  "query": "process.name : \"powershell.exe\" and process.command_line : \"*-EncodedCommand*\""
}

Alert: Office spawning a shell

  • KQL: process.parent.name : ("winword.exe" or "excel.exe" or "outlook.exe") and process.name : ("cmd.exe" or "powershell.exe")
  • Action: auto-tag phish-suspect, create case, notify SOC on-call.

Alert: New Service Creation

  • KQL: event.action : "service_created" OR (process.name : "sc.exe" AND process.command_line : "create")
  • Action: run enrichment (hash reputation), add to case, trigger host isolation playbook.

Alert: Excessive failed logons

  • KQL (count rule): event.action : "authentication_failure"
  • Trigger: count > X in Y minutes per source.ip or user.name (tune threshold).
  • Action: add to watchlist, throttle noise, escalate if persistent.

Installation // concise steps

1) Install & configure Elasticsearch (Ubuntu)

bash
wget -qO - https://artifacts.elastic.co/GPG-KEY-elasticsearch | sudo gpg --dearmor -o /usr/share/keyrings/elasticsearch-keyring.gpg
sudo apt-get install apt-transport-https
echo "deb [signed-by=/usr/share/keyrings/elasticsearch-keyring.gpg] https://artifacts.elastic.co/packages/9.x/apt stable main" | sudo tee /etc/apt/sources.list.d/elastic-9.x.list
sudo apt-get update && sudo apt-get install elasticsearch
# edit /etc/elasticsearch/elasticsearch.yml:
# network.host: 0.0.0.0
# http.port: 9200
# discovery.type: single-node
sudo systemctl enable --now elasticsearch
sudo systemctl status elasticsearch

When installing for the first time, security configuration will be displayed, including the default password for the elastic user

Note: Elasticsearch & Kibana versions must match.

2) Install & configure Kibana

bash
sudo apt install kibana
# edit /etc/kibana/kibana.yml:
# server.host: "0.0.0.0"
# server.port: 5601
# elasticsearch.hosts: ["http://localhost:9200"]
sudo systemctl enable --now kibana
# browse to http://<IP>:5601 to finish setup (enrollment token / initial config as needed)

3) Install Winlogbeat (Windows host)

  1. Download and extract Winlogbeat (match ES version).
  2. Edit winlogbeat.yml — configure winlogbeat.event_logs and output.logstash or output.elasticsearch.
  3. Install as service (PowerShell):
powershell
cd 'C:\Program Files\winlogbeat'
.\install-service-winlogbeat.ps1
Start-Service winlogbeat
Get-Service winlogbeat
  1. Verify the winlogbeat-* index appears in Kibana Discover.
